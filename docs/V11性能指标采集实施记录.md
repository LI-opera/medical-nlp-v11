## 当前统计口径

- 作业耗时：优先使用最近一次 `benchmark.job.end.duration_ms`。
- Case 延迟：只统计该作业对应的 `benchmark.case.end.duration_ms`。
- 分类延迟：按 case 的 category 分组，同时读取结果文件中的 correct 计算分类准确率。
- P50/P95：对本次已记录样本使用线性插值计算，不代表长期线上分位数。
- 吞吐：`total_cases / job_duration_seconds`。
- 阶段耗时：只统计与当前 Benchmark 作业的 `job_id` 或 case `request_id` 能关联、且明确含有 `duration_ms` 的 dependency/pipeline 记录。

报告不会修改 Benchmark 结果，也不会重新运行 LLM、Embedding 或 Milvus。

## 当前实际结果

本轮使用已有日志生成的报告，当前最近一次可识别作业为 50 条 case、2 个 worker：

- 总耗时：`186863.64 ms`
- 平均 case 延迟：`6517.82 ms`
- P50：`6129.84 ms`
- P95：`11873.82 ms`
- 最大 case 延迟：`17957.16 ms`
- 吞吐：`0.268 case/s`

阶段日志当前可提供：LLM 调用、向量检索和 pipeline final 等已记录事件的耗时。

## 明确限制

1. 这是 Benchmark 运行报告，不是生产长期监控系统。
2. 只有日志实际记录 `duration_ms` 的阶段才会出现阶段指标；没有记录时会标记不可用。
3. 当前没有把 Embedding、Milvus、LLM 的每个内部子步骤全部拆成独立计时指标。
4. `ruff` 未在当前虚拟环境中安装，因此本轮无法执行 Ruff 检查；pytest 已通过。
5. 真实 integration/live 测试仍由环境变量控制，不会被性能报告脚本隐式启动。

## 后续扩展

后续如果需要比较串行和并行性能，应固定同一批 cases，分别运行 `workers=1` 和 `workers=2`，保存两份报告后再比较，不应把不同数据集的结果直接横向比较。
