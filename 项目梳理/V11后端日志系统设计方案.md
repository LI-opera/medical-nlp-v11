# V11 后端日志系统设计方案

本文是 medical-nlp V11 后端日志系统的可执行设计方案。目标不是到处加 `print`，而是建立一套能追踪“单句分析、benchmark、错误分析、候选沉淀、写入 primary”等核心流程的结构化日志体系。

## 1. 设计目标

后端日志要解决 5 类问题：

```text
1. 单句 Analyze 为什么得到这个结果？
2. benchmark 上传运行为什么慢、错在哪个 case？
3. fallback / coverage / standardization / verifier 哪一层失败？
4. 哪些操作改动了项目数据，比如写入 primary？
5. 前端报错时，能不能用 request_id / job_id 追到后端真实过程？
```

日志系统不应该记录大量无意义文本，而应该记录可筛选、可复盘、可做页面展示的结构化事件。

推荐格式：

```text
JSONL
```

也就是一行一个 JSON 对象：

```json
{"ts":"2026-07-11T10:30:00.123+08:00","level":"INFO","event":"pipeline.start","request_id":"ana_xxx","text_hash":"..."}
```

## 2. 总体分层

后端日志建议分 5 个文件：

```text
backend/logs/app.jsonl
backend/logs/dependency.jsonl
backend/logs/pipeline.jsonl
backend/logs/benchmark.jsonl
backend/logs/audit.jsonl
```

### 2.1 app.jsonl

用途：

```text
记录 API 级别状态、启动、接口异常、全局错误。
```

适合内容：

```text
FastAPI 启动
健康检查异常
接口 500
未捕获异常
外部依赖初始化失败
```

格式统一使用 JSONL，避免后续前端日志查看页兼容多种格式。

### 2.2 dependency.jsonl

用途：

```text
记录外部依赖和基础能力是否可用。
```

覆盖对象：

```text
DeepSeek / Qwen LLM 配置与调用
HuggingFace embedding 模型加载
Milvus 连接
SNOMED / RxNorm collection load
向量检索 search
```

它回答：

```text
这次失败是业务判断失败，还是 LLM / Milvus / embedding 没起来？
DeepSeek API key 是否缺失？
Milvus 是否连上？
SNOMED / RxNorm collection 是否能 load？
向量 search 是 0 结果还是依赖异常？
```

dependency 日志只记录依赖状态，不记录完整业务链路。比如一次标准化失败时：

```text
pipeline.jsonl
  standardization.retrieve failed

dependency.jsonl
  dependency.vector_search.error Milvus connection refused
```

这样可以把“业务逻辑问题”和“外部依赖问题”分开。

### 2.3 pipeline.jsonl

用途：

```text
记录单次文本分析主链路。
```

覆盖路径：

```text
/expand/simple
ABBRService.expand_verify_with_retry
_get_abbreviation_candidates
primary retriever
fallback retriever
coverage evaluator
MedicalRetriever / StdService
ABBVerifier
reflection
final result
```

这是最重要的日志。

### 2.4 benchmark.jsonl

用途：

```text
记录 benchmark job 和每个 case 的运行情况。
```

覆盖路径：

```text
POST /benchmark/cases/jobs
GET /benchmark/cases/jobs/{job_id}
run_benchmark
error_analysis_report
error_triage
collect_fallback_candidate_promotions
```

它回答：

```text
这轮 benchmark 跑了多少 case？
第几个 case 慢？
哪个 stage 慢？
哪个 case 报错？
postprocess 卡在哪一步？
```

### 2.5 audit.jsonl

用途：

```text
记录会改变项目数据的操作。
```

覆盖路径：

```text
candidate-promotions/apply
candidate-promotions/apply-single
apply_fallback_candidate_promotions.py
上传 benchmark cases 并覆盖 benchmark_results.json
未来可能的配置修改、词典修改
```

它回答：

```text
什么时候写入了 primary？
写入了哪些 abbreviation / expansion？
哪些被跳过？
谁触发的？
哪个文件被改了？
```

## 3. 统一关联字段

每条日志都尽量带以下字段。

### 3.1 通用字段

```json
{
  "ts": "2026-07-11T10:30:00.123+08:00",
  "level": "INFO",
  "event": "pipeline.start",
  "component": "ABBRService",
  "request_id": "ana_20260711_103000_ab12cd",
  "job_id": null,
  "case_id": null,
  "duration_ms": 123,
  "ok": true
}
```

字段解释：

```text
ts
  带时区时间。

level
  DEBUG / INFO / WARNING / ERROR。

event
  结构化事件名，后面统一靠它筛选。

component
  产生日志的模块。

request_id
  单句 analyze 或单个 benchmark case 的链路 ID。

job_id
  benchmark 上传运行任务 ID。

case_id
  benchmark case ID。

duration_ms
  当前步骤耗时。

ok
  当前步骤是否成功。
```

### 3.2 文本字段脱敏策略

第一版不建议完整记录临床原文。建议记录：

```text
text_hash
text_preview
text_len
```

示例：

```json
{
  "text_hash": "sha256:abcd...",
  "text_preview": "The patient has SOB and CP.",
  "text_len": 27
}
```

当前项目虽然是本地开发，但医疗文本天然敏感，所以默认不记录 `text_preview`。

建议：

```text
默认只记录 text_hash / text_len。
本地调试时可以通过 LOG_TEXT_PREVIEW=1 临时打开 preview。
preview 最多保留前 160 个字符。
```

## 4. request_id / job_id 设计

### 4.1 Analyze request_id

每次 `/expand/simple` 请求生成：

```text
ana_YYYYMMDD_HHMMSS_随机短码
```

示例：

```text
ana_20260711_103012_a81f2c
```

这个 ID 要贯穿：

```text
api.main.expand_abbreviation_simple
ABBRService.expand_verify_with_retry
candidate retrieval
coverage
standardization
verifier
final response
```

第一版如果不想大改函数签名，可以用轻量上下文：

```python
contextvars.ContextVar
```

例如：

```python
current_request_id.set(request_id)
```

后续各模块写日志时自动读取当前 request_id。

### 4.2 Benchmark job_id

现在已有：

```text
POST /benchmark/cases/jobs
```

返回 `job_id`。

benchmark 日志必须带：

```text
job_id
case_id
case_index
case_total
```

每个 case 内部最好再生成一个 request_id：

```text
bench_20260711_103100_jobid_case001
```

这样可以把 benchmark case 和单句 pipeline 复用同一套 pipeline 日志。

## 6. 各模块接入点

### 6.1 `backend/api/main.py`

需要接入：

```text
/expand/simple
/benchmark/cases/jobs
/candidate-promotions/apply
/candidate-promotions/apply-single
/analysis/diagnose
```

日志点：

```text
1. API 收到请求。
2. 生成 request_id / job_id。
3. 调用服务开始。
4. 返回成功/失败。
5. 对写文件接口记录 audit。
```

重点：

```text
/benchmark/cases/jobs
```

当前已经有 job 状态：

```text
queued
preparing
running_benchmark
saving_results
error_analysis_report
error_triage
fallback_promotions
completed / failed
```

这些 stage 每次变化时，都应该写一条 `benchmark.job.stage`。


#### 6.2.4 verifier

记录 verifier 输入规模和输出，不记录完整 prompt。

```json
{
  "event": "standardization.verify",
  "mapping_count": 2,
  "validation_count": 2,
  "overall_valid": true
}
```

每个 record 的结果：

```json
{
  "event": "standardization.coded",
  "abbreviation": "SOB",
  "expansion": "shortness of breath",
  "chosen_concept": {
    "concept_id": 312437,
    "concept_name": "Dyspnea",
    "domain_id": "Condition"
  }
}
```

或：

```json
{
  "event": "standardization.withheld",
  "abbreviation": "ABC",
  "expansion": "Airway, Breathing, Circulation",
  "reason": "no faithful SNOMED concept among retrieved candidates",
  "retrieved_top": ["Route of breathing", "Ventilation"]
}
```

反思是后续排错重点，建议记录：

### 7.0 硬性工程约束

日志系统必须遵守：

```text
日志失败绝不能影响业务运行。
```

也就是说：

```text
1. 日志文件写入失败，不能导致 /expand/simple 返回 500。
2. 日志轮转失败，不能中断 benchmark。
3. JSON 序列化失败，不能影响 primary 写入。
4. 日志目录不存在时应自动创建；创建失败时最多降级到 stderr。
```

实现建议：

```text
structured_logger 内部捕获所有日志写入异常。
业务代码只调用 log_xxx，不处理日志异常。
日志模块可以在开发模式下把写入异常输出到 stderr，但不能向上抛。
```

### 7.1 `trace_context.py`

职责：

```text
保存当前 request_id / job_id / case_id。
```

建议实现：

```python
from contextvars import ContextVar

request_id_var = ContextVar("request_id", default=None)
job_id_var = ContextVar("job_id", default=None)
case_id_var = ContextVar("case_id", default=None)
```

提供函数：

```python
set_trace_context(request_id=None, job_id=None, case_id=None)
get_trace_context()
clear_trace_context()
```

## 8. 日志轮转策略

第一版建议使用 Python 标准库：

```python
logging.handlers.RotatingFileHandler
```

不要自己手写复杂 rename 逻辑，尤其 Windows 下文件占用和 rename 容易出问题。

轮转策略：

```text
单文件超过 10MB 后，重命名为 .1
最多保留 5 个历史文件
```

示例：

```text
pipeline.jsonl
pipeline.jsonl.1
pipeline.jsonl.2
```

不建议第一版引入复杂日志服务。

## 9. 日志开关

通过环境变量控制：

```text
LOG_LEVEL=INFO
LOG_TEXT_PREVIEW=0
LOG_LLM_RAW_OUTPUT_ON_ERROR=1
LOG_RAW_OUTPUT_MAX_CHARS=2000
LOG_MAX_FILE_MB=10
LOG_MAX_BACKUPS=5
```

解释：

```text
LOG_TEXT_PREVIEW
  是否记录 text_preview。默认关闭，本地调试可临时打开。

LOG_LLM_RAW_OUTPUT_ON_ERROR
  只有 LLM 返回 invalid_json / invalid_schema 时，是否记录 raw_output。

LOG_RAW_OUTPUT_MAX_CHARS
  raw_output 最大记录长度，避免 LLM 异常输出撑爆日志。
```

## 10. 实施步骤

建议分 6 批做，不要一次全塞进去。

### 10.0 最新实施顺序（以本节为准）

本轮讨论后，实施顺序调整为：

```text
1. structured_logger + trace_context 基础设施
2. dependency.jsonl
3. app.jsonl / API 请求日志
4. benchmark.jsonl
5. pipeline.jsonl
6. audit.jsonl
```

这样排的原因是：

```text
先解决“依赖是否可用”和“接口是否正常进入”的可观测性，
再记录 benchmark 长任务，
最后再深入到单句 pipeline 的内部节点和 primary 写入审计。
```

也就是说，第一版不要一上来就把 ABBRService 每个内部细节全部打满。
先保证以下问题能被快速定位：

```text
1. 后端有没有收到请求。
2. 请求是否分配了 request_id / job_id。
3. LLM / Embedding / Milvus / collection 是否可用。
4. benchmark 当前跑到哪个阶段、哪个 case。
5. 单句 pipeline 最终是成功、拒绝还是异常。
6. 写入 primary 的动作是否可追溯。
```

下面的批次说明保留更细的接入点，但真正执行时以上面的顺序为准。

## 11. 不建议第一版做的事

第一版不要做：

```text
1. 不要做数据库日志系统。
2. 不要做 ELK / Loki / Grafana。

4. 不要记录完整 prompt。

6. 不要把日志写进 error_analysis_report.json。

```

理由：

```text

先把 request_id / job_id / JSONL 跑通，比引入复杂工具更重要。
前端日志查看页等后端事件名、字段和文件结构稳定后再做。
```

## 12. 未来前端日志如何接入

第一版先只落后端文件日志，不做前端日志查看页，也不让前端批量上报 console。

当前阶段前端只保留现有错误展示即可。

等后端日志事件名、字段和文件结构稳定后，再考虑：

```text
POST /frontend-log
前端日志查看页
```

前端只上报关键事件：

```text
analyze.click
benchmark.upload.click
benchmark.job.poll_error
primary.apply.click
api.error
```

上报时带：

```text
request_id
job_id
route

message
```

这样前端日志可以和后端日志通过 ID 串起来。

## 13. 最小可用版本定义

MVP 日志系统只要做到这些，就算第一版成功：

```text
1. 每次 /expand/simple 有 request_id，app.jsonl 能看到 request start/end/error。
2. dependency.jsonl 能看到 LLM config、embedding load、Milvus connect、collection load、vector_search。
3. 每次 benchmark cases run 有 job_id。
4. benchmark.jsonl 能看到 job stage 和 case start/end。
5. pipeline.jsonl 能看到 start/final/error。
6. audit.jsonl 能看到写入 primary 的 appended/skipped。
7. 日志文件不会无限增长，使用 RotatingFileHandler 做简单轮转。
8. 日志失败不会影响业务运行，写日志异常不能导致接口 500。
```

### 13.1 dependency 补充验收

由于后端依赖 Milvus、Embedding 模型和 LLM，MVP 还必须满足：

```text
dependency.jsonl 能看到：
1. LLM config check / create ok / config error。
2. embedding load start / ok / error。
3. Milvus connect start / ok / error。
4. SNOMED / RxNorm collection load start / ok / error。
5. vector_search start / ok / error。
```

这部分日志用于区分：

```text
业务链路判断失败
vs
外部依赖不可用
```

例如 Milvus 没启动时，前端不应该只看到 500，后端应在 `dependency.jsonl` 里明确留下：

```text
dependency.milvus.connect_error
或 dependency.collection.load_error
或 dependency.vector_search.error
```

## 14. 面试表达

可以这样说：

> 我没有简单用 print 调试，而是给后端设计了一套结构化日志体系。依赖层用 dependency.jsonl 记录 LLM、embedding、Milvus 和 collection 是否可用；单句分析使用 request_id 贯穿缩写发现、候选召回、coverage、标准化检索、verifier 和最终状态；benchmark 使用 job_id 和 case_id 记录长任务阶段、每个 case 的结果和耗时；涉及修改候选库的操作单独进入 audit 日志。日志采用 JSONL，方便后续按 request_id/job_id 检索，也方便前端做日志查看页面或错误复盘。
