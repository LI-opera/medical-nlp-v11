# 第一部分 · 项目状态与背景

## 1.1 项目是什么

**Medical NLP**:一个"医学缩写扩写 + SNOMED CT 术语标准化"服务。输入一句临床文本(含缩写如 CP / MS / SOB),输出:① 把缩写扩写成全称的整句;② 每个扩写词对应的 SNOMED 标准概念编码。

技术栈:
- **后端** FastAPI + Uvicorn,懒加载(`service=None`,首个请求才建 `ABBRService`)
- **LLM** DeepSeek Chat(`deepseek-chat`, temperature=0),经 `langchain-deepseek`
- **Embedding** BAAI/bge-m3(normalize=True)
- **向量库** Milvus(COSINE,collection `concepts_only_name`),装 SNOMED 概念
- **NER** HuggingFace `Clinical-AI-Apollo/Medical-NER`(token-classification, aggregation="simple")
- **核心理念** Retrieval + Verification + Reflection:"**不要直接相信 LLM 的生成,加独立校验 + 反思重试**"。Verify→Reflect→Retry 循环 = Agent 雏形。
- **LangGraph** 版状态机已实现但未接 API(`backend/graph/`),是简历"Agent"关键词落点。

## 1.5 现状 V9 Stable 的真实运行流程(已核对代码)

主入口 `ABBRService.expand_verify_with_retry(text, max_retries=2)`(`backend/services/abbr_service.py`,约 313–512 行):

```text
1. simple_llm_expansion(text)  ← 约64–134行
     内部 _get_abbreviation_candidates(text)  ← 约515–595行：
        逐 token → gate _should_consider_abbreviation(597–628) 判是否像缩写
        → 词典召回 candidate_retriever.retrieve(abbr)
        → 空则 fallback_retriever.retrieve(abbr, text)（LLM 生成候选）
        → coverage_evaluator.evaluate() 返回 plausible_candidates(字符串列表)
        → filtered_candidates = [c for c in candidates if c["expansion"] in plausible_expansions]  ← 584行，脆弱字符串匹配
     然后把 candidates 塞进 prompt，★用 LLM 造句扩写★ → expanded_text + mappings(带 source 字段)
2. for attempt in range(max_retries+1):   最多 1 正常 + 2 反思 = 3 轮
     a. valid_mappings = [m for m in current_mappings if m.get("expansion")]  ← 365行
     b. 若 valid_mappings 空 → 早停返回 stop_reason="coverage_failed_no_valid_expansion"  ← 371行
     c. standardizer.standardize(current_expanded_text)  ← 406行，整句NER+检索，存档但★不喂verify★
     d. 对每个 valid_mapping: retriever.retrieve(expansion, top_k=10, domain_filter=None, score≥0.6)[:3]  ← 413行
          攒 mapping_standardizations
     e. verifier.verify_mappings(原句, expanded_text, mapping_standardizations)  ← 441行
     f. attempts.append(...)  全程留痕
     g. overall_valid==True → 成功返回  ← 472行
     h. attempt>=max_retries → 失败返回  ← 481行
     i. reflector.reflect(...) → ★整句重写★ current_expanded_text=revised; current_mappings=revised_mappings  ← 493–505行
```

关键代码事实(面试 + 改动都要知道):
- `overall_valid = sentence.is_valid AND len(mappings)>0 AND all(m.is_valid)`(在 `abbr_verifier.py`)——`all()` 偏严,一个错整轮重做。
- `retriever.retrieve` 的 `domain_filter` 参数**功能是好的**(`medical_retriever.py` 82 行真的会按 domain 过滤),但全程被传 `None` → **死参数**。
- `MappingSupportVerifier` 在 `__init__` 初始化了,但 `_filter_mappings_by_context_support` 的调用在 `expand_verify_with_retry` 里**整段注释**(约 348–360 行)。**保留勿删**,是有意的实验分支。
- `_rebuild_expanded_text`(229 行)用的是朴素 `str.replace(abbr, expansion)`——**有子串误伤风险**(CP 会命中 CPR 里的 CP),V11 要换成 token 边界正则。
- reflect 用整句重写,粒度粗,可能改坏本来对的映射。
- 跑 benchmark:`backend/evaluation/run_benchmark.py`(50 例、6 类、exact-match);错误归因 `error_analysis_report.py`。已知弱项:**low_context 类历史只有 ~20%**。

# 第二部分 · 我们之前讨论出的核心设计结论(V11 的"为什么")

> 这些是和上一个 Claude 窗口逐轮深挖推出来的。**完整版在 `22` 篇**,这里浓缩成你指挥 Codex 时必须守住的"设计意图",防止 Codex 把 what 做了、把 why 做歪。

1. **确定性优先**:扩写的本质是"把缩写换成选定的全称"。V9 用 LLM 造句反而引入"丢否定 / 幻觉",而这正是后面 verify 要补救的。**能用确定性替换做的,就别用 LLM 生成。**

2. **确定性替换是"mapping 粒度隔离"的使能条件(两者绑死)**:只有"局部 token 替换"才物理隔离——改 MS 那个位置不可能动到 CP。若仍用 LLM 整句重写,它一开口就是整句,无法只改一个。所以"确定性替换"和"按 mapping 粒度只修出错项"是同一个决策的两面,要一起做。

3. **工作单位从"整句"降到"单个 mapping"**:每个 mapping 走自己的状态链 `PENDING → 检索 → verify → LOCKED_OK / LOCKED_ABSTAIN`。失败只隔离重修它自己,LOCKED_OK 的冻结复用(增量重算)。能区分失败模式:"扩写选错"(换候选)vs"SNOMED 缺概念"(未必换)。

4. **循环边界**:召回阶段(gate→词典→fallback→NER→coverage)在**循环外**只算一次,因为候选池不随重试变(gate 确定性、其余昂贵且结果基本不变)。检索+verify+reflect 在**循环内**,因为 reflect 会改"被选中的 expansion",而**检索查的是"被选中的那个词"、不是候选池**——池固定 ≠ 检索输入固定,reflect 一换词检索必须重跑。

5. **reflect 两种模式 + 单候选→弃权**:"换候选"需要池里有 ≥2 个。所以 **fallback 改为一次产 top-k(≥3)**,留出可换空间;某 mapping 候选用尽仍不过 → **LOCKED_ABSTAIN(弃权不扩,保留原缩写/送人工)**,这是"不该扩就不扩"的安全失败。**循环继续条件 = 还有没有 mapping 既没过又有未试候选**,不是单纯轮次;`max_retries` 只作防死循环的兜底。

6. **早停判的是"mappings 整列表为空"**(整句没有任何缩写扩成功),不是"某条 mapping 缺 expansion"。重构版里失败缩写**不进 mappings**,所以凡进列表的都带 expansion。

7. **NER 重新安置 + 让输出被消费**:V9 里 NER 在整句标准化、算了不用。重构挪到 **fallback 后做候选校验(is_medical 杀垃圾)+ 产类型 label**;label 再去**软约束 SNOMED 检索的 domain**(激活那个死参数)。NER 是本地模型、与 deepseek 不同源 → **跨模型第二意见**,缓解同源盲区。词典候选用元数据 label、fallback 候选用 NER label,**召回路输出一致字段**。

8. **交付物升级**:出口除 `expanded_text + mappings`,加 `standardized_entities`(每个 LOCKED_OK 的 SNOMED concept_id/code)——兑现"标准化平台",不止扩写。

## 批次 1 · coverage 选唯一 + 确定性替换(核心 ②③)

**改动点**

1. `backend/services/abbr_candidate_coverage_evaluator.py · evaluate()`
   - 在返回 JSON 里**新增 `best_expansion`(单个,最契合上下文的那个候选字符串)**;`plausible_candidates` 保留向后兼容。
   - prompt 加一条规则:"从 plausible 中选出唯一最契合的,放进 best_expansion;coverage_ok=false 时 best_expansion=null。"
2. `backend/services/abbr_service.py · _get_abbreviation_candidates()`(约 515–595)
   - 用 `coverage.get("best_expansion")` 为每个缩写定**唯一 chosen expansion**;并把对应候选的 **label/domain 元数据**一并带出(词典候选从 `ABBR_CANDIDATES` 元数据取)。
   - 选不出(best_expansion=null)→ 该缩写**不产生 chosen 条目**。
3. `backend/services/abbr_service.py` 新增 `_build_expanded_text_deterministic(text, chosen)`
   - 对每个 `{abbr→expansion}` 用 `re.sub(r"\bABBR\b", expansion, text)` 按 **token 边界**替换(转义 abbr);**多替换从后往前按原始位置替,避免 offset 错位**。升级/取代现有朴素的 `_rebuild_expanded_text`(229 行)。
4. `backend/services/abbr_service.py · expand_verify_with_retry()`
   - 把第 1 步的 `simple_llm_expansion`(LLM 造句)替换为:`_get_abbreviation_candidates` 选唯一 → `_build_expanded_text_deterministic` 生成 `expanded_text` + `mappings=[{abbr, expansion, label, source}]`。**移除扩写阶段的 LLM 调用。**
   - `simple_llm_expansion` 可保留供老接口,但主链路不再用。
   - 早停哨兵改判 **`if not mappings`**(整列表空),措辞 `stop_reason` 保留。

**不要碰**:verify / reflect 逻辑这批先不动(下一批);MappingSupportVerifier 注释保持。

**验收**
- 单测:`"Patient denies CP"` → `"Patient denies chest pain"`,否定保留;`"CPR"` 不被误伤。
- benchmark **net accuracy ≥ 基线**(尤其不能让本来对的类掉分)。否定保持类应改善或持平。
- 不正 → 回退。

## 批次 2 · mapping 粒度状态机 + 失败隔离 + 增量重算(核心 ⑤)

**改动点**:重构 `expand_verify_with_retry` 的循环(约 362–512)

1. 给每个 mapping 加 `status`:`PENDING / LOCKED_OK / LOCKED_ABSTAIN`,并存其**候选池 + 已试候选集**(从 `_get_abbreviation_candidates` 的 `candidates` 带进来)。
2. 循环体改为**对未锁定的 mapping 逐个处理**:
   - 检索:**只对本轮 expansion 变过的 mapping 重跑** `retriever.retrieve`;LOCKED_OK 的复用缓存。
   - verify:见批次 5 可瘦身;本批先沿用 `verify_mappings`,但**按 mapping 读取各自的 `is_valid`**。
   - 通过 → `LOCKED_OK`;不通过 → 有未试候选则 reflect 换一个(见下),无则 `LOCKED_ABSTAIN`。
3. reflect 改造(`abbr_reflection_service.py` + 调用处):从"整句重写"改为**"给指定的出错 mapping,从其候选池里改选一个未试候选"**(选择型)。只更新该 mapping 的 expansion,并标记"已变,下轮重检索";对的 mapping 不动。然后用 `_build_expanded_text_deterministic` 重新拼整句。
4. **继续条件**:`while 存在 mapping 既非 LOCKED_OK 也非 LOCKED_ABSTAIN`;`max_retries` 作兜底上限。全部锁定即退出。
5. 出口汇总每个 mapping 终态;`attempts` 留痕保留。

**验收**
- 多缩写用例(`"Patient has CP and MS"`,CP 对 MS 错):CP 一轮锁定后**不再被重检索/不被改写**;只有 MS 在换候选。可在日志/attempts 验证。
- benchmark **net ≥ 批次 1**;多缩写用例的 LLM/检索调用次数**下降**(量一下)。
- 不正 → 回退。

## 批次 3 · NER 安置 + label + fallback top-k(① + 召回一致字段)

**改动点**
1. `backend/services/abbr_candidate_fallback_retriever.py`:确保**一次返回 top-k(≥3)** 候选(prompt 已倾向多候选,补"返回至少3个按可能性排序"并在解析处保留多个)。
2. fallback 之后,对每个候选 expansion 调 **NER**(`ner_service` / `MedicalStandardizer` 暴露一个轻量 `is_medical(text)->bool, label`):
   - `is_medical=False` → 丢弃该候选(省 coverage 的 LLM)。
   - `is_medical=True` → 候选带上 `label`。
3. 词典候选:从 `backend/data/abbr_candidates.py` 给每条加/读取 `domain`/`label` 元数据,保证**词典候选与 fallback 候选都带 label**(召回路一致字段)。

**验收**:fallback 候选带 label;明显非医学的 fallback 候选被丢;benchmark net ≥ 批次 2。

## 批次 4 · label 软约束 domain(④,激活死参数)

**改动点**
1. `backend/services/medical_retriever.py · retrieve()`:新增 `domain_boost: str|None`,在 `_rerank_results` 里对 **domain_id 匹配的结果加分(软加分,非硬过滤)**;保留原 `domain_filter` 不动。
2. `abbr_service.py` 检索调用:把 `domain_filter=None` 旁边加 `domain_boost = label→domain 映射`。
3. 写一个 `label→domain_id` 映射表(NER 标签 → SNOMED domain)。

**为什么软加分**:映射错时顶多排序略偏,不会像硬过滤那样**弄丢正确答案**。

**验收**:检索精度持平或上升;benchmark net ≥ 批次 3。

## 批次 5 · verify 瘦身 + 出口 standardized_entities(⑤尾 + 交付物)

**改动点**
1. `abbr_verifier.py · verify_mappings`:加开关,**确定性替换路径下可跳过句子级保意检查**(因为替换不改坏句子),只保留映射级。默认可先保留句子级,用 flag 控制,benchmark 验证后再决定。
2. `backend/api/schemas.py · SimpleExpandResponse`:新增 `standardized_entities: list[dict]`。
3. `backend/api/main.py`:从 LOCKED_OK 的 mapping_standardizations 取 SNOMED `concept_id/concept_code`,填入响应。

**验收**:响应含标准编码;去句子级后 benchmark net 不掉;不正则保留句子级。
