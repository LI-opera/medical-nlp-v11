整个程序设计缺陷：无缩写的句子 = gate 全跳过 → 无候选无映射 → 早停 `success=False`、文本不动、不做 SNOMED 检索。

> **改进方向**:把"实体标准化"和"缩写扩写"拆成两条独立路径——先标准化原句所有实体(NER→检索),再单独处理缩写;或者至少把 `success=False / coverage_failed` 这个误导性状态,改成正确区分"无缩写(正常)"和"扩写失败(异常)"。

### 精炼后的设计

既然你担心层层重复+成本,可以**用顺序解决**——让便宜的先上:



```text
fallback 生成候选
   ↓ NER（本地模型，比 coverage 的 LLM API 便宜快）
   ├─ is_medical=否 → 直接杀掉（明显垃圾，不必再花 coverage 的 LLM）
   └─ is_medical=是 → 带着 label 进 coverage
   ↓ Coverage（贵的 LLM，只对"是医学词"的候选判上下文契合）
```



```text
对每个 expansion：
  if 来源是 fallback:                    ← 词典候选跳过校验，直接信（坑3）
     ner_result = NER(expansion)         ← 跨模型校验（坑4：可批量）
     if not ner_result.is_medical:
        标记为低置信 / 交 error analysis 复核   ← 软处理，不硬中断（坑1）
     domain = LABEL_TO_DOMAIN.get(ner_result.label)
  检索(expansion, domain_boost=domain)   ← 软约束，加分不硬删（坑2）
```

**一句话:你的方向对——把 NER 从"发现整句实体"改成"校验+分类干净的 expansion",既给它承重活、又免了合并、还跨模型缓解同源盲区。但三处要"软":NER 判不出别硬中断、domain 用加分别硬过滤、校验只针对 fallback 候选。**

实施这一过程时需要将路1也就是候选库直接选的那种，在它的候选库里加上domain医学标签元数据这样fallback通过ner检索之后也声称的便签这样去snomed大家都一致。

2.**`plausible_candidates` 是字符串，上游用"精确匹配"过滤**：`c["expansion"] in plausible_expansions`。如果 LLM 返回的字符串和原候选**大小写/措辞稍有出入**（"Mitral Stenosis" vs "mitral stenosis"），匹配就漏，候选被错误丢弃。这是个真实的脆弱耦合。

###  

```
 字符串精确匹配为什么会"漏"

看上游真实代码(第 14 篇会细讲):


plausible_expansions = coverage.get("plausible_candidates", [])   # LLM 返回的文字串列表
filtered_candidates = [
    c for c in candidates if c["expansion"] in plausible_expansions
]


`candidates`(召回来的)长这样:


[{"abbreviation":"MS", "expansion":"mitral stenosis"}, ...]   # 注意：小写


`plausible_expansions`(coverage 的 LLM 返回的)假设是:


["Mitral Stenosis"]   # LLM 手抖大写了


那句过滤的意思是:**只保留 `c["expansion"]` 这个字符串"一字不差"地出现在 `plausible_expansions` 里的候选。**


"mitral stenosis" in ["Mitral Stenosis"]   # → False！大小写不同，不算相等


结果:`mitral stenosis` 这个**本来对的候选被丢掉了**,只因为 LLM 把首字母大写了。`in` 对字符串是"完全相等"判断,差一个字符就 miss。

**更稳的修法**:别用文字串对账。让 coverage 返回候选的**编号**:


# coverage 改成返回 {"plausible_indices": [0]}
filtered_candidates = [candidates[i] for i in coverage["plausible_indices"]]
# 用编号取，不依赖 LLM 把字符串原样抄回来


或者退一步,两边都先 `.lower().strip()` 再比。核心思想:**不要依赖 LLM 把字符串一字不差地复述。**
```

3.**coverage 判断本身也是 LLM，也会错**：把对的判成 false（over-abstention，该扩没扩）或把错的判成 true（漏网）。而且confidence也是自己生成的。

十一.

- **过滤是"软约束"，不是代码硬截断**：注意 `simple_llm_expansion` 把**整个结构**（candidates + filtered_candidates + coverage）都 dump 进 prompt，靠**规则文字**让 LLM "优先用 filtered"。代码并没有只把 filtered 传进去。所以"只在支持的候选里选"是**靠 LLM 听话**，不是强制的。
  → 面试这么说："候选过滤目前是 prompt 软约束——我把完整候选和 filtered 都给 LLM，靠规则让它优先用 filtered。更稳的是代码层只传 filtered_candidates，做成硬约束，LLM 想越界都没得选。"
- **gate 现在是"全大写、纯字母、≤8字符"三条正字法规则,真实临床文本的脏数据它扛不住。
- **扩写质量完全依赖候选质量**：garbage in → garbage out。候选错了/漏了，LLM 再听话也扩不对。这把压力前移到召回和 coverage。
- **`parse_error` 降级很粗**：JSON 解析失败时，直接把 LLM 原始输出当 `expanded_text`、mappings 给空。这种情况下结果基本不可用，只是不崩。
  → "解析失败是兜底不是修复，可以加重试或更强的结构化输出（如 function calling / JSON mode）。"

十二.

- 在verifier里面裁判也是 LLM，且可能是同一个模型 → 同源盲区，confidence也是自报。理想是 cross-model 校验
- **`mapping_validations` 数量等于输入"靠 LLM 自觉**：prompt 要求数量一致，但**没有代码校验**。如果 LLM 少返回一个，`all(...)` 会在不完整的列表上判断，可能误判通过。
  → "应该在代码层断言返回数量等于输入数量，不一致直接判失败或重试。"

十三.

- "校验和反思共享同一个模型的盲区，verifier 误判会污染 reflect。理想是 cross-model，或给 reflect 也加确定性约束。"
- **整句重写，可能误伤对的部分**：reflect 重写**整个** expanded_text，不是只改出错的那个缩写。多缩写句子里，本来对的可能被一起改坏（回扣第 12 篇 all 偏严 + 整句重来）。
  → "更好的是按 mapping 粒度只修出错的那个，锁定已通过的。"
- **`parse_error` 兜底返回上次扩写**：JSON 解析失败时，原样返回 `previous_expanded_text`——这一轮反思**白做**，下一轮 verify 还是同样结果，可能空转到次数耗尽。
- **没有跨 case 学习**：每次反思独立，不积累"这类错误以前怎么修好的"经验。成本上每轮重试都多一次 reflect+verify 的 LLM 调用。

十四.

**多轮 = LLM 调用叠加，慢且贵**：一次请求里，扩写阶段就有 coverage（可能还有 fallback）+ 扩写；之后每轮再 verify + reflect。最坏 3 轮下来好几次 LLM 调用。思考一下是否能够识别本身就不行的提前撞墙节省一次循环。

**整句 standardize（步骤 c）算了但没喂 verifier**（回扣第 7 篇）：冗余计算，verify 实际用的是步骤 d。更好的是把它喂给 verifier 变有用。⚠️ **注意别删错地方**:`expand_and_standardize`(第 160 行那个方法)是**另一个独立方法**,它的 standardization 是真返回、真被测试用的,**那个不能删**。要删只删 `expand_verify_with_retry` 里第 406 行那次。

十五.

**没有错误处理 / try-except**：如果 DeepSeek 超时、Milvus 连不上，请求会直接抛异常 500，没有优雅降级。

**全局单例 `service`，没考虑并发安全**：多个请求同时进来都共享同一个实例。当前够用（FastAPI 单进程 + 模型只读），但高并发下 Milvus/LLM 客户端是否线程安全没验证。

十六.

- **只容器化了 API,Milvus 没进容器**:所谓"一键部署"是半成品——还得**宿主机先装好 Milvus 并跑 `create_milvus_db.py` 建好库**,API 容器才能用。

  → "应该加 `.dockerignore` 排除 `.env`,密钥只在运行时通过 env_file/environment 注入,不进镜像。这是个安全隐患,我会修。"

- **精确集合相等偏严,没有部分分(partial credit)**:多缩写 case 里扩对 2 个错 1 个,直接算整题错。无法体现"对了一部分"。
  → "评分是 exact-match,偏严。可以补充 mapping 级的 precision/recall/F1,给部分分,更细粒度看能力。"

  → "否定测试用子串匹配是近似,可能漏报或误判,更严谨要做句法/语义级的否定范围检查。"
