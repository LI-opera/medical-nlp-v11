# V11 前端日志系统设计方案

本文是 medical-nlp V11 前端日志系统的可执行设计方案。

当前后端日志系统已经实现第一版：

```text
backend/logs/app.jsonl
backend/logs/dependency.jsonl
backend/logs/pipeline.jsonl
backend/logs/benchmark.jsonl
backend/logs/audit.jsonl
```

前端日志系统的目标不是重复后端日志，而是补齐“用户在页面上做了什么、页面状态如何变化、请求是否从前端成功发出、响应是否正常回到前端”这部分证据。

---

## 0. 评审后收紧原则

根据当前 `frontend/app.js` 的真实结构，前端日志第一版应该保持轻量。
不要一上来做成完整日志平台，也不要把所有页面状态变化都写日志。

第一版只解决三个问题：

```text
1. 用户点了什么。
2. 浏览器向哪个后端接口发了请求，成功还是失败，耗时多久。
3. 页面在 Analyze / Benchmark / Error Analysis / Fallback Promotions 的关键流程卡在哪一步。
```

第一版明确不做：

```text
1. 不记录 render() 日志。
2. 不默认记录 textarea 每次 text_change。
3. 不记录完整临床文本。
4. 不记录完整 Raw JSON。
5. 不记录完整 benchmark 上传文件内容。
6. 不把前端日志失败变成业务失败。
7. 不第一批强制实现 /frontend-log 落盘。
```

因此，本文后续所有设计都按这个边界执行：

```text
第一阶段：浏览器本地 frontendLogger.buffer + console。
第二阶段：确认字段有用后，再考虑 POST /frontend-log -> backend/logs/frontend.jsonl。
```

---

## 1. 为什么前端也需要日志

后端日志回答的是：

```text
API 有没有收到请求？
ABBRService 有没有跑完？
LLM / Milvus / Embedding 是否可用？
benchmark 跑到哪个 case？
primary 写入了什么？
```

前端日志回答的是：

```text
用户点击了哪个按钮？
当前 API 地址是什么？
请求有没有从浏览器发出去？
请求耗时多少？
页面是否拿到了响应？
页面渲染时是否出错？
上传 benchmark 文件是否解析失败？
轮询 job 是否中断？
用户是否确认写入 primary？
```

两者关系：

```text
前端日志 = 用户操作与浏览器侧状态
后端日志 = 服务端链路与依赖状态
```

它们不互相替代，而是通过 `request_id / job_id / event` 串起来。

---

## 2. 当前前端真实结构

当前前端是原生单页应用：

```text
frontend/index.html
frontend/app.js
frontend/styles.css
```

核心结构：

```text
state
  保存 route、apiBase、analyzeResult、benchmark、errors、promotions 等页面状态。

fetchJson(path, options)
  所有后端 API 请求的统一出口。

render()
  根据 state.route 渲染页面。

bindRouteEvents()
  绑定按钮点击、文件上传、路由切换等事件。
```

这意味着前端日志第一版不需要引入 React、Vue、Redux 或日志 SDK。

最合适的设计是：

```text
在 app.js 里增加一个轻量 frontendLogger。
在 fetchJson 统一记录 API 请求。
在关键用户动作处记录 UI event。
在 window.onerror / unhandledrejection 捕获页面异常。
```

---

## 3. 第一版日志边界

第一版只做：

```text
1. 浏览器本地日志 buffer。
2. console 可读输出。
```

说明：

```text
/frontend-log 不放在第一批强制实现。
先在浏览器里验证字段是否有用，再决定是否落盘。
```

第一版不做：

```text
1. 不做前端日志查看页。
2. 不引入第三方日志平台。
3. 不记录完整临床文本。
4. 不记录完整 benchmark 文件内容。
5. 不记录完整 Raw JSON 响应。
6. 不把日志写进 error_analysis_report.json。
```

原因：

```text
当前项目还在本地交付阶段。
前端日志第一版应先解决可观测性，而不是做平台工程。
```

## 4. 日志文件与后端接收方式

建议后端未来新增：

```text
POST /frontend-log
```

写入：

```text
backend/logs/frontend.jsonl
```

注意：

```text
这是第二阶段能力，不是第一版必须项。
第一版先用 frontendLogger.buffer + console 验证日志字段。
```

但实施顺序建议分两步：

```text
阶段 1：
  只在浏览器本地 buffer + console 输出。

阶段 2：
  后端新增 /frontend-log，前端批量 flush。
```

如果直接一步到位，也可以同时做阶段 1 和阶段 2。

## 5. 前端日志统一字段

每条前端日志建议是一个 JSON 对象：

```json
{
  "ts": "2026-07-11T14:20:00.123+08:00",
  "level": "INFO",
  "event": "ui.analyze.click",
  "component": "frontend",
  "route": "analyze",
  "api_base": "http://127.0.0.1:8001",
  "session_id": "fe_20260711_142000_ab12cd",
  "frontend_request_id": "fe_req_20260711_142001_ef34",
  "backend_request_id": null,
  "job_id": null,
  "duration_ms": 123,
  "ok": true
}
```

字段解释：

```text
ts
  浏览器当前时间。

level
  INFO / WARNING / ERROR。

event
  结构化事件名。

component
  固定 frontend，或细分 analyze / benchmark / promotions。

route
  当前页面 route，例如 analyze、benchmarkOverview。

api_base
  当前右上角 API 地址。

session_id
  浏览器本次会话 ID。

frontend_request_id
  前端发起某个 API 请求时生成的 ID。

backend_request_id
  如果后端响应未来返回 request_id，可记录在这里。

job_id
  benchmark job 轮询时使用。

duration_ms
  API 或动作耗时。

ok
  当前事件是否成功。
```

## 6. 脱敏策略

前端日志默认不记录完整临床文本。

Analyze 输入只记录：

```text
text_len
text_hash
```

本地调试时可以临时开启：

```text
FRONTEND_LOG_TEXT_PREVIEW=1
```

但由于当前是浏览器静态页面，没有构建环境变量，第一版可以用 localStorage 控制：

```text
localStorage.setItem("medicalNlpFrontendLogPreview", "1")
```

默认：

```text
不记录 text_preview。
```

benchmark 上传文件只记录：

```text
file_name
file_size
case_count
category_count
```

不记录完整文件内容。

## 7. 事件设计

### 7.1 页面与路由事件

```text
ui.app.load
ui.route.change
ui.api_base.save
ui.health.check_start
ui.health.check_ok
ui.health.check_error
```

用途：

```text
确认用户打开了页面。
确认用户是否切换了 Analyze / Benchmark。
确认右上角 API 地址是否设置正确。
```

### 7.2 Analyze 页面事件

```text
ui.analyze.sample_click
ui.analyze.clear_click
ui.analyze.click
ui.analyze.result_ok
ui.analyze.result_error
ui.diagnosis.click
ui.diagnosis.result_ok
ui.diagnosis.result_error
```

建议记录字段：

```text
text_len
text_hash
mapping_count
mapping_state_count
success
expansion_success
standardization_success
duration_ms
```

不要记录完整 `analyzeResult`。

说明：

```text
不默认记录 ui.analyze.text_change。
textarea 每次输入都会触发 input 事件，如果逐字记录会造成日志噪声，也可能变相暴露输入内容。
第一版只在点击分析时记录 text_len / text_hash。
```

### 7.3 Benchmark 页面事件

```text
ui.benchmark.refresh_click
ui.benchmark.load_ok
ui.benchmark.load_error
ui.benchmark.upload_click
ui.benchmark.file_read_ok
ui.benchmark.file_parse_error
ui.benchmark.job_create_ok
ui.benchmark.job_poll
ui.benchmark.job_completed
ui.benchmark.job_failed
```

建议记录字段：

```text
file_name
file_size
case_count
job_id
stage
progress
current
total
accuracy
duration_ms
```

轮询日志需要限频：

```text
不要每秒轮询一次就写一条日志。
只在 stage 改变、progress 跨过 10% 阶段、completed、failed、poll error 时记录。
```

### 7.4 Error Analysis 页面事件

```text
ui.errors.refresh_click
ui.errors.load_ok
ui.errors.load_error
ui.errors.slice_select
```

建议记录字段：

```text
failed_case_count
selected_error_slice
triage_exists
duration_ms
```

### 7.5 Fallback Promotions 页面事件

```text
ui.promotions.refresh_click
ui.promotions.load_ok
ui.promotions.load_error
ui.promotions.apply_click
ui.promotions.apply_confirm
ui.promotions.apply_cancel
ui.promotions.apply_ok
ui.promotions.apply_error
ui.promotions.apply_single_click
ui.promotions.apply_single_confirm
ui.promotions.apply_single_ok
ui.promotions.apply_single_error
```

建议记录字段：

```text
total_items
new_item_count
already_exists_count
abbreviation
expansion_hash
domain
appended_count
skipped_count
duration_ms
```

不要记录完整 candidate list。

### 7.6 全局异常事件

```text
frontend.error
frontend.unhandled_rejection
frontend.render_error
```

建议记录字段：

```text
message
stack_preview
route
api_base
```

stack 最多截断：

```text
2000 chars
```

## 8. 前后端 request_id 如何关联

当前后端 `/expand/simple` 内部已经生成：

```text
request_id = ana_...
```

但目前响应体没有返回这个 request_id。

如果要让前端日志和后端日志完全串起来，建议下一步让后端响应增加：

```json
{
  "request_id": "ana_20260711_135536_7cb94ce6"
}
```

然后前端记录：

```text
frontend_request_id
backend_request_id
```

第一版可先这样：

```text
frontend_request_id = fe_req_...
```

后端没有返回 request_id 时，前端日志只记录自己的请求 ID。

第二版再补：

```text
后端返回 request_id。
前端把 backend_request_id 写进日志。
```

## 9. 推荐实现结构

在 `frontend/app.js` 中新增：

```text
frontendLogger
```

建议结构：

```js
const frontendLogger = {
  sessionId: getOrCreateSessionId(),
  buffer: [],
  enabled: true,
  log(event, fields = {}) {},
  info(event, fields = {}) {},
  warn(event, fields = {}) {},
  error(event, fields = {}) {},
  flush() {},
};
```

辅助函数：

```text
hashText(value)
safeJsonSize(value)
truncate(value, maxChars)
nowIso()
newFrontendRequestId()
```

注意：

```text
前端日志失败不能影响页面使用。
所有 logger 内部必须 try/catch。
```

## 10. fetchJson 改造方案

当前所有 API 请求都走：

```js
async function fetchJson(path, options = {}) { ... }
```

这是最佳接入点。

建议改造成：

```text
1. 生成 frontend_request_id。
2. 记录 api.request_start。
3. fetch 成功后记录 api.request_ok。
4. fetch 失败后记录 api.request_error。
5. duration_ms 统一记录。
```

事件：

```text
api.request_start
api.request_ok
api.request_error
```

字段：

```text
method
path
status
status_text
duration_ms
frontend_request_id
response_size
error_type
error_summary
```

不要记录完整 response body。
即使当前 `fetchJson()` 抛错会读取后端响应文本，日志里也只允许保存脱敏后的摘要。

## 11. 本地 buffer 与 flush 策略

第一版建议保留最近 300 条前端日志：

```text
frontendLogger.buffer max = 300
```

超过后丢弃最早的：

```text
FIFO
```

本地调试时可以暴露：

```js
window.frontendLogger = frontendLogger
```

这样可以在浏览器 Console 输入：

```js
frontendLogger.buffer
frontendLogger.flush()
```

如果第二阶段实现 `/frontend-log`，flush 规则：

```text
1. 每 10 条自动 flush。
2. 页面隐藏 beforeunload / visibilitychange 时 flush。
3. 失败时保留 buffer，下次再试。
```

## 12. 后端 /frontend-log 设计

如果要落盘，后端新增：

```text
POST /frontend-log
```

请求：

```json
{
  "logs": [
    {
      "event": "ui.analyze.click",
      "level": "INFO",
      "session_id": "fe_xxx"
    }
  ]
}
```

后端写入：

```text
backend/logs/frontend.jsonl
```

后端校验：

```text
1. logs 必须是 list。
2. 单次最多 100 条。
3. 单条日志最大 8KB。
4. 丢弃 unknown 超大字段。
5. 后端补 server_ts。
```

不要让 `/frontend-log` 抛 500 影响页面。

如果日志写入失败：

```text
返回 ok=false，但前端页面业务不受影响。
```

## 13. 文件实施顺序

建议分 4 批做。

### 批次 1：前端本地 logger

修改：

```text
frontend/app.js
```

新增：

```text
frontendLogger
text hash
safe truncate
window.onerror
window.onunhandledrejection
```

先不改后端。

验收：

```text
打开页面后 console 能看到 ui.app.load。
点击 Analyze 后 buffer 中能看到 ui.analyze.click。
请求成功后能看到 api.request_ok。
```

### 批次 2：关键 UI 事件接入

修改：

```text
frontend/app.js
```

接入：

```text
Analyze
Diagnosis
Benchmark upload / polling
Error slice select
Promotion apply / single apply
API base save
Route change
```

验收：

```text
用户关键操作都能在 frontendLogger.buffer 中看到。
```

### 批次 3：后端 /frontend-log（第二阶段，可选）

这一批不要和第一版前端 logger 强绑定。
只有当本地 buffer 的字段确认有价值后，再做后端落盘。

修改：

```text
backend/api/main.py
backend/utils/structured_logger.py
```

新增：

```text
frontend.jsonl
log_frontend()
POST /frontend-log
```

验收：

```text
frontendLogger.flush() 后 backend/logs/frontend.jsonl 有日志。
```

### 批次 4：前后端 request_id 串联

修改：

```text
backend/api/main.py
backend/api/schemas.py
frontend/app.js
```

目标：

```text
/expand/simple 响应返回 request_id。
前端日志记录 backend_request_id。
```

验收：

```text
frontend.jsonl 中的 backend_request_id
可以在 app.jsonl / pipeline.jsonl / dependency.jsonl 中找到同一个 request_id。
```

## 14. 最小可用版本定义

MVP 做到以下内容即可：

```text
1. 前端有 session_id。
2. fetchJson 自动记录 api.request_start / ok / error。
3. Analyze 点击和结果记录在 buffer 中。
4. benchmark 上传、阶段变化、完成/失败记录在 buffer 中。
5. promotion 写入确认、成功/失败记录在 buffer 中。
6. window.onerror / unhandledrejection 能进入 buffer。
7. 默认不记录完整临床文本。
8. 前端日志失败不影响页面功能。
9. 不记录 render()。
10. 不默认记录 textarea text_change。
```

如果做后端落盘版，MVP 再增加：

```text
11. /frontend-log 能写 backend/logs/frontend.jsonl。
12. 单次上报失败不影响业务请求。
```

## 15. 面试表达

可以这样说：

> 后端日志只能说明服务端链路是否执行成功，但不能说明用户在浏览器里做了什么。因此我又设计了一层前端结构化日志。前端日志不记录完整病历文本，只记录 text_hash、text_len、页面 route、API 地址、用户动作、请求耗时和错误摘要。所有 API 请求都通过统一的 fetchJson 出口记录 request_start、request_ok、request_error；Analyze、Benchmark、Fallback Promotions 等关键交互记录 UI event。第一版先保存在浏览器 buffer 和 console 中，不记录 render，也不默认记录每次输入变化。等字段确认有价值后，再通过 `/frontend-log` 批量落到 `backend/logs/frontend.jsonl`，并和后端的 request_id / job_id 串联起来。

## 16. 一句话总结

```text
后端日志看“系统怎么跑”，前端日志看“用户怎么点、浏览器怎么请求、页面怎么失败”。
```

前端日志第一版要轻：

```text
统一入口、结构化、脱敏、可关闭、失败不影响业务。
```
