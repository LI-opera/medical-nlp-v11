# Medical NLP 项目地图 + 运行流程图

> 用途：面试 RAG / Agent 工程师前的项目自查。读完这一份，你应该能在白板上画出整条链路，并说清楚每个模块为什么存在、面试官会从哪里追问。
>
> 一句话定位：**把"医学缩写扩写 + SNOMED 术语标准化"做成一条带候选召回、RAG 检索、LLM 校验、反思重试的可评估后端服务。**

---

## 0. 30 秒电梯陈述（先背这段）

这是一个医学文本标准化后端。用户发一段临床文本（含 SOB、HTN 这类缩写），系统先用**候选召回 + coverage 过滤**确定每个缩写的可能展开，再让 **LLM 基于候选**把缩写扩成全称；扩写后的文本走 **NER → 向量检索（bge-m3 + Milvus）→ 规则重排**做 SNOMED 标准化；然后 **LLM Verifier** 逐句逐项校验扩写是否改变原意、是否被 SNOMED 支持；不通过就触发 **Reflection 反思修正并重试**（最多 2 次）。整套服务用 **FastAPI** 暴露、**LangGraph** 做了节点化重构、有 **Benchmark + Error Analysis** 离线评估、用 **Docker** 部署。

面试三个关键词：**RAG（双路检索）、Agent（Verify→Reflect→Retry 闭环）、Evaluation（benchmark + 错误归因）**。

## 3. 模块地图（按调用层级，从入口到底层）

每个模块给：**它做什么 / 输入→输出 / 被谁调 / 调谁 / 面试重点**。带 ⭐ 的是面试必问。

### ⭐ ABBRService —— 主编排器
`services/abbr_service.py`（661 行，全项目大脑）

- **做什么**：在 `__init__` 里把所有子服务实例化复用（LLM、标准化器、检索器、校验器、反思器、候选三件套）；对外暴露主链路 `expand_verify_with_retry()`。
- **输入→输出**：`clinical_text, max_retries` → `{original_text, final_expanded_text, success, attempts, final_result}`。
- **被谁调**：`api/main.py`、`run_benchmark.py`、`evaluate_abbr_expansion.py`、`graph` 节点、各测试脚本。
- **调谁**：下面所有服务。
- **核心方法**：
  - `simple_llm_expansion()` —— 召回候选 → 拼 prompt → LLM 扩写 → 解析 JSON（带 markdown 清洗 + parse_error 兜底）。
  - `_get_abbreviation_candidates()` —— 缩写候选召回入口（见第 4.1 节）。
  - `_should_consider_abbreviation()` —— **关键 gate**：已知缩写放行；未知但全大写且 ≤8 字符走 fallback；小写未知词跳过。降低噪声/过度扩写。
  - `expand_verify_with_retry()` —— **主链路**：扩写 → 若无有效 expansion 直接失败返回（避免空转）→ 标准化 → 逐 mapping 做 SNOMED 检索 → Verify → 通过即返回，否则 Reflect 后重试。
- **面试重点**：(1) 为什么把子服务放 `__init__` 复用（避免重复加载模型）；(2) retry 上限 `max_retries+1` = 1 正常 + 2 反思；(3) `coverage_failed_no_valid_expansion` 早停设计；(4) `_filter_mappings_by_context_support` 当前被注释（V9 稳定版禁用 MappingSupportVerifier，属实验模块）——能讲清"为什么砍掉"是加分项。

### ABBRCandidateRetriever —— 主候选召回
`services/abbr_candidate_retriever.py`

- **做什么**：从本地 `ABBR_CANDIDATES` 词典按缩写查候选展开。纯字典查找，无 LLM、零延迟。
- **输入→输出**：`abbreviation` → `[{abbreviation, expansion}]`。
- **面试重点**：第一层召回为什么用规则/词典而不是直接 LLM——**确定性 + 省 token + 可控**。

### ABBRCandidateFallbackRetriever —— 兜底候选（LLM）
`services/abbr_candidate_fallback_retriever.py`

- **做什么**：本地词典查不到时，用 LLM 结合上下文**生成候选列表**。注意——只生成候选，不直接改写原句。
- **输入→输出**：`abbreviation, context_text` → `{abbreviation, candidates[], reason}`。
- **面试重点**：召回的"二级兜底"思想；为什么职责严格限制为"只产候选不改句"（解耦，便于后续 coverage 统一裁决）。

### ABBRCandidateCoverageEvaluator —— 候选覆盖度评估
`services/abbr_candidate_coverage_evaluator.py`

- **做什么**：用 LLM 判断候选集中哪些被上下文支持，输出 `plausible_candidates` 供过滤。
- **输入→输出**：`original_text, abbreviation, candidates` → `{coverage_ok, confidence, plausible_candidates, reason, issues}`。
- **面试重点**：这是**召回与生成之间的"闸门"**——先裁决"上下文到底支不支持"，再让扩写 LLM 在 `filtered_candidates` 里选，显著抑制过度扩写/歧义误选。

### ⭐ MedicalStandardizer —— 标准化编排
`services/medical_standardizer.py`

- **做什么**：组合 NER + 检索，对整句抽实体并为每个实体召回 SNOMED 候选。
- **输入→输出**：`text` → `{input_text, entities:[{entity, entity_label, entity_score, candidates[]}]}`。
- **调谁**：`NERService`、`MedicalRetriever`。

### NERService —— 医学实体识别
`services/ner_service.py`

- **做什么**：加载 HuggingFace `Clinical-AI-Apollo/Medical-NER` pipeline 抽实体，并**合并相邻实体**（如 chest + pain → chest pain）。
- **输入→输出**：`text` → `[{text, label, score, start, end}]`。
- **面试重点**：`_merge_adjacent_entities` 的合并规则（BIOLOGICAL_STRUCTURE+SIGN_SYMPTOM 等），用 start/end 间隙判断是否连续。

### ⭐ MedicalRetriever —— RAG Retriever 层
`services/medical_retriever.py`

- **做什么**：调 `StdService` 做向量检索 → **规则重排 `_rerank_results`** → 按 domain/score 过滤 → 包成 document 结构。
- **输入→输出**：`query, top_k, domain_filter, score_threshold` → `[{page_content, metadata{concept_id, concept_name, domain_id, concept_code, score, rerank_score}}]`。
- **重排规则**：完全相等 +0.5 / query 开头 +0.3 / 包含 +0.15；术语过长按字数扣分（>10 −0.25 …）。
- **面试重点**：**为什么向量召回后还要规则重排**——bge-m3 语义相似 ≠ 术语精确匹配，短而精确的标准术语应优先；这是典型的"向量召回 + 业务规则 rerank"两阶段检索。

### ⭐ StdService —— Embedding + Milvus 检索底座
`services/std_service.py`

- **做什么**：`__init__` 里建 embedding 模型、连 Milvus、`load_collection`；`search_similar_terms` 把 query 向量化后做向量搜索。
- **输入→输出**：`query, limit` → `[{input, concept_id, concept_name, domain_id, concept_code, FSN, score}]`。
- **环境变量**：`MILVUS_URI`、`MILVUS_COLLECTION_NAME`（默认 `concepts_only_name`）。
- **面试重点**：`embed_query` → `client.search(anns_field="vector")`；`score = round(distance,4)`（注意 distance 的语义，越小越近还是越大越近要和索引度量对齐）。

### ABBVerifier —— 扩写质量校验（LLM）
`services/abbr_verifier.py`

- **做什么**：句子级 + mapping 级双层校验，汇总 `overall_valid`。
- **输入→输出**：`original_text, expanded_text, mapping_standardizations` → `{sentence_validity, mapping_validations[], overall_valid}`。
- **通过条件**：句子有效 **且** mapping 数 > 0 **且** 每个 mapping 都有效。
- **面试重点**：校验标准是"只扩缩写、不加诊断/症状、扩写被 SNOMED 候选支持"——这是 Agent 自检环节，把 LLM 既当生成器又当裁判（LLM-as-judge）。

### ABBRReflectionService —— 反思修正（LLM）
`services/abbr_reflection_service.py`

- **做什么**：Verify 不通过时，基于原文 + 上次扩写 + 校验结果 + 候选集，生成修正版扩写与 mapping。
- **输入→输出**：`original_text, previous_expanded_text, verification, abbreviation_candidates` → `{revised_expanded_text, revised_mappings, reason}`。
- **面试重点**：**这是把 pipeline 变成 Agent 的关键** —— 失败信号回灌 → 重新生成 → 再校验，形成闭环。

### MappingSupportVerifier —— 上下文支持校验（实验/已禁用）
`services/mapping_support_verifier.py`

- **做什么**：判断当前文本是否支持某个 `abbreviation → expansion` 映射。
- **当前状态**：在 `__init__` 已初始化，但主链路 `_filter_mappings_by_context_support` 调用**被注释**（V9 稳定版禁用）。
- **面试重点**：能说清"为什么留代码但禁用"（减少一次 LLM 调用 / 与 coverage 职责重叠 / 稳定性权衡）就是真正理解项目，而不是 AI 堆的功能。

### FastAPI App —— 服务入口
`api/main.py` + `api/schemas.py`

- **路由**：`GET /`、`GET /health`、`GET /benchmark/summary`、`GET /error-analysis/summary`、`POST /expand/simple`（完整 debug 版 `/expand` 被注释）。
- **懒加载**：`service=None` → 首次 `/expand/simple` 才 `get_service()` 实例化 `ABBRService`（避免启动即加载 NER/embedding/Milvus/LLM）。
- **面试重点**：懒加载动机、`/health` 只查 API 不查 Milvus、`/expand/simple` 只返回 `success/expanded_text/mappings`（不返 verification/attempts）。

### LangGraph Workflow —— 节点化重构
`graph/abbr_graph_state.py` / `abbr_graph_nodes.py` / `abbr_graph.py`

- **做什么**：把 `ABBRService` 的线性流程拆成 `expand → standardize → verify →(should_continue)→ reflect → standardize …` 四节点共享一个 `ABBRGraphState`。
- **当前状态**：图能独立构建/测试，但**未接入 FastAPI**，API 仍直接调 `expand_verify_with_retry()`。
- **面试重点**：为什么要图化（可视化、条件边、可扩展并行/分支）；`should_continue()` 的退出条件 = 成功 或 超过 max_retries。

### Benchmark + Error Analysis —— 评估闭环
`evaluation/run_benchmark.py` / `error_analysis_report.py`

- **Benchmark**：遍历 `ABBR_BENCHMARK_CASES` → 跑主链路 → `compare_mappings` + `compare_text_contains` → `accuracy = correct/total` → 写 `benchmark_results.json`。
- **Error Analysis**：读 benchmark 结果 → 筛失败 case → `classify_error_type` + `classify_taxonomy`（Over/Under Expansion、Wrong Disambiguation、Semantic Preservation Failure、Unknown）→ 写 `error_analysis_report.json`。
- **面试重点**：**有评估和错误归因是这个项目最大的亮点**。能背出错误分类体系，并举一个失败案例 + 归因，远胜于只讲功能。

## 4. 两条检索链路（RAG 面试核心）

### 4.1 缩写候选召回链（先 gate 后三级召回）

```text
文本 → 切 token → _should_consider_abbreviation(gate)
     → 主候选库(规则/词典)  primary
     → 查不到则 LLM fallback 生成候选
     → CoverageEvaluator(LLM) 裁决 plausible_candidates
     → filtered_candidates 交给扩写 LLM
```

### 4.2 SNOMED 标准概念检索链（向量 RAG + 规则重排）

```text
扩写文本 / 每个 expansion
     → NER 抽实体
     → MedicalRetriever.retrieve
     → StdService.search_similar_terms
     → bge-m3 embed_query → Milvus 向量搜索 TopK
     → _rerank_results(规则重排) → domain/score 过滤
     → documents(带 metadata)
```

### 4.3 Milvus 库构建（离线，`tools/create_milvus_db.py`）

```text
读 SNOMED_5000.csv → 建 bge-m3 → 测维度 → 连 Milvus
→ drop 旧 collection → 建 schema → 建向量索引
→ 逐行 embed concept_name → insert → flush → load
```

## 5. 面试速记表（背这张就够应付 80%）

| 维度 | 一句话答案 |
| --- | --- |
| 最重要的 5 个类 | `ABBRService` / `MedicalRetriever` / `StdService` / `ABBVerifier` / `ABBRGraphNodes` |
| 最重要的 5 个函数 | `expand_verify_with_retry` / `_get_abbreviation_candidates` / `retrieve` / `verify_mappings` / `build_abbr_graph` |
| RAG 体现在哪 | 缩写候选召回（规则+LLM 兜底+coverage）+ SNOMED 向量检索（bge-m3+Milvus+规则重排） |
| Agent 体现在哪 | Verify → Reflect → Retry 闭环（失败信号回灌重生成） |
| 工程亮点 | 懒加载、JSON 解析兜底、coverage 早停、Benchmark + Error Taxonomy 评估闭环 |
| 诚实可讲的不足 | LangGraph 未接 API；MappingSupportVerifier 被禁用；SNOMED 仅样例数据；评估集偏小 |

> 提醒：面试官常追问"哪部分是你自己想的设计"。建议把 **gate 规则、coverage 闸门、规则重排、retry 早停、错误分类体系** 作为你能深入解释的"设计决策"，这些都是有 trade-off、能讲故事的点。

## 6. 运行流程图（盒式分段风格）

> 下图模仿你给的那张"开诊断会话"流程图的排版：**分段标题 + 盒子 + ▼ 箭头 + 旁注**。
> 一次性发生的事用 `═══` 段，循环用底部回流注释。建议等宽字体查看。

```text
═══════════════ 一次性：启动 API 服务（进程级 · 静态 · 不变）═══════════════

FastAPI App（api/main.py · Uvicorn 加载 api.main:app）
   app = FastAPI(...)                          ← 只注册路由，不碰模型
   routes: GET / · /health · /benchmark/summary · /error-analysis/summary
           POST /expand/simple                 ← 主业务入口
   service = None                              ← 懒加载哨兵
        │
        ▼  首个 POST /expand/simple 到达时才触发
┌──────────────────────────────────────────────────────────┐
│ get_service() → ABBRService.__init__()（一次性重对象加载）   │
│   ChatDeepSeek(temperature=0)   MedicalStandardizer        │
│   MedicalRetriever → StdService(embed bge-m3 + Milvus.load)│
│   ABBVerifier   ABBRReflectionService                     │
│   Candidate / Fallback / Coverage 三件套                    │
│   MappingSupportVerifier（已初始化但主链路禁用）             │
└──────────────────────────────────────────────────────────┘
        │
        ▼  之后所有请求复用同一个 service 实例

═══════════════ 一次请求：expand_verify_with_retry（attempt 循环）═══════════════

请求体「Patient has SOB and CP.」
        │
        ▼  POST /expand/simple { text, max_retries=2 }

──────── ① 缩写候选召回：先 gate 后三级（确定下哪些缩写、给哪些候选）────────
_get_abbreviation_candidates(text)
   切 token → _should_consider_abbreviation 「gate」
        ├─ 已知缩写(SOB/CP…)            → 放行
        ├─ 未知但全大写 ≤8 字符(AKI/XYZ) → 放行走 fallback
        └─ 小写未知词                   → 跳过（防噪声/防过度扩写）
        ▼
   路1 ABBRCandidateRetriever  主候选库(词典 · 规则 · 零延迟)
   路2 ABBRCandidateFallbackRetriever  本地空→LLM 生成候选(只产候选不改句)
        ▼
   CoverageEvaluator(LLM)「闸门」: 判断上下文支持谁 → plausible_candidates
        ▼
   filtered_candidates = 候选 ∩ plausible       ← 交给扩写 LLM 的最终候选集
        │
        ▼

──────── ② LLM 扩写：在候选里选，不自由发挥（simple_llm_expansion）────────
   prompt(text + filtered_candidates + 8 条规则) → ChatDeepSeek
   解析 JSON（去 markdown ```），parse_error 兜底
   产出 → expanded_text + mappings[{abbreviation, expansion, source}]
        │
        ▼
   ┌─ 早停哨兵：valid_mappings(有 expansion 的) 为空？
   │     是 → stop_reason="coverage_failed_no_valid_expansion"
   │          success=False 直接返回（不空转重试）
   └─ 否 → 继续
        │
        ▼

──────── ③ 标准化 + 逐项 SNOMED 检索（RAG · 唯一用向量库的地方）────────
   MedicalStandardizer.standardize(expanded_text)
        └ NERService 抽实体(+相邻实体合并: chest+pain→chest pain)
   对每个 expansion 单独检索：
        MedicalRetriever.retrieve(query=expansion, top_k=10, score≥0.6)
            └ StdService.search_similar_terms
                  bge-m3 embed_query → Milvus 向量搜索 TopK
            └ _rerank_results「规则重排」
                  =query +0.5 · 开头 +0.3 · 包含 +0.15 · 过长按字数扣分
            └ domain / score 过滤 → 取 Top-3 候选
   产出 → standardization + mapping_standardizations[{abbr, expansion, candidates}]
        │
        ▼

──────── ④ 校验：LLM-as-judge 双层（ABBVerifier.verify_mappings）────────
   输入 original_text + expanded_text + mapping_standardizations
   sentence_validity「只扩缩写? 没加诊断? 保留否定/不确定?」
   mapping_validations「每项被 SNOMED 候选支持?」
   overall_valid = 句子有效 且 mapping 数>0 且 每项都有效
        │
        ▼
   ◇ overall_valid == True ? ───是──▶ success=True，返回 final_result ──▶ 出口⑥
        │否
        ▼
   ◇ attempt_index >= max_retries(2) ? ──是──▶ success=False，返回 ──▶ 出口⑥
        │否
        ▼

──────── ⑤ 反思修正：把失败信号回灌重生成（ABBRReflectionService.reflect）────────
   输入 original_text + 上次 expanded_text + verification + candidates
   产出 revised_expanded_text / revised_mappings / reason
        │
        ▼
   current_expanded_text = revised_expanded_text
   revised_mappings 非空 → current_mappings 更新
        │
        └───────────▲ 回到 ③ 标准化，开始下一轮 attempt（最多 1 正常 + 2 反思）

──────── ⑥ 出口：组装响应 ────────
   final_result = attempts[-1]
   SimpleExpandResponse → { success, expanded_text, mappings }
        │
        ▼
┌─────────────────────────────────────────────┐
│ HTTP 200 JSON（调用方看到的）                  │
│   success      : true / false                │
│   expanded_text: "Patient has shortness of   │
│                   breath and chest pain."     │
│   mappings     : [SOB→…, CP→…]               │

## 7. 怎么用这份文档准备面试
