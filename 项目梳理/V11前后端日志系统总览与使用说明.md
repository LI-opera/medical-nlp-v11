# V11 前后端日志系统总览与使用说明

本文说明 medical-nlp V11 当前已经实现的日志系统：它记录什么、写在哪里、怎么查看、怎么把一次前端点击和后端 pipeline 串起来。

这不是面向医疗人员的页面功能，而是面向开发调试、项目交付和面试讲解的工程能力。

---

## 1. 日志系统解决什么问题

medical-nlp V11 的一次分析不是一个简单函数调用，而是一条完整链路：

```text
浏览器页面
  -> FastAPI 接口
  -> ABBRService 缩写扩写主链路
  -> fallback / coverage / verifier LLM
  -> StdService 向量检索与标准化
  -> benchmark / error analysis / fallback promotions
```

没有日志时，出问题只能靠猜：

```text
用户到底有没有点击按钮？
前端有没有发请求？
后端有没有收到请求？
Milvus / LLM 是否可用？
pipeline 卡在哪一步？
benchmark 跑到哪个 case？
primary 候选库是否真的被写入？
```

现在日志系统把这些问题拆成两层：

```text
前端日志：回答“用户和浏览器发生了什么”。
后端日志：回答“API、pipeline、LLM、Milvus、benchmark、写入动作发生了什么”。
```

## 4. 后端各日志文件的作用

### 4.1 app.jsonl

记录 API 层和服务层入口事件。

主要看：

```text
api.expand_simple.start
api.expand_simple.end
api.expand_simple.error
```

适合回答：

```text
后端有没有收到请求？
接口是否成功返回？
接口总耗时多少？
返回的 success / expansion_success / standardization_success 是什么？
```

---

### 4.2 pipeline.jsonl

记录 ABBRService 主链路。

主要看：

```text
pipeline.start
pipeline.final
```

适合回答：

```text
识别出几个目标缩写？
几个扩写成功？
几个 CODED？
几个 WITHHELD？
几个 NOT_EXPANDED？
本次失败是扩写失败还是标准化失败？
```

典型字段：

```text
request_id
frontend_request_id
job_id
case_id
target_count
expanded_count
coded_count
withheld_count
not_expanded_count
success
expansion_success
standardization_success
duration_ms
```

---

### 4.3 dependency.jsonl

记录外部依赖和重资源调用。

主要看：

```text
dependency.embedding.load_start / load_ok / load_error
dependency.milvus.connect_start / connect_ok / connect_error
dependency.milvus.search_start / search_ok / search_error
dependency.llm.create_ok
dependency.llm.call_start / call_ok / call_error
```

适合回答：

```text
Milvus 是否连通？
collection 是否加载？
向量检索是否成功？
DeepSeek / LLM 是否调用成功？
Embedding 是否加载成功？
```

---

### 4.4 benchmark.jsonl

记录 benchmark 运行与后处理流程。

主要看：

```text
benchmark.job.start
benchmark.case.start
benchmark.case.end
benchmark.case.error
benchmark.job.end
benchmark.postprocess.start
benchmark.postprocess.stage_ok
benchmark.postprocess.end
benchmark.api_job.start
benchmark.api_job.end
benchmark.api_job.error
```

适合回答：

```text
benchmark 跑到第几个 case？
哪个 case 报错？
上传 benchmark cases 的异步 job 当前在哪个阶段？
error_analysis / triage / fallback promotions 后处理是否完成？
```

---

### 4.5 audit.jsonl

记录会改变项目数据的写入动作。

当前主要用于：

```text
fallback promotions 写入 primary 缩写候选库
```

适合回答：

```text
谁触发了候选库写入？
写入目标文件是什么？
新增多少条？
跳过多少条？
```

---

### 4.6 frontend.jsonl

记录浏览器前端手动 flush 到后端后的事件。

主要看：

```text
ui.app.load
ui.health.check_start
ui.health.check_ok
ui.analyze.click
api.request_start
api.request_ok
api.request_error
ui.analyze.result_ok
ui.benchmark.*
ui.promotions.*
frontend.error
frontend.unhandled_rejection
```

适合回答：

```text
用户是否真的点击了按钮？
前端是否真的发出了请求？
请求是否成功返回？
页面是否渲染成功？
这条前端请求对应后端哪个 request_id？
```

注意：

```text
frontend.jsonl 不是实时自动落盘。
前端默认先写入浏览器内存 buffer。
需要手动执行 window.frontendLogger.flushToServer() 后才会写入 backend/logs/frontend.jsonl。
```

## 5. 前端日志代码在哪里

核心文件：

```text
frontend/utils/frontend_logger.js
frontend/app.js
```

加载顺序：

```html
<script src="/frontend/utils/frontend_logger.js?v=__LOGGER_VERSION__"></script>
<script src="/frontend/app.js?v=__APP_VERSION__"></script>
```

说明：

```text
frontend_logger.js 必须先于 app.js 加载。
app.js 使用 window.frontendLogger 记录业务事件。
```

版本号：

```text
/app 页面会由后端根据文件修改时间自动替换 __LOGGER_VERSION__、__APP_VERSION__、__STYLE_VERSION__。
所以修改 frontend_logger.js / app.js / styles.css 后，正常不需要手动改 URL 参数。
```

如果页面仍然加载旧文件：

```text
Ctrl + F5 强制刷新
确认打开的是 http://127.0.0.1:8001/app
不要打开 http://127.0.0.1:8001/
```

## 6. 怎么查看前端日志

打开网页后按：

```text
F12 -> Console
```

查看当前前端日志 buffer：

```js
window.frontendLogger.buffer
```

表格方式查看：

```js
window.frontendLogger.print()
```

只看最近 10 条：

```js
window.frontendLogger.print(10)
```

清空前端日志 buffer：

```js
window.frontendLogger.clear()
```

手动落盘到后端：

```js
await window.frontendLogger.flushToServer()
```

落盘后查看：

```text
backend/logs/frontend.jsonl
```

## 7. Console 自动输出规则

当前规则：

```text
INFO / WARNING:
  默认只进入 frontendLogger.buffer，不自动刷屏。

ERROR:
  进入 frontendLogger.buffer，同时 console.error 自动显示。

手动查看:
  window.frontendLogger.print()
```

为什么这样设计：

```text
INFO 如果全部自动打印，Analyze / Benchmark 操作几次后 Console 会非常吵。
ERROR 自动显示是必要的，因为前端异常应该第一时间暴露。
INFO 仍然完整保存在 buffer 里，需要时手动查看。
```

临时打开 INFO 自动输出：

```js
localStorage.setItem("medicalNlpFrontendLogConsole", "1")
```

然后刷新页面。

关闭 INFO 自动输出：

```js
localStorage.removeItem("medicalNlpFrontendLogConsole")
```

然后刷新页面。

## 8. 隐私和日志大小控制

第一版日志系统刻意不记录：

```text
完整临床文本
完整 Raw JSON
完整后端 response body
完整 benchmark 上传文件内容
textarea 每次输入变化
render() 级别的高频事件
```

Analyze 文本只记录：

```text
text_len
text_hash
```

这样可以做到：

```text
能定位是不是同一次输入
不把完整医疗文本直接写入日志
```

前端落盘接口 `/frontend-log` 也会过滤：

```text
text
raw_json
analysis_result
response_body
```

并截断过长字符串，避免日志暴涨。

## 9. 前后端 request_id 如何串联

一次 Analyze 请求会有两个关键 ID：

```text
frontend_request_id
  前端生成，请求 /expand/simple 时通过 X-Frontend-Request-Id 请求头传给后端。

request_id / backend_request_id
  后端生成，例如 ana_20260711_194217_fdeb684e。
  后端写入 app / pipeline / dependency 日志，并返回给前端。
  前端日志里保存为 backend_request_id。
```

串联方式：

```text
frontend.jsonl
  api.request_ok
  frontend_request_id = fe_req_...
  backend_request_id = ana_...

app.jsonl
  api.expand_simple.start / end
  request_id = ana_...
  frontend_request_id = fe_req_...

pipeline.jsonl
  pipeline.start / final
  request_id = ana_...
  frontend_request_id = fe_req_...

dependency.jsonl
  dependency.llm.* / dependency.milvus.*
  request_id = ana_...
```

这样就能从“用户点击”一路追到“后端哪个环节慢或失败”。

## 10. 一次 Analyze 怎么排查

第一步，看前端：

```js
window.frontendLogger.print()
```

确认有没有：

```text
ui.analyze.click
api.request_start /expand/simple
api.request_ok /expand/simple
ui.analyze.result_ok
backend_request_id = ana_...
```

第二步，看 API：

```powershell
Select-String -Path backend/logs/app.jsonl -Pattern "ana_..."
```

第三步，看 pipeline：

```powershell
Select-String -Path backend/logs/pipeline.jsonl -Pattern "ana_..."
```

第四步，看依赖：

```powershell
Select-String -Path backend/logs/dependency.jsonl -Pattern "ana_..."
```

常见判断：

```text
没有 ui.analyze.click
  -> 用户没有点按钮，或前端事件绑定有问题。

有 ui.analyze.click，但没有 api.request_start
  -> 前端点击逻辑到请求逻辑之间有问题。

有 api.request_start，但 app.jsonl 没有对应 request
  -> API 地址、端口、网络或 CORS 问题。

app.jsonl 有 start，但没有 end
  -> 后端接口中途异常或卡住。

pipeline.final not_expanded_count > 0
  -> 扩写阶段失败。

pipeline.final withheld_count > 0
  -> 扩写成功，但标准化不敢绑定。

dependency.jsonl 有 dependency.llm.call_error
  -> LLM 调用失败。

dependency.jsonl 有 dependency.milvus.search_error
  -> Milvus / 向量检索失败。
```

## 12. 暂不做 P3 日志页面的原因

P3 原本可以做“前端日志查看页面”或“Request Trace Viewer”。

但当前项目页面主要面向医疗人员：

```text
Analyze
Benchmark Overview
Error Analysis
Fallback Promotions
```

医疗人员不需要直接看前后端 JSONL 日志。  
如果 P3 只是把 JSON 文件搬到网页上，价值不高，还会干扰主产品体验。

因此当前建议：

```text
P3 暂缓。
保留日志文件和 request_id 串联能力。
需要排查问题时，开发者通过日志文件和 Console 查看。
未来如果要做 P3，应做成隐藏的开发者 Trace Viewer，而不是普通用户页面。
```

未来 P3 更合理的形态：

```text
/debug/trace

输入 request_id:
  ana_...

页面汇总:
  前端点击
  API 请求
  pipeline start/final
  dependency LLM/Milvus 调用
  总耗时
  失败原因
```

## 13. 面试说法

可以这样讲：

> 我给 V11 增加了一套轻量结构化日志系统。前端日志记录用户操作、API 请求、页面结果和浏览器错误；后端日志按 app、pipeline、dependency、benchmark、audit 分层记录。Analyze 请求会由前端生成 frontend_request_id，通过请求头传给后端；后端生成 request_id 并返回前端，所以一次用户点击可以从 frontend.jsonl 串到 app.jsonl、pipeline.jsonl 和 dependency.jsonl。日志默认不记录完整临床文本，只记录 text_len 和 text_hash，避免把敏感文本写入日志。当前没有把日志暴露给普通用户页面，因为医疗人员不需要看工程日志；后续如果需要，会做成隐藏的开发者 Trace Viewer。

## 14. 一句话总结

```text
前端日志回答“用户和浏览器发生了什么”，后端日志回答“服务端和模型链路发生了什么”；request_id 把两者串起来，形成完整的可观测性闭环。
```
