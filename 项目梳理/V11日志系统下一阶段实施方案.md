# V11 日志系统下一阶段实施方案

本文面向 medical-nlp V11 当前已经完成的日志系统第一版，回答下一步该怎么继续做。

当前已完成：

```text
1. 后端 JSONL 日志第一版。
2. 前端本地 frontendLogger.buffer。
3. 前端日志工具已抽离到 frontend/utils/frontend_logger.js。
4. /app 已支持 app.js / frontend_logger.js / styles.css 自动版本号。
```

当前还没完成：

```text
1. 前端日志查看页面。
2. Request Trace 聚合页面。
3. 生产级远程日志平台。
```

截至本次实现，P1 / P2 已完成：

```text
P1：前后端 request_id 串联
P2：前端日志落盘到 backend/logs/frontend.jsonl
```

后续继续推进：

```text
P3：前端日志查看页面 / Request Trace 页面
P4：生产级远程日志平台
```

从专业工程角度看，这 4 件事不应该一起做。
建议按以下优先级推进：

```text
P1：前后端 request_id 串联
P2：前端日志落盘
P3：前端日志查看页面
P4：生产级远程日志平台
```

原因：

```text
request_id 是串联所有日志的主轴。
没有 request_id，frontend.jsonl 即使落盘，也只能看“前端发生了什么”，很难和后端某次 pipeline 对上。
```

---

## 1. 当前日志系统现状

### 1.1 后端现状

后端日志代码：

```text
backend/utils/structured_logger.py
backend/utils/trace_context.py
```

后端日志文件：

```text
backend/logs/app.jsonl
backend/logs/dependency.jsonl
backend/logs/pipeline.jsonl
backend/logs/benchmark.jsonl
backend/logs/audit.jsonl
```

当前 `/expand/simple` 已经在后端内部生成：

```text
request_id = ana_...
```

代码位置：

```text
backend/api/main.py
expand_abbreviation_simple()
```

它已经进入后端日志：

```text
app.jsonl
pipeline.jsonl
dependency.jsonl
```

但目前问题是：

```text
后端响应体没有把 request_id 返回给前端。
```

所以前端现在无法记录：

```text
backend_request_id
```

### 1.2 前端现状

前端日志代码：

```text
frontend/utils/frontend_logger.js
frontend/app.js
```

当前前端日志只保存在浏览器内存：

```js
window.frontendLogger.buffer
window.frontendLogger.print()
```

当前前端每次请求会生成：

```text
frontend_request_id = fe_req_...
```

但目前问题是：

```text
frontend_request_id 只存在前端日志里。
后端不知道这个 frontend_request_id。
后端 request_id 也没有返回给前端。
```

因此现在前后端还没有形成完整闭环。

---

## 2. 总体目标

下一阶段日志系统目标不是“日志更多”，而是“日志能串起来”。

理想链路：

```text
用户点击 Analyze
  -> 前端生成 frontend_request_id
  -> 请求头带 X-Frontend-Request-Id
  -> 后端生成 backend request_id
  -> 后端日志同时记录 request_id + frontend_request_id
  -> 后端响应返回 request_id
  -> 前端 api.request_ok 记录 backend_request_id
  -> 前端日志落盘 frontend.jsonl
  -> 日志查看页面可按 request_id 查一整条链路
```

最终要能回答：

```text
这次用户点击，对应哪次后端请求？
这次后端请求对应哪条 pipeline？
这次 pipeline 调了哪些 LLM / Milvus？
这次结果是否 CODED / WITHHELD / NOT_EXPANDED？
如果失败，失败在哪一层？
```

## 3. P1：前后端 request_id 完整串联

### 3.1 为什么先做 P1

这是最值得先做的一步。

如果先做 frontend.jsonl，但没有 request_id 串联，会出现：

```text
前端日志一堆。
后端日志一堆。
但两边对不上。
```

所以第一步应该先建立关联字段。

### 3.2 字段设计

前端生成：

```text
frontend_request_id
```

后端生成：

```text
request_id
```

为了前后端语义清晰，前端日志里建议叫：

```text
backend_request_id
```

最终字段关系：

```text
frontend_request_id
  前端一次 fetchJson 请求的 ID。

backend_request_id
  后端本次 API / pipeline 的 request_id。

request_id
  后端日志内部字段。
```

### 3.3 前端请求头

修改：

```text
frontend/app.js
fetchJson()
```

每次请求加请求头：

```http
X-Frontend-Request-Id: fe_req_...
```

注意：

```text
不要把 frontend_request_id 放 query string。
它是日志关联字段，用 header 更合适。
```

### 3.4 后端读取请求头

修改：

```text
backend/api/main.py
```

FastAPI 端点增加：

```python
from fastapi import Header

x_frontend_request_id: str | None = Header(default=None, alias="X-Frontend-Request-Id")
```

在后端日志里记录：

```text
frontend_request_id
```

例如：

```json
{
  "event": "api.expand_simple.start",
  "request_id": "ana_...",
  "frontend_request_id": "fe_req_..."
}
```

### 3.5 后端响应返回 request_id

修改：

```text
backend/api/schemas.py
SimpleExpandResponse
```

新增字段：

```python
request_id: str | None = None
```

修改：

```text
backend/api/main.py
expand_abbreviation_simple()
```

响应增加：

```json
{
  "request_id": "ana_..."
}
```

### 3.6 前端记录 backend_request_id

修改：

```text
frontend/app.js
fetchJson()
```

当响应 JSON 中存在：

```text
request_id
```

前端 `api.request_ok` 记录：

```text
backend_request_id: data.request_id
```

Analyze 业务事件也记录：

```text
ui.analyze.result_ok
backend_request_id
```

### 3.7 P1 验收标准

输入：

```text
The patient has SOB and CP.
```

前端 Console：

```js
window.frontendLogger.print()
```

应看到：

```text
api.request_start
frontend_request_id = fe_req_...

api.request_ok
frontend_request_id = fe_req_...
backend_request_id = ana_...

ui.analyze.result_ok
backend_request_id = ana_...
```

后端日志：

```text
backend/logs/app.jsonl
backend/logs/pipeline.jsonl
backend/logs/dependency.jsonl
```

应能按：

```text
request_id = ana_...
```

找到同一次请求。

## 4. P2：前端日志落盘到 backend/logs/frontend.jsonl

### 4.1 做 P2 的前提

建议 P1 完成后再做 P2。

因为落盘后的前端日志应该包含：

```text
frontend_request_id
backend_request_id
session_id
route
event
duration_ms
```

否则落盘价值不够大。

### 4.2 后端新增接口

新增：

```text
POST /frontend-log
```

请求体：

```json
{
  "logs": [
    {
      "ts": "2026-07-11T15:00:00.000Z",
      "level": "INFO",
      "event": "ui.analyze.click",
      "session_id": "fe_...",
      "frontend_request_id": "fe_req_...",
      "backend_request_id": "ana_...",
      "route": "analyze"
    }
  ]
}
```

写入：

```text
backend/logs/frontend.jsonl
```

### 4.3 后端校验规则

必须限制：

```text
1. logs 必须是 list。
2. 单次最多 100 条。
3. 单条最大 8KB。
4. 超大字段截断。
5. 不允许把完整临床文本写入。
6. 后端补 server_ts。
```

失败原则：

```text
/frontend-log 失败不能影响页面业务。
```

建议返回：

```json
{
  "ok": true,
  "accepted": 10,
  "dropped": 0
}
```

即使写入失败，也可以返回：

```json
{
  "ok": false,
  "accepted": 0,
  "dropped": 10,
  "error": "..."
}
```

但不要让前端主流程崩掉。

### 4.4 前端 flush 策略

修改：

```text
frontend/utils/frontend_logger.js
```

新增：

```js
frontendLogger.flushToServer()
```

建议规则：

```text
1. buffer 达到 20 条时自动 flush。
2. ERROR 出现时触发一次 flush。
3. 页面隐藏 visibilitychange 时 flush。
4. beforeunload 时尝试 navigator.sendBeacon。
5. flush 成功后只删除已发送的日志。
6. flush 失败保留 buffer，后续重试。
```

第一版可以先简单：

```text
手动调用 window.frontendLogger.flushToServer()
```

确认稳定后再自动 flush。

### 4.5 P2 验收标准

操作：

```text
打开 /app
点击 Analyze
执行 window.frontendLogger.flushToServer()
```

应生成：

```text
backend/logs/frontend.jsonl
```

里面至少有：

```text
ui.app.load
ui.health.check_ok
ui.analyze.click
api.request_start
api.request_ok
ui.analyze.result_ok
```

并包含：

```text
session_id
frontend_request_id
backend_request_id
route
event
level
```

## 5. P3：前端日志查看页面

### 5.1 什么时候做 P3

建议在 P1、P2 后做。

原因：

```text
没有落盘就没有稳定数据源。
没有 request_id 就没有查询主轴。
```

### 5.2 页面位置

建议新增导航：

```text
Logs
  Frontend Logs
  Backend Logs
  Request Trace
```

也可以第一版只做一个页面：

```text
Logs
```

### 5.3 后端查询接口

新增：

```text
GET /logs/frontend
GET /logs/backend
GET /logs/request/{request_id}
```

第一版可只做：

```text
GET /logs/frontend?limit=200
```

返回：

```json
{
  "items": [...],
  "total_returned": 200
}
```

注意：

```text
只读 backend/logs/*.jsonl 的尾部。
不要一次性读完整大文件。
```

### 5.4 前端页面功能

第一版建议展示：

```text
时间
level
event
route
frontend_request_id
backend_request_id
duration_ms
error_summary
```

筛选：

```text
level
event
route
backend_request_id
```

详情：

```text
点击一条日志，展开 JSON。
```

### 5.5 Request Trace 页面

这是最有价值的最终形态。

输入：

```text
request_id = ana_...
```

页面展示：

```text
Frontend Logs
  ui.analyze.click
  api.request_start
  api.request_ok
  ui.analyze.result_ok

Backend App Logs
  api.expand_simple.start
  api.expand_simple.end

Pipeline Logs
  pipeline.start
  pipeline.final

Dependency Logs
  llm.call_ok
  milvus.search_ok
```

这样它就不是单纯日志页面，而是“单次请求链路追踪页面”。

### 5.6 P3 验收标准

```text
1. 页面能读取 frontend.jsonl。
2. 能按 level / route / event 筛选。
3. 能按 backend_request_id 找到前端日志。
4. 能跳转或展示同 request_id 的后端日志。
```

## 6. P4：生产级远程日志平台

### 6.1 现在不建议马上做

当前项目还处在本地展示、求职交付、功能完善阶段。

生产级远程日志平台会引入：

```text
部署成本
鉴权
隐私合规
日志采样
日志保留周期
远程服务依赖
```

现在做容易过度工程化。

### 6.2 什么时候需要 P4

满足以下条件再考虑：

```text
1. 项目部署到公网或多人使用。
2. 需要长期追踪真实用户错误。
3. 需要跨机器排查问题。
4. backend/logs 本地文件已经不够用。
5. 需要告警，比如 LLM 连续失败、Milvus 断连。
```

### 6.3 可选方案

轻量方案：

```text
Docker volume + 定期打包 logs
```

中等方案：

```text
Grafana Loki + Promtail
```

云服务方案：

```text
Sentry
Datadog
OpenTelemetry Collector
CloudWatch / Azure Monitor
```

### 6.4 本项目推荐路线

短期不做 P4。

建议路线：

```text
先完成 P1 / P2 / P3。
如果未来真的部署，再评估 Sentry 或 OpenTelemetry。
```

## 8. 推荐下一步直接做什么

建议下一步直接做：

```text
P1：前后端 request_id 完整串联。
```

它的投入不大，但收益最大。

做完以后，你可以在面试里讲：

> 我给系统加了前后端链路追踪。前端每次请求生成 frontend_request_id，通过请求头传给后端；后端生成 request_id 并返回给前端。这样前端日志、API 日志、pipeline 日志和 dependency 日志可以通过同一个 request_id 串起来，定位问题时能从用户点击追到模型和向量检索层。

这比单纯说“我加了日志”更有工程含量。

## 10. 一句话总结

```text
下一阶段不要先追求“日志更多”，要先追求“日志能串起来”。
```

所以优先级应该是：

```text
request_id 串联 > frontend.jsonl 落盘 > 日志查看页面 > 远程日志平台
```
