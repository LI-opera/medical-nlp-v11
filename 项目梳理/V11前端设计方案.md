# V11 前端设计方案

## 1. 前端定位

这个前端不是营销页，也不是简单 demo 页。

它应该是一个面向项目展示、调试和评估复盘的工作台：

```text
医学缩写标准化工作台
```

前端第一版要服务 3 个目标：

```text
1. 让用户输入一句临床文本，看到缩写扩写与标准化结果。
2. 让面试官能直观看到系统链路：候选召回、扩写、标准化、成功/失败状态。
3. 让开发者能看 benchmark、错误分析和 fallback 候选沉淀结果。
```

前端第一版不应该包装成：

```text
全医学实体标准化平台
```

因为当前后端主链路处理的是“缩写扩写与标准化”，不是整句所有医学实体标准化。

## 2. LangGraph 要不要接入前端

第一版不建议把 LangGraph 作为前端核心依赖。

原因：

```text
当前生产主链路是 ABBRService.expand_verify_with_retry。
LangGraph 当前更适合流程可视化和面试讲解，不是必要的生产执行引擎。
```

前端第一版可以做“流程感”，但不需要真的调用 LangGraph。

推荐定位：

```text
生产执行：调用 FastAPI /expand/simple 或后续 /expand/debug
流程展示：根据返回的 success_breakdown / mappings / mapping_states 渲染步骤条
LangGraph：后续作为可选的“流程图解释页”或“调试视图”
```

也就是说：

```text
不用为了前端而重构后端主链路。
```

前端设计里可以保留一个二期增强：

```text
流程图视图：展示 route -> retrieve -> verify -> reflect 的标准化图。
```

但第一版不用强行接入。

## 3. 技术栈建议

建议从零创建：

```text
frontend/
```

推荐技术栈：

```text
React + Vite + TypeScript
Tailwind CSS
lucide-react 图标
recharts 图表
```

原因：

```text
React/Vite：启动快，适合项目展示。
TypeScript：结果数据结构复杂，类型能减少混乱。
Tailwind：快速做干净的工作台界面。
lucide-react：按钮、状态、导航图标统一。
recharts：benchmark 分类指标适合做柱图/环图。
```

不建议第一版上太重的 UI 框架。

## 4. 整体信息架构

第一版建议 4 个主页面：

```text
1. 缩写标准化工作台
2. Benchmark 仪表盘
3. 错误分析页面
4. fallback 候选沉淀页面
```

可选第 5 个页面：

```text
5. 流程说明 / 架构图页面
```

但第 5 页可以后置。第一版优先做真正能操作和展示结果的页面。

路由建议：

```text
/                  -> 缩写标准化工作台
/benchmark         -> Benchmark 仪表盘
/errors            -> 错误分析页面
/promotions        -> fallback 候选沉淀页面
/pipeline          -> 流程说明页，可选
```

## 5. 全局布局

采用工作台布局：

```text
左侧导航栏 + 顶部状态条 + 主内容区
```

左侧导航：

```text
Analyze
Benchmark
Errors
Promotions
Pipeline
```

顶部状态条：

```text
API Health
Milvus/LLM 提示
最后一次 benchmark 时间
当前环境
```

主内容区风格：

```text
信息密度适中
表格清晰
状态标签明显
避免大面积营销式 hero
避免过度装饰
```

视觉风格建议：

```text
背景：浅灰或近白
主色：医疗蓝绿，但不要全页面单色
状态色：
  CODED / success -> green
  WITHHELD -> amber
  NOT_EXPANDED / failure -> red
  fallback -> blue
  primary -> gray/indigo
```

## 6. 页面一：缩写标准化工作台

### 6.1 页面目标

这是前端最核心页面。

用户应该能：

```text
输入临床文本
点击 Analyze
看到扩写后的句子
看到每个缩写的扩写结果
看到标准化概念
看到成功/失败状态
```

### 6.2 页面结构

建议分成 4 个区域：

```text
输入区
结果总览区
文本对照区
结果明细区
```

#### 输入区

组件：

```text
Textarea
Analyze 按钮
Clear 按钮
示例句选择菜单
```

示例句：

```text
The patient denies SOB.
The patient has HTN and DM.
The ABG revealed respiratory acidosis with hypoxemia.
The patient has RA with joint pain and morning stiffness.
```

按钮状态：

```text
idle
loading
success
error
```

loading 时提示：

```text
Analyzing abbreviation candidates...
```

#### 结果总览区

展示：

```text
success
expansion_success
standardization_success
target_count
expanded_count
coded_count
withheld_count
not_expanded_count
```

建议做成状态卡：

```text
Overall
Expansion
Standardization
Records
```

#### 文本对照区

左侧：

```text
Original Text
```

右侧：

```text
Expanded Text
```

缩写和扩写可以做高亮：

```text
SOB -> shortness of breath
HTN -> hypertension
```

第一版如果后端没有返回 span，可以先不做精确高亮，只展示纯文本。

#### 结果明细区

使用 tabs：

```text
Mappings
Standardized Entities
Debug States
```

`Mappings` 表格：

```text
abbreviation
expansion
source
status
```

`Standardized Entities` 表格：

```text
abbreviation
expansion
concept_name
concept_code
domain_id
score
```

`Debug States` 需要后端返回 `mapping_states`。

如果第一版后端暂时没有 debug 接口，就先显示：

```text
Debug details require /expand/debug
```

### 6.3 当前可用接口

已有：

```text
POST /expand/simple
```

请求：

```json
{
  "text": "The patient denies SOB."
}
```

响应可用字段：

```text
success
expansion_success
standardization_success
success_breakdown
expanded_text
mappings
standardized_entities
```

### 6.4 建议新增接口

为了前端更好展示链路，建议新增：

```text
POST /expand/debug
```

返回完整：

```text
attempts
mapping_states
abbreviation_candidates
mapping_standardizations
verification
failure
coverage
```

原因：

```text
/expand/simple 适合用户看结果。
/expand/debug 适合项目展示、错误分析和调试。
```

## 7. 页面二：Benchmark 仪表盘

### 7.1 页面目标

展示系统当前评估表现。

用户应该能看到：

```text
总案例数
benchmark accuracy
business success
expansion success
standardization success
各 category 表现
```

### 7.2 页面结构

```text
顶部指标卡
分类表现图表
失败标签统计
最近 benchmark 文件信息
```

顶部指标卡：

```text
Total Cases
Benchmark Accuracy
Business Success
Expansion Success
Standardization Success
```

图表：

```text
category_stats 柱状图
每个 category 的 correct / total
business / expansion / standardization 对比
```

### 7.3 当前可用接口

已有：

```text
GET /benchmark/summary
```

但它目前返回偏基础。

### 7.4 建议扩展接口

建议新增：

```text
GET /benchmark/results
```

返回：

```text
benchmark_results.json 全量或分页结果
```

建议新增：

```text
POST /benchmark/run
```

但第一版可以不做，因为 benchmark 会调用 LLM 和 Milvus，耗时较长，还涉及 API key 成本。

第一版前端可以先做：

```text
读取已有 benchmark_results.json
展示历史结果
不从前端触发 benchmark
```

## 8. 页面三：错误分析页面

### 8.1 页面目标

把错误分析从“看 JSON/MD 文件”变成可浏览的结构化页面。

用户应该能看到：

```text
总体失败数量
失败标签分布
每个失败 case 的原因
扩写失败与标准化失败分开看
LLM triage 的中文解释
```

### 8.2 页面结构

```text
错误总览
失败标签筛选
失败 case 列表
case 详情抽屉 / 右侧详情面板
```

失败标签：

```text
benchmark_mismatch
expansion_blocked
standardization_failure
```

注意：

```text
这些标签可重叠，不能简单相加。
```

case 详情应该展示：

```text
id
category
text
expected_mappings
predicted_mappings
mapping_delta
mapping_states
failure_type
failure_subtype
fallback_called
fallback_candidate_count
fallback_reason
retrieved_top
```

### 8.3 当前可用接口

已有：

```text
GET /error-analysis/summary
```

但只适合总览。

### 8.4 建议新增接口

建议新增：

```text
GET /error-analysis/report
```

返回：

```text
backend/evaluation/error_analysis_report.json
```

建议新增：

```text
GET /error-analysis/triage
```

返回：

```text
backend/logs/triage/error_triage_report.md
```

或者直接返回 markdown 文本：

```json
{
  "markdown": "...",
  "updated_at": "..."
}
```

## 9. 页面四：fallback 候选沉淀页面

### 9.1 页面目标

这是项目的“自我增强闭环”展示页。

用户应该能：

```text
看到本轮 fallback 成功且 CODED 的候选
检查 abbreviation / expansion / domain / chosen_concept
勾选确认
点击加入 primary 候选库
```

### 9.2 页面结构

```text
候选汇总卡
候选表格
候选详情面板
批量操作栏
写入结果提示
```

汇总卡：

```text
total_items
new_item_count
already_exists_count
selected_count
```

表格字段：

```text
checkbox
abbreviation
expansion
domain
support_count
case_ids
chosen_concept.concept_name
already_exists
```

详情面板：

```text
examples
chosen_concepts
candidate_to_append
source_result_file
```

操作：

```text
全选新候选
取消选择
只看未存在
只看某个 domain
加入 primary
```

### 9.3 当前文件

当前后端已经生成：

```text
backend/evaluation/fallback_candidate_promotions.json
backend/evaluation/fallback_candidate_promotions.md
```

### 9.4 建议新增接口

建议新增：

```text
GET /candidate-promotions
```

读取：

```text
fallback_candidate_promotions.json
```

建议新增：

```text
POST /candidate-promotions/apply
```

请求：

```json
{
  "approved_items": [
    {
      "abbreviation": "ABG",
      "expansion": "arterial blood gas",
      "domain": "Measurement"
    }
  ],
  "batch_note": "Added after frontend review at 2026-07-10 12:00:00"
}
```

响应：

```json
{
  "appended": [],
  "skipped": []
}
```

注意：

```text
前端不要直接编辑 abbr_candidates.py。
必须调用后端 apply 逻辑，由后端做 append + 去重 + backup。
```


内容：

```text

```



## 11. 前端数据类型建议

建议定义核心类型。

### 11.1 ExpandSimpleResponse

```ts
type ExpandSimpleResponse = {
  success: boolean;
  expansion_success: boolean;
  standardization_success: boolean;
  success_breakdown?: SuccessBreakdown;
  expanded_text: string;
  mappings: Mapping[];
  standardized_entities: StandardizedEntity[];
};
```

### 11.2 Mapping

```ts
type Mapping = {
  abbreviation: string;
  expansion: string;
  label?: string | null;
  source?: "primary" | "fallback" | "none" | string;
  status?: string;
};
```

### 11.3 StandardizedEntity

```ts
type StandardizedEntity = {
  abbreviation: string;
  expansion: string;
  concept_id: string;
  concept_name: string;
  concept_code: string;
  domain_id: string;
  score?: number;
};
```

### 11.4 CandidatePromotion

```ts
type CandidatePromotion = {
  abbreviation: string;
  expansion: string;
  domain: string;
  already_exists: boolean;
  support_count: number;
  case_ids: string[];
  examples: Array<{
    id: string;
    category: string;
    text: string;
    final_expanded_text: string;
  }>;
  chosen_concepts: Array<{
    concept_id: string;
    concept_name: string;
    domain_id: string;
    concept_code: string;
  }>;
  candidate_to_append: {
    expansion: string;
    domain: string;
  };
};
```

## 13. 第一版开发顺序

建议按这个顺序做：

```text
1. 创建 frontend 项目骨架。
2. 做 AppShell + Sidebar。
3. 做 AnalyzePage，先打通 /expand/simple。
4. 做 BenchmarkPage，打通 /benchmark/summary。
5. 后端补 /candidate-promotions 读取接口。
6. 做 PromotionsPage，只展示不写入。
7. 后端补 /candidate-promotions/apply。
8. PromotionsPage 增加勾选写入。
9. 后端补 /error-analysis/report 和 /error-analysis/triage。
10. 做 ErrorsPage。
11. 最后做 PipelinePage。
```

这样顺序比较稳：

```text
先把最核心的用户输入与结果展示跑通，
再做评估和运维页面，
最后做解释型页面。
```

## 15. 前端展示重点

前端要避免只展示一个“成功/失败”。

应该把系统的核心技术设计展示出来：

```text
候选来源：primary / fallback
扩写是否成功：expansion_success
标准化是否成功：standardization_success
标准概念是否绑定：CODED / WITHHELD
失败为什么发生：failure_type / failure_subtype / evidence
系统如何增强：fallback 成功候选沉淀
```

这样前端才不是“壳”，而是把后端技术亮点展示出来。

## 16. 面试展示路径

面试时可以按这个顺序演示：

```text
1. 在 Analyze 页面输入一句含缩写的临床文本。
2. 展示 expanded_text。
3. 展示 mappings，说明 primary/fallback 来源。
4. 展示 standardized_entities，说明 SNOMED/RxNorm 标准化。
5. 打开 Benchmark 页面，展示整体指标。
6. 打开 Errors 页面，展示失败不是黑盒，有 failure_type/evidence。
7. 打开 Promotions 页面，展示 fallback 成功候选如何被人工确认后沉淀进 primary。
```

这条展示路径能讲清：

```text
功能
技术
评估
错误分析
系统增强
```

## 17. 第一版不要做的事

第一版暂时不要做：

```text
用户登录
权限系统
多租户
数据库后台管理
在线编辑所有候选库
自动运行 benchmark
强制 LangGraph 执行主流程
过度复杂的动态图编辑器
```

这些会分散重点。

第一版要做的是：

```text
把当前后端能力清楚、稳定、可解释地展示出来。
```

## 18. 总结

前端第一版的核心不是“做一个漂亮网页”，而是做一个：

```text
医学缩写标准化系统的可解释工作台
```

它应该让人一眼看懂：

```text
输入是什么
系统扩写了什么
标准化到了哪里
为什么成功或失败
benchmark 表现如何
系统如何通过 fallback 候选沉淀变得更稳定
```

LangGraph 暂时不作为第一版核心依赖，保留为后续流程可视化增强即可。
