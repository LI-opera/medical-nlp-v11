# V11 项目总览与主数据流(先看这一篇)· V11

> 文件范围:`backend/api/main.py`、`backend/services/*`、`backend/data/*`、`backend/tools/*`、`backend/evaluation/*`、`backend/graph/*`
> 衔接:这是 V11 模块文档的入口篇。它不深挖某一个文件,而是先把整个项目拆成几条线:请求入口、候选召回、确定性扩写、向量检索、标准概念校验、反思重检索、评估分析、实验图包装。
> **V11 最大变化(相对 V9 必看)**:项目从"LLM 扩写一句话"升级成了**受控的 retrieval-first 医学文本标准化流水线**。LLM 不再自由改写整句,而是在几个明确节点里做候选覆盖判断、标准概念选择、反思检索词建议;真正替换文本由确定性代码完成。

## 核心速记

> 1. **一句定位**:这是一个医学缩写扩写 + 医学概念标准化原型,目标是把临床短句里的 `SOB/CP/HTN/ASA` 这类缩写,扩成完整医学术语,并尽量对齐 SNOMED / RxNorm 标准概念。
> 2. **主链路入口**:`POST /expand/simple` → `ABBRService.expand_verify_with_retry()`。前者是 API 门面,后者才是真正的生产主编排。
> 3. **V11 的主思想**:**候选先行,受控决策**。先从词典/LLM 兜底召回候选,再判断候选是否覆盖上下文,再确定性替换,再检索标准概念,再让 verifier 在候选里选或弃码。
> 次要(trivia):`MedicalStandardizer.standardize()` 是早期"整句 NER + 检索"编排,V11 主链路没有把它作为最终标准化主干,但复用了它内部的 `NERService`。

## 这个项目到底在解决什么

大白话:医生笔记里经常有缩写,比如:

```text
The patient denies SOB but reports CP.
```

人能读懂它可能是:

```text
The patient denies shortness of breath but reports chest pain.
```

但机器要稳定做到这件事,不能只靠 LLM 猜。因为:

- `CP` 可能是 chest pain,也可能是 cerebral palsy / chronic pancreatitis;
- `ASA` 是药品成分 aspirin,应该走 RxNorm,不是硬塞进 SNOMED 疾病库;
- 有些大写 token 看起来像缩写,但上下文不足时宁可不扩;
- 扩写完还要能落到标准医学概念,否则只是"看起来像医学术语"。

所以 V11 做成了一条比较克制的流水线:

```text
原始文本
  ↓
找出像缩写的 token
  ↓
候选召回(primary 词典 / fallback LLM)
  ↓
coverage 判断:候选里有没有上下文支持的合理扩写
  ↓
确定性文本替换:只替换被选中的缩写
  ↓
按 domain 路由到 SNOMED / RxNorm 检索标准概念
  ↓
verifier 在候选概念里选忠实概念,或明确弃码
  ↓
必要时反思:提出同义检索词,再检索、再验证
  ↓
返回 expanded_text + mappings + standardized_entities
```

这也是后面所有模块文档的主线。

## V11 一共几个框架

我建议把 V11 分成 **8 个框架**。这里的"框架"不是软件框架,而是你理解项目时的 8 块系统边界。

### 1. API 入口框架

相关文件:

- `backend/api/main.py`
- `backend/api/schemas.py`

核心入口:

- `get_service()`
- `expand_abbreviation_simple()`
- `POST /expand/simple`
- `GET /benchmark/summary`
- `GET /error-analysis/summary`

作用:负责接 HTTP 请求、懒加载 `ABBRService`、把内部复杂结果裁剪成 API 输出。

重点理解:

- `/health` 只证明 API 活着,不检查 Milvus / LLM;
- `/expand/simple` 是当前主接口;
- 被注释掉的 `/expand` 是调试版/旧完整输出入口,不是当前启用接口。

### 2. 配置与模型工厂框架

相关文件:

- `backend/utils/embedding_config.py`
- `backend/utils/embedding_factory.py`
- `backend/utils/llm_config.py`
- `backend/utils/llm_factory.py`

作用:集中创建 Embedding 和 LLM。

当前默认:

```text
Embedding: BAAI/bge-m3
LLM: DeepSeek deepseek-chat
可选 LLM 配置: Qwen,但不是所有调用点都统一走 factory
```

重点理解:

- Embedding 走 HuggingFace,自动选择 cuda / cpu;
- LLM factory 支持 DeepSeek/Qwen;
- 有些服务仍直接 `ChatDeepSeek(...)`,后续可统一收口到 factory。

### 3. 离线知识库构建框架

相关文件:

- `backend/tools/build_concept_csv.py`
- `backend/tools/build_rxnorm_csv.py`
- `backend/tools/rebuild_milvus.py`
- `backend/tools/rebuild_rxnorm_milvus.py`
- `backend/data/snomed_clinical.csv`
- `backend/data/rxnorm_clinical.csv`

作用:把医学概念 CSV 变成 Milvus 向量库。

两套 collection:

```text
SNOMED: concepts_only_name
RxNorm: rxnorm_concepts
```

重点理解:

- 这是离线流程,不是用户请求时跑;
- `rebuild_milvus.py` 建 SNOMED;
- `rebuild_rxnorm_milvus.py` 建 RxNorm;
- 两者结构基本一致:读 CSV → bge-m3 批量向量化 → 插入 Milvus → load collection。

### 4. 向量检索底座框架

相关文件:

- `backend/services/std_service.py`
- `backend/services/medical_retriever.py`

作用:把 query 变向量,去 Milvus 搜标准概念,再做轻量规则重排。

两层职责:

```text
StdService
  只负责 embedding + Milvus search + source 选 collection

MedicalRetriever
  负责 exact/prefix/substring/domain_boost/length penalty 重排
  再包装成 documents
```

V11 关键变化:

- `StdService.collections` 同时管理 SNOMED 和 RxNorm;
- `search_similar_terms(query, source="snomed")` 支持多源;
- `MedicalRetriever.retrieve(..., source=...)` 把 source 继续透传到底层。

### 5. 医学实体识别与整句标准化框架

相关文件:

- `backend/services/ner_service.py`
- `backend/services/medical_standardizer.py`

作用:识别文本里的医学实体,再检索标准概念候选。

当前地位:

- `NERService.extract_entities()` 仍然有用;
- `NERService.is_medical()` 被 V11 fallback 候选用来推断 domain;
- `MedicalStandardizer.standardize()` 更像早期整句标准化路径,不是 V11 主输出里的核心标准化主干。

一句话:这块是"医学实体识别能力",但 V11 的缩写标准化主干已经转向 `ABBRService` 的 per-abbreviation record 状态机。

### 6. 缩写候选召回与覆盖判断框架

相关文件:

- `backend/data/abbr_candidates.py`
- `backend/services/abbr_candidate_retriever.py`
- `backend/services/abbr_candidate_fallback_retriever.py`
- `backend/services/abbr_candidate_coverage_evaluator.py`

作用:先回答"这个缩写可能是什么意思",再判断"这些候选里有没有一个适合当前上下文"。

数据流:

```text
token
  ↓
_should_consider_abbreviation()
  ↓
primary: ABBR_CANDIDATES 查表
  ↓ 若查不到
fallback: LLM 只生成候选列表,不改写句子
  ↓
coverage evaluator 选择 best_expansion 或弃权
```

重点理解:

- primary 是人工候选库,可信度更高;
- fallback 是非词典缩写兜底,因此 V11 对 fallback 更保守;
- coverage 是扩不扩的闸门,不是最终 SNOMED 概念选择器。

### 7. 主编排、校验与反思框架

相关文件:

- `backend/services/abbr_service.py`
- `backend/services/abbr_verifier.py`
- `backend/services/error_collector.py`

核心入口:

- `ABBRService.expand_verify_with_retry()`
- `ABBRService._build_expanded_text_deterministic()`
- `ABBRService._route_source()`
- `ABBRService._reflect_refine_standardization()`
- `ABBVerifier.verify_mappings()`
- `ABBVerifier.propose_requeries()`

作用:这是 V11 的主状态机。它把一个缩写从"被发现"一路带到最终状态:

```text
NOT_EXPANDED
PENDING
CODED
WITHHELD
ABSTAIN
```

重点理解:

- 扩写文本不是 LLM 生成,而是 `_build_expanded_text_deterministic()` 替换;
- `coverage` 决定缩写是否扩写;
- `verify_mappings()` 只负责在检索候选里选忠实标准概念,或弃码;
- `_reflect_refine_standardization()` 只在标准化不够好时提出新检索词,不会凭空创造概念。

### 8. 评估、错误分析与实验图框架

相关文件:

- `backend/evaluation/run_benchmark.py`
- `backend/evaluation/run_benchmark_parallel.py`
- `backend/evaluation/concept_match.py`
- `backend/evaluation/error_analysis_report.py`
- `backend/evaluation/error_triage.py`
- `backend/graph/standardization_graph.py`
- `backend/graph/render_graph.py`

作用:验证系统效果、统计失败类型、把主链路包装成可视化图。

当前状态:

- `benchmark_results.json` 记录当前 benchmark 结果;
- `error_analysis_report.py` 读 benchmark 结果做规则归因;
- `error_triage.py` 读 unresolved logs,用 LLM 提假设,再用真实数据自检;
- `StandardizationGraph` 是 LangGraph 实验/可视化包装,不在 FastAPI 热路径。

## 主请求数据流:从 API 到返回结果

下面是当前最重要的一条链,后面文档都围绕它展开:

```text
POST /expand/simple
  文件:backend/api/main.py
  函数:expand_abbreviation_simple()

    ↓ get_service() 懒加载

ABBRService()
  文件:backend/services/abbr_service.py
  初始化:
    - MedicalStandardizer()
    - NERService
    - MedicalRetriever()
    - ABBVerifier()
    - ABBRCandidateRetriever()
    - ABBRCandidateFallbackRetriever()
    - ABBRCandidateCoverageEvaluator()

    ↓

expand_verify_with_retry(text)

    ↓ 1. 找缩写 + 候选召回

_get_abbreviation_candidates(text)
  - token gate: _should_consider_abbreviation()
  - primary: ABBRCandidateRetriever.retrieve()
  - fallback: ABBRCandidateFallbackRetriever.retrieve()
  - coverage: ABBRCandidateCoverageEvaluator.evaluate()

    ↓ 2. 建统一 record

records = [
  {
    abbreviation,
    candidates,
    expansion,
    domain,
    std_cache,
    std_concept,
    status,
    failure
  }
]

    ↓ 3. 确定性替换

_build_expanded_text_deterministic(text, visible_records)

    ↓ 4. 对每个 PENDING 扩写检索标准概念

MedicalRetriever.retrieve(
  query=expansion,
  domain_boost=domain,
  source=_route_source(domain)
)

    ↓ 5. 底层向量检索

StdService.search_similar_terms(query, source)
  - source="rxnorm" if domain=="Drug"
  - else source="snomed"

    ↓ 6. verifier 选概念或弃码

ABBVerifier.verify_mappings(
  original_text,
  expanded_text,
  mapping_standardizations
)

    ↓ 7. 反思重检索

_reflect_refine_standardization()
  - propose_requeries()
  - retriever.retrieve(new_query)
  - verify_mappings(new_candidates)

    ↓ 8. 汇总输出

final_result:
  expanded_text
  mappings
  mapping_standardizations
  mapping_states

    ↓

/expand/simple 裁剪成:
  success
  expanded_text
  mappings
  standardized_entities
```

## 为什么 V11 这么设计

### 1. 防止 LLM 自由扩写导致幻觉

V9/V10 很容易讲成"LLM 看到一句话,直接扩写并标准化"。但 V11 更克制:

```text
LLM 不直接改原句
LLM 只在规定任务中做判断:
  - fallback 只给候选,不改句
  - coverage 只判断候选是否覆盖,不新增候选
  - verifier 只在检索候选里选,不发明概念
  - reflection 只提检索词,不创造标准概念
```

这就是项目最值得讲的工程思想。

### 2. 把不确定性拆成几个小问题

原问题:

```text
请把临床句子里的医学缩写正确扩写并标准化。
```

拆完后:

```text
这个 token 值得当缩写吗?
这个缩写有哪些候选?
候选里有没有上下文支持的扩写?
选中的扩写替换到原句哪里?
这个扩写应该去 SNOMED 还是 RxNorm?
检索候选里哪个概念最忠实?
如果没有精确概念,能否换一个同义检索词再找?
```

每个小问题都有明确输入输出,也更容易调试和评估。

### 3. 状态机让失败可解释

V11 的 record 会留下:

```text
status: NOT_EXPANDED / PENDING / CODED / WITHHELD / ABSTAIN
failure: type / stage / reason / evidence
```

这比只返回一个 `success=false` 好很多。因为失败能定位到:

- coverage 没过;
- fallback 候选不可信;
- 检索不到合适标准概念;
- verifier 认为候选都不忠实;
- 反思也没救回来。

## 死代码 / 盲肠提醒

- `ABBRService.__init__` 里的 `self.abbr_dict` 是早期 V0 硬编码词典,当前 V11 主链路没有读取它。真正候选来自 `data/abbr_candidates.py`。
- `ABBVerifier.verify()` 是早期"整句扩写 + 整句标准化"校验接口,当前主链路主要用 `verify_mappings()`。
- `MedicalStandardizer.standardize()` 仍可运行,但当前 `/expand/simple` 主链路没有把它作为最终标准化输出的主干。
- `graph/standardization_graph.py` 是 LangGraph 可视化/实验封装,不是 FastAPI 热路径。
- `api/main.py` 中注释掉的 `/expand` 是旧完整调试版接口,当前启用的是 `/expand/simple`。

这些不是说"没用",而是学习时要分清:哪些是主链路,哪些是历史遗留/实验支线。

## 目前效果与诚实边界

从当前 `backend/evaluation/benchmark_results.json` 看:

```text
Total Cases: 74
Correct: 71
Accuracy: 0.9594594594594594
```

这个数字只能说明:在当前 benchmark 集合上,这条受控流水线效果不错。不能夸成"真实临床泛化已验证"。

已知边界:

- 候选词典还是项目级轻量库,不是完整医学缩写数据库;
- 低上下文缩写仍可能过扩或弃权;
- LLM 输出 JSON 仍有解析风险,虽然代码做了 fallback;
- Milvus / 模型 / API key 都是运行前提;
- 药品多源路由当前主要靠 `domain=="Drug"` 这个标签;
- `domain`、`source`、Milvus collection 名之间是约定式映射,后续可以常量化/枚举化。

## 面试怎么说

**合格版(30 秒)**:
> 这个项目是一个医学缩写扩写和标准概念对齐的后端原型。V11 的核心不是让 LLM 直接改写句子,而是做成 retrieval-first 流水线:先从候选词典或 LLM 兜底召回扩写候选,再做 coverage 判断,通过确定性替换生成扩写文本,然后按 domain 路由到 SNOMED 或 RxNorm 做向量检索,最后由 verifier 在候选概念里选择忠实标准概念或弃码。

**优秀版(1 分钟)**:
> 我把这个项目从"LLM 自由扩写"收敛成了一个受控的状态机。输入一句临床文本后,系统先用 gate 找出像缩写的 token,已知缩写走本地候选词典,未知大写缩写才走 LLM fallback,但 fallback 也只允许生成候选,不能改写原句。然后 coverage evaluator 判断候选集中是否有上下文支持的扩写,被选中的缩写用确定性 token 边界替换,避免误伤子串和否定结构。标准化阶段会根据候选 domain 做多源路由,Drug 去 RxNorm,其它去 SNOMED,检索 top candidates 后让 verifier 只在候选里选忠实概念,选不到就 WITHHELD。对于非精确或弃码结果,还有一层反思重检索,只提出同义检索词,再检索再验证。这样 LLM 的自由度被限制在几个可审计节点,每个缩写也有状态和 failure reason,方便 benchmark 和 error analysis。

## 易错点 / 面试问答

**Q:这个项目是不是一个普通 RAG?**  
A:不是普通问答 RAG。它用 embedding + Milvus 做候选检索,但目标不是生成答案,而是医学缩写扩写和标准概念 grounding。更准确说是 retrieval-assisted normalization pipeline。

**Q:LLM 在哪里用?**  
A:主要四处:未知缩写 fallback 候选召回、候选 coverage 判断、标准概念 faithful selection、反思阶段提出新检索词。它不直接生成最终扩写句子。

**Q:为什么要有 SNOMED 和 RxNorm 两个库?**  
A:SNOMED 更适合疾病、症状、检查、部位等临床概念;RxNorm 更适合药品成分。V11 通过 `domain=="Drug"` 路由到 RxNorm,其它默认走 SNOMED。

**Q:coverage 和 verifier 有什么区别?**  
A:coverage 判断"缩写候选里有没有适合上下文的扩写",决定扩不扩、选哪个扩写。verifier 判断"检索回来的标准概念是否忠实表达这个扩写",决定编码还是弃码。一个管扩写,一个管标准概念。

**Q:为什么最终文本不用 LLM 生成?**  
A:为了可控。确定性替换能保留原句结构、否定词和其它上下文,也避免 LLM 增删诊断或治疗信息。

**Q:LangGraph 是生产链路吗?**  
A:不是。`graph/standardization_graph.py` 是把单个 mapping 的路由、检索、verify、反思环显式画出来的实验/可视化包装。生产主链路仍是 `ABBRService.expand_verify_with_retry()`。

## 一句话总结

> V11 的 medical-nlp 项目是一条受控的医学缩写标准化流水线:API 接收文本后,`ABBRService` 先做缩写候选召回和 coverage 闸门,再用确定性替换生成扩写文本,接着按 domain 路由到 SNOMED/RxNorm 向量库检索标准概念,由 verifier 选择忠实概念或弃码,必要时通过反思重检索补救。它的核心价值不是"用了 LLM",而是把 LLM 限制在可审计的小决策节点里,用候选、检索、状态机和错误日志把医学文本标准化做得更稳、更可解释。
