# V11 项目改进方案与下一步路线

> 文件范围:`backend/services/*`、`backend/evaluation/*`、`backend/tools/*`、`Dockerfile`、`docker-compose.yml`
> 衔接:第 00-21 篇已经把 V11 当前框架、主链路、评估、错误分析、LangGraph 和 Docker 都拆完了。本篇不再介绍某个节点,而是收口:当前 V11 还差什么、哪些是生产化短板、哪些是研究/面试时可以主动讲的下一步。
> **V11 必看定位**:这不是“推翻重做”。V11 的主线已经成立:候选召回 → coverage 选唯一 → 确定性替换 → 多源检索 → verifier 判忠实/弃码 → 反思重检索 → benchmark/error analysis。改进重点是把它从可运行原型推进到更可靠、更可评估、更可部署。

## 核心速记

> 1. **第一优先级**:把“无缩写正常句子”和“缩写扩写失败”分开。现在无缩写会早停 `success=False`,语义上不够准确。
> 2. **第二优先级**:补强评估体系。主 benchmark 只判扩写,concept benchmark 才判标准化;还需要 mapping 级 P/R/F1、更多真实数据、更多负例。
> 3. **第三优先级**:提升检索数据质量。SNOMED/RxNorm 已全量化,但仍主要查 `concept_name`;同义词/FSN/别名、多向量、多字段还没充分利用。
> 4. **第四优先级**:生产化容器和运行稳定性。当前只容器化 API,Milvus/模型缓存/healthcheck/密钥安全还没完整治理。
> 次要(trivia):很多“改进点”已经不是 V9 那些设想了,因为 V11 已经把确定性替换、mapping 状态机、多源路由、错误日志、LangGraph 可视化都落成了。

## 先说结论:V11 当前是什么水平

V11 已经不是早期“LLM 造句扩写”的玩具链路。

它现在能讲清楚三条能力:

```text
1. 扩写层:
   74 例主 benchmark,当前 71/74 = 0.9594594594594594
   失败集中在 low_context_over_expansion

2. 标准化层:
   concept benchmark 给“选忠实概念 / 弃码”单独建尺子
   主 benchmark 不再被误当成标准化层全部证据

3. 多源检索层:
   SNOMED + RxNorm 双 collection
   Drug → RxNorm,其它 → SNOMED
   药品 A/B 能解释“补正确知识源”而不是放松护栏
```

所以后续优化不该再回到“大改架构”的思路,而是沿着这条线补短板:

```text
状态语义更准确
评估更细
检索更强
运行更稳
部署更完整
```

## 改进1 · 区分“无缩写”与“扩写失败”

当前主链路里,如果 gate 没找到任何缩写,或者 coverage 没产出 expansion,会进入早停。

问题是:

```text
无缩写的普通医学句子
```

和:

```text
有缩写,但系统没能扩出来
```

不应该都被用户理解为失败。

例如:

```text
The patient has chest pain.
```

这句话没有缩写,系统文本不动是合理的。当前如果返回 `success=False`,调用方容易误以为系统处理失败。

### 建议方案

把早停状态拆细:

```text
NO_ABBREVIATION_DETECTED     # 正常,没有需要扩写的缩写
COVERAGE_FAILED              # 有候选,但 coverage 不支持
NO_CANDIDATES                # gate 放行,但候选召回为空
ALL_ABSTAINED                # 候选耗尽,最终弃权
```

API 侧可以新增:

```json
{
  "success": true,
  "status": "NO_ABBREVIATION_DETECTED",
  "expanded_text": "The patient has chest pain.",
  "mappings": [],
  "standardized_entities": []
}
```

或者保守一点,不改 `success` 含义,新增:

```json
{
  "status": "...",
  "reason": "..."
}
```

### 为什么优先级高

因为这是用户可见语义。

它不影响核心算法,但会影响:

```text
前端展示
调用方判断
benchmark 解释
错误日志归因
```

这是一个小改动,但工程收益很实在。

## 改进2 · 补上“纯实体标准化”路径

当前主链路是:

```text
缩写扩写 → expansion 标准化
```

但如果输入没有缩写,系统不会主动对原句里的医学实体做 SNOMED/RxNorm 标准化。

例如:

```text
The patient has chest pain and diabetes mellitus.
```

这里没有缩写,但 `chest pain`、`diabetes mellitus` 本身也可以标准化。

V11 当前有 `NERService` 和 `MedicalStandardizer`,但 API 主输出主要来自 `ABBRService` 的 mapping 级标准化。

### 建议方案

把能力拆成两条路径:

```text
路径 A: abbreviation normalization
  缩写 → expansion → 标准概念

路径 B: entity standardization
  原句实体 → 标准概念
```

API 可以选择:

```json
{
  "expanded_text": "...",
  "abbreviation_mappings": [...],
  "standardized_abbreviations": [...],
  "standardized_entities": [...]
}
```

其中:

```text
standardized_abbreviations = 缩写扩写后的概念
standardized_entities      = 原句 NER 抽到的非缩写实体概念
```

### 注意边界

这不是把旧 `MedicalStandardizer.standardize()` 原样塞回主链路。

更稳的是:

```text
把“缩写扩写标准化”和“普通实体标准化”作为两个并列产品能力
分别评估、分别输出、不要混成一个 success
```

否则容易回到 V9 的整句标准化盲肠。

## 改进3 · 主 benchmark 加 mapping 级指标

当前主 benchmark 的 `correct` 是 case 级:

```text
一个 case 里多扩一个或少扩一个 → 整题错
```

这很严格,适合做回归闸门,但看不出部分能力。

例如一句话有三个缩写:

```text
CP, SOB, DM
```

系统扩对两个、错一个,当前整 case 算错。

### 建议方案

保留 case-level exact accuracy,同时新增:

```text
mapping_precision
mapping_recall
mapping_f1
over_expansion_count
under_expansion_count
wrong_disambiguation_count
```

这样能回答:

```text
错误主要是多扩?
还是漏扩?
还是扩写义项选错?
```

第 18 篇 error analysis 已经有 taxonomy 雏形,可以把它前移成 benchmark 的正式指标。

### 为什么重要

当前 71/74 很好,但面试或研究里别人会追问:

```text
那错的 3 条到底错多少?
有没有部分分?
```

mapping 级指标能让评估更像 NLP 系统,而不只是 demo accuracy。

## 改进4 · concept benchmark 扩成标准化层主尺子

第 17 篇已经讲过:

```text
主 benchmark 主要判 abbreviation expansion。
```

第 18/当前源码里也明确:

```text
concept benchmark 才测给定正确扩写后,标准化选概念是否忠实/规范。
```

当前 `concept_benchmark_cases.py` 的口径很好:

```text
PASS      = 选中 {prefer}+accept 中任意忠实概念
canonical = 选中 prefer 最规范概念
```

### 建议方案

把 concept benchmark 从辅助脚本提升成常规验收:

```text
每次改 retriever / verifier / reflection / source routing:
  1. 跑主 benchmark
  2. 跑 concept benchmark
  3. 跑 drug source A/B
```

并把结果固定记录:

```text
Expansion accuracy
Concept PASS
Concept canonical
Drug source A/B
```

### 继续补哪些 case

1. **真负例**

当前 concept benchmark 里真负例偏少。需要补:

```text
库里没有忠实概念 → 应该 WITHHELD
```

2. **俗称/同义词**

例如:

```text
heart attack → Myocardial infarction
high blood pressure → Hypertensive disorder
fluid in the lungs → Pulmonary edema
```

3. **RxNorm 边界**

当前药品主要是成分级:

```text
aspirin / methotrexate / acetaminophen ...
```

后续可补:

```text
商品名
复方药
剂型/剂量
拼写变体
```

但要明确边界:这些不是已经证明的能力。

## 改进5 · 检索库加入同义词 / FSN / 多字段

V11 已经把 SNOMED/RxNorm 库规模做起来了,但检索主要围绕概念名。

真实医学概念有很多表达:

```text
shortness of breath
dyspnea
difficulty breathing
```

如果只 embedding `concept_name`,召回上限会受限。

### 建议方案

离线建库时扩展记录:

```text
concept_id
concept_name
FSN
synonym
vocabulary_id
domain_id
concept_class_id
```

可以有两种建法:

### 方案 A:多条记录

同一个 `concept_id` 的每个 synonym 单独入库:

```text
record 1: concept_id=123, text="Dyspnea"
record 2: concept_id=123, text="Shortness of breath"
record 3: concept_id=123, text="Difficulty breathing"
```

检索后按 `concept_id` 合并。

### 方案 B:多字段拼接

把多个名字拼成一个检索文本:

```text
concept_name + synonyms + FSN
```

简单,但可解释性和精确匹配会差一些。

### 推荐

优先方案 A。

因为它能保留:

```text
哪个 synonym 被命中
同一个 concept_id 下哪些文本召回
```

这对 verifier 和 error analysis 更有用。

## 改进6 · rerank 权重从经验常数走向可评估

`MedicalRetriever` 目前的重排规则是启发式:

```text
exact match bonus
prefix bonus
substring bonus
domain boost
length penalty
```

这很适合 MVP,透明、可解释。

但缺点是:

```text
权重靠经验
不同 source 命名风格不同
SNOMED 和 RxNorm 可能不该用同一套长度惩罚
```

### 建议方案

先不要直接上复杂 reranker,而是做小步可评估:

1. 固定一组 concept benchmark / source A/B;
2. 给 rerank 权重做配置化;
3. 跑网格或小规模 ablation;
4. 只保留净收益明确的权重变化。

可配置项:

```text
exact_bonus
prefix_bonus
substring_bonus
domain_bonus
long_name_penalty
score_threshold
top_k
verify_window
```

未来再考虑:

```text
cross-encoder reranker
医学同义词词表
source-specific rerank
```

## 改进7 · LLM 输出结构化和数量校验

项目里多个节点依赖 LLM 返回 JSON:

```text
fallback candidate retriever
coverage evaluator
verifier
reflection/requery proposer
error triage
```

当前已有 JSON 解析兜底,但还可以更硬。

### 建议方案

1. **候选返回用 index**

coverage 不要只返回 expansion 字符串,最好返回候选 index:

```json
{
  "best_index": 1,
  "plausible_indices": [1, 2]
}
```

这样不依赖 LLM 原样复述:

```text
"mitral stenosis" vs "Mitral Stenosis"
```

2. **校验 mapping_validations 数量**

`verify_mappings` 返回数量应等于输入 mapping 数。

如果不一致:

```text
直接失败 / 重试 / 标记 parse_error
```

不能让不完整返回悄悄通过。

3. **统一 Pydantic schema**

为 LLM 输出定义结构:

```text
FallbackCandidates
CoverageDecision
VerificationResult
RequeryProposal
```

减少各处手写解析。

## 改进8 · 错误分析 taxonomy 合并

第 18 篇提到当前错误分析有两层:

```text
error_type
taxonomy
```

它们有重叠,早期为了快速看不同粒度可以理解,但长期维护会口径分裂。

### 建议方案

合并成一个层级 taxonomy:

```json
{
  "major_type": "Over Expansion",
  "sub_type": "Extra Abbreviation Expansion",
  "signals": {
    "extra_abbreviations": ["QRS"],
    "missing_abbreviations": [],
    "wrong_expansions": []
  },
  "stage": "expansion",
  "confidence": "rule"
}
```

并支持一个 case 多标签:

```text
既多扩又漏扩
```

不要只因为 `extra_abbrs` 优先判断,就掩盖 `missing_abbrs`。

## 改进9 · 运行时稳定性和异常处理

当前很多错误会直接抛出:

```text
DeepSeek key 缺失
LLM 超时
Milvus 连不上
collection 不存在
模型下载失败
JSON parse error
```

这对原型能接受,生产化不够。

### 建议方案

1. **统一异常模型**

API 返回:

```json
{
  "success": false,
  "status": "DEPENDENCY_UNAVAILABLE",
  "error": {
    "type": "MILVUS_CONNECTION_FAILED",
    "message": "...",
    "recoverable": true
  }
}
```

2. **分层 healthcheck**

```text
/health  API 存活
/ready   Milvus + collection + key + model ready
```

3. **LLM 调用重试和超时**

不同节点要有不同策略:

```text
fallback 失败 → 可以弃权
coverage 失败 → 可以保守不扩
verifier 失败 → 不给码或重试
```

4. **错误写入 error_collector**

运行时依赖失败也应进入 JSONL,否则线上问题无法沉淀。

## 改进10 · 并发与资源生命周期

FastAPI 当前是全局懒加载:

```python
service = None
```

第一次请求创建 `ABBRService`,后续复用。

优点:

```text
避免每次请求重复加载模型
```

风险:

```text
并发下多个请求同时第一次进来可能重复初始化
Milvus/LLM/transformers pipeline 是否线程安全未验证
```

### 建议方案

1. 初始化加锁;
2. 明确 uvicorn worker 数;
3. 压测并发请求;
4. 对 LLM 调用加限流/队列;
5. 对 embedding/NER 模型做单例缓存;
6. benchmark parallel 和 API 并发分开考虑。

尤其要注意:

```text
多 worker = 每个进程一份模型
```

内存会乘倍增长。

## 改进11 · Docker 生产化

第 21 篇已经讲过当前 Docker 边界。

下一步建议:

1. 加 `.dockerignore`;
2. compose 编排 Milvus 或明确外部 Milvus;
3. 显式配置 `MILVUS_RXNORM_COLLECTION`;
4. 模型缓存 volume;
5. Linux `host-gateway`;
6. healthcheck;
7. 非 root 用户;
8. 锁定依赖版本并做镜像大小治理。

优先级最高的是:

```text
.dockerignore 排除 backend/.env
```

因为这是安全问题。

## 改进12 · LangGraph 的真实下一步

当前 LangGraph 是可视化 + parity,不进热路径。

不要为了“更 agentic”强行接进 API。

更有价值的下一步是:

```text
1. 增加更多 parity 样例,覆盖 Drug/RxNorm、WITHHELD、ABSTAIN、反思触发;
2. 把 render_graph.py 的 parity 纳入回归命令;
3. 若未来要上 human-in-the-loop,再考虑让 graph 承担 checkpoint/review;
4. 如果只是普通请求响应,继续让生产主链路保持函数式。
```

一句话:

```text
LangGraph 要继续做“能证明自己没改坏行为”的可视化工具,
除非 checkpoint/人工审核/流式状态真的成为需求。
```

## 路线优先级表

| 优先级 | 改进项 | 原因 |
|---|---|---|
| P0 | `.dockerignore` 排除 `.env` | 密钥安全 |
| P0 | 区分无缩写 vs 扩写失败 | API 语义正确性 |
| P1 | mapping 级 P/R/F1 | 评估更细,定位更准 |
| P1 | concept benchmark 常规化 | 标准化层不能靠主 benchmark 代替 |

| P2 | synonym/FSN 多向量建库 | 提升检索召回上限 |

| P2 | LLM 输出 index/schema 化 | 降低解析和字符串耦合 |
| P3 | Milvus compose 编排 | 更接近一键部署 |
| P3 | LangGraph parity 扩展 | 保持可视化与生产对齐 |

## 面试怎么讲

可以这样说:

> V11 已经把主链路从自由生成收敛成 retrieval-first 状态机:候选召回、coverage 选唯一、确定性替换、多源检索、verifier 忠实性判断和有界反思都已经跑通。下一步我不会推翻架构,而是补生产化短板。第一是 API 语义,把“无缩写正常返回”和“扩写失败”拆开;第二是评估,在 case accuracy 外加 mapping 级 P/R/F1,并把 concept benchmark 常规化;第三是检索数据,把 SNOMED/RxNorm 的 synonym、FSN、多字段加入向量库;第四是部署和稳定性,补 .dockerignore、healthcheck、Milvus 编排、模型缓存和异常结构化。这样项目会从可演示原型,变成更可靠的工程系统。

如果被问“最大的短板是什么”,可以答:

> 当前最大短板不是主链路不通,而是评估和生产化还不够成熟。主 benchmark 只有 74 例,且主要判扩写;标准化层虽然有 concept benchmark,但还需要更多真负例和边界样本。运行上也还依赖宿主机 Milvus 和外部模型缓存,不是完整一键生产部署。

## 一句话总结

V11 后续不该再大拆大改主链路,而应围绕“语义更准确、评估更细、检索更强、运行更稳、部署更完整”推进:先修无缩写状态和 `.dockerignore`,再把主 benchmark 扩成 mapping 级指标、把 concept benchmark 常规化,随后补 synonym/FSN 多向量建库、rerank ablation、LLM schema 校验、异常处理和完整容器编排。
