# 先读这个 —— V11 从 0 到 1 主线学习路线

> 这篇不是新增技术模块,而是给 `00-22` 那套模块文档重新排一条“人能读懂的路”。  
> 前面那些文档更像零件说明书:每篇讲一个模块。但你现在需要的是“这个项目为什么一步步长成这样”。  
> 所以本篇按**需求递进**来讲:先有一个最小需求,发现它不够,再加下一层。每一层只回答三个问题:为什么要它、数据变成什么、该回看哪篇模块文档。

## 先给你一句总定位

这个项目不是“输入一句话,让 LLM 帮我改写”。

它现在更像一个受控流水线:

```text
用户输入临床短句
  ↓
找出可能是缩写的 token
  ↓
给每个缩写找候选 expansion
  ↓
判断当前上下文到底支持哪个 expansion
  ↓
用确定性代码替换缩写,不让 LLM 自由改句子
  ↓
把每个 expansion 去医学标准库里找标准概念
  ↓
按词的类型决定查 SNOMED 还是 RxNorm
  ↓
让 verifier 在候选概念里选忠实概念,选不到就弃码
  ↓
如果标准概念不够好,提出新检索词再检索再判
  ↓
返回 expanded_text + mappings + standardized_entities
```

你可以把它理解成:

```text
缩写扩写系统 + 医学标准编码系统 + 离线评估系统
```

不是单纯 RAG,也不是单纯 NER,也不是单纯 LLM。

## 为什么原来的 00-22 会读着跳

因为它是按“文件/模块”排的:

```text
词典
embedding
LLM
建库
StdService
retriever
NER
...
```

这种顺序适合查资料,但不适合第一次理解项目。

第一次理解应该按“需求演化”排:

```text
我要扩写 SOB
  ↓
怎么知道 SOB 是缩写?
  ↓
怎么知道 SOB 能扩成 shortness of breath?
  ↓
CP 有多个意思怎么办?
  ↓
不能让 LLM 改坏原句怎么办?
  ↓
扩完以后怎么给 SNOMED/RxNorm 编码?
  ↓
标准库怎么查?
  ↓
查错了怎么拦住?
  ↓
系统好不好怎么评估?
```

下面就按这个顺序重新走一遍。

# 第一层:最小需求 —— 把医学缩写扩成全称

## 需求 1:用户给一句话,我想把 SOB 扩成 shortness of breath

输入:

```text
The patient denies SOB.
```

理想输出:

```text
The patient denies shortness of breath.
```

这听起来很简单,最暴力的方法是:

```python
text.replace("SOB", "shortness of breath")
```

但这个方法马上会遇到问题:

1. 你怎么知道哪些 token 是缩写?
2. 你怎么知道缩写对应什么全称?
3. 如果一个缩写有多个意思怎么办?
4. 替换时会不会误伤别的词?
5. LLM 如果重写整句,会不会把 `denies` 改没?

所以项目第一阶段要解决的是:

```text
先别标准化、别 SNOMED、别 RxNorm。
先安全地把缩写扩成全称。
```

对应模块:

```text
12_ABBRService_缩写Gate与确定性替换.md
01_缩写候选词典_abbr_candidates.md
09_ABBRCandidateRetriever_主候选召回_abbr_candidate_retriever.md
10_ABBRCandidateFallbackRetriever_LLM兜底候选召回.md
11_ABBRCandidateCoverageEvaluator_覆盖度闸门.md
```

建议你读的时候不要一口气读 01-11。按下面这个故事读。

## 需求 2:先决定“谁是缩写”

输入一句话:

```text
The patient has CP and DM.
```

系统先做 gate:

```text
The      → 普通词,跳过
patient  → 普通词,跳过
has      → 普通词,跳过
CP       → 可能是缩写,放行
and      → 普通词,跳过
DM       → 可能是缩写,放行
```

这一步不要想复杂。

它就是一个门卫:

```text
哪些 token 值得进入缩写处理流程?
```

输出大概是:

```text
["CP", "DM"]
```

回看:

```text
12_ABBRService_缩写Gate与确定性替换.md
```

这里先只看 gate 和确定性替换部分,不要管后面的状态机。

## 需求 3:给缩写找候选 expansion

现在系统知道要处理:

```text
CP
DM
```

第一选择不是问 LLM,而是查本地词典:

```python
ABBR_CANDIDATES
```

例如:

```text
DM → diabetes mellitus
CP → chest pain / cerebral palsy
```

注意这里的输出不是最终答案,只是候选池。

数据变成:

```json
[
  {
    "abbreviation": "CP",
    "candidates": [
      {"expansion": "chest pain", "domain": "Condition"},
      {"expansion": "cerebral palsy", "domain": "Condition"}
    ]
  },
  {
    "abbreviation": "DM",
    "candidates": [
      {"expansion": "diabetes mellitus", "domain": "Condition"}
    ]
  }
]
```

这里你只需要记住两个点:

```text
1. 词典负责高可信候选召回。
2. domain 字段以后会决定查 SNOMED 还是 RxNorm。
```

回看:

```text
01_缩写候选词典_abbr_candidates.md
09_ABBRCandidateRetriever_主候选召回_abbr_candidate_retriever.md
```

## 需求 4:词典没有怎么办?

比如:

```text
The patient was seen by PCP.
```

如果 `PCP` 不在词典,系统不能直接放弃,于是走 fallback:

```text
LLM 根据上下文生成候选 expansion
```

但这里有一条铁律:

```text
LLM 只生成候选,不允许改写原句。
```

也就是说它只能输出类似:

```json
{
  "abbreviation": "PCP",
  "candidates": [
    {"expansion": "primary care physician"},
    {"expansion": "pneumocystis pneumonia"}
  ]
}
```

它不能直接把句子改成:

```text
The patient was seen by primary care physician.
```

因为最终选哪个候选,还要交给下一层 coverage 判断。

回看:

```text
10_ABBRCandidateFallbackRetriever_LLM兜底候选召回.md
```

## 需求 5:多个候选里到底选哪个?

现在来到最关键的一步。

候选池可能是:

```text
CP → chest pain / cerebral palsy
```

句子是:

```text
The patient denies CP.
```

系统需要判断:

```text
在这句话里,CP 更像 chest pain,不是 cerebral palsy。
```

这就是 coverage evaluator。

它问的问题不是:

```text
CP 医学上能不能表示 chest pain?
```

而是:

```text
当前这句话的上下文,支持哪个 expansion?
```

输出从“候选列表”收敛成“唯一最佳”:

```json
{
  "abbreviation": "CP",
  "best_expansion": "chest pain",
  "coverage_ok": true,
  "confidence": 0.9
}
```

如果上下文太弱,例如:

```text
The patient has QRS.
```

系统可能仍会误扩,这就是当前 benchmark 失败集中处:

```text
low_context_over_expansion
```

回看:

```text
11_ABBRCandidateCoverageEvaluator_覆盖度闸门.md
17_Benchmark评估系统_run_benchmark.md
18_ErrorAnalysis_错误原因分析系统.md
```

但第一次读时只需要记住:

```text
coverage 是“候选到唯一 expansion”的裁判。
```

## 需求 6:选好 expansion 后,怎么改文本?

以前的危险做法是:

```text
让 LLM 重写整句话
```

但这会引入问题:

```text
The patient denies SOB.
```

LLM 可能改坏否定:

```text
The patient has shortness of breath.
```

所以 V11 改成确定性 token 边界替换:

```text
SOB → shortness of breath
CP  → chest pain
```

只替换缩写位置,其它词原样保留。

输入:

```text
The patient denies SOB but reports CP.
```

mapping:

```json
[
  {"abbreviation": "SOB", "expansion": "shortness of breath"},
  {"abbreviation": "CP", "expansion": "chest pain"}
]
```

输出:

```text
The patient denies shortness of breath but reports chest pain.
```

这里是项目非常重要的分水岭:

```text
LLM 负责小决策。
代码负责真正改文本。
```

回看:

```text
12_ABBRService_缩写Gate与确定性替换.md
14_ABBRService主状态机_expand_verify_with_retry.md
```

# 第二层:扩写完还不够 —— 要对齐医学标准概念

## 需求 7:扩成 chest pain 后,还想知道它对应哪个标准概念

扩写层输出:

```text
CP → chest pain
```

但医学标准化还要进一步问:

```text
chest pain 在 SNOMED 里对应哪个 concept_id?
```

这就是标准化层。

目标输出不只是:

```json
{"abbreviation": "CP", "expansion": "chest pain"}
```

而是:

```json
{
  "abbreviation": "CP",
  "expansion": "chest pain",
  "concept_id": "...",
  "concept_name": "Chest pain",
  "concept_code": "..."
}
```

这时项目进入 RAG 部分:

```text
query = "chest pain"
  ↓
embedding
  ↓
Milvus 向量搜索
  ↓
取回候选标准概念
  ↓
verifier 选择忠实概念
```

回看:

```text
02_Embedding配置与工厂_utils.md
04_离线建库工具_tools.md
05_StdService_多源向量底座_std_service.md
06_MedicalRetriever_检索层与规则重排_medical_retriever.md
13_ABBVerifier_标准概念忠实性校验_abbr_verifier.md
```

## 需求 8:标准库从哪里来?

这里才需要读“离线建库工具”。

你可以先这样理解:

```text
离线阶段:
  从 Athena / CSV 里拿 SNOMED/RxNorm 概念
  ↓
  每个 concept_name 算 embedding
  ↓
  插进 Milvus collection

在线阶段:
  用户 query 算 embedding
  ↓
  去 Milvus 找相似 concept
```

V11 当前有两个 collection:

```text
concepts_only_name   # SNOMED
rxnorm_concepts      # RxNorm 药品
```

不是一个 collection 里两个字段,而是两个物理集合。

为什么分开?

```text
SNOMED 擅长疾病/症状/操作。
RxNorm 擅长药品成分。
```

如果 aspirin 去 SNOMED 查,可能查到中毒、过敏、制品等概念,不是干净的药品成分。

所以 V11 加了第二个药品库 RxNorm。

回看:

```text
04_离线建库工具_tools.md
05_StdService_多源向量底座_std_service.md
21_Docker容器化与运行环境.md
```

## 需求 9:怎么决定查 SNOMED 还是 RxNorm?

这一步其实非常简单。

候选 expansion 里有 domain:

```json
{"expansion": "aspirin", "domain": "Drug"}
```

主链路判断:

```python
domain == "Drug" → source = "rxnorm"
其它             → source = "snomed"
```

所以:

```text
ASA → aspirin → Drug → rxnorm_concepts
CP  → chest pain → Condition → concepts_only_name
```

这就是所谓多源路由。

它不是 LLM 自己决定去哪查,而是确定性规则:

```text
domain 决定 source。
```

回看:

```text
01_缩写候选词典_abbr_candidates.md
05_StdService_多源向量底座_std_service.md
06_MedicalRetriever_检索层与规则重排_medical_retriever.md
15_ABBRService_多源路由与标准化反思重检索.md
```

第一次读不要被“agentic RAG”这个词吓到。

这里的 agentic 主要体现在:

```text
系统会根据当前词的 domain 决定查哪个知识源。
```

但这个决策是确定性的。

## 需求 10:Milvus 找回来的候选一定对吗?

不一定。

比如 query:

```text
chest pain
```

Milvus 可能返回:

```text
Chest pain
Chest pain rating
Chest pain scale
```

这些都“相似”,但不都忠实。

所以需要 verifier。

它的任务不是重新判断 CP 要不要扩成 chest pain。

那个已经是 coverage 的活。

verifier 现在问的是:

```text
在这些标准概念候选里,哪个忠实表达 expansion?
如果都不忠实,就不要给码。
```

输出:

```json
{
  "chosen_index": 0,
  "standardization_faithful": true,
  "reason": "Chest pain exactly matches the expansion."
}
```

如果没有忠实概念:

```json
{
  "chosen_index": null,
  "standardization_faithful": false
}
```

状态变成:

```text
CODED    = 找到忠实标准码
WITHHELD = 扩写可以保留,但不给标准码
```

回看:

```text
13_ABBVerifier_标准概念忠实性校验_abbr_verifier.md
14_ABBRService主状态机_expand_verify_with_retry.md
```

# 第三层:标准化不够好怎么办 —— 反思重检索

## 需求 11:如果检索没捞到最规范概念,能不能再搜一次?

比如:

```text
shortness of breath
```

直接搜可能找到:

```text
Difficulty breathing
```

这是忠实的,但更规范的概念可能是:

```text
Dyspnea
```

所以 V11 有反思重检索。

注意它不是让 LLM 直接编一个 concept。

它只让 LLM 提新检索词:

```text
shortness of breath → dyspnea
```

然后重新走:

```text
新检索词
  ↓
Milvus 搜
  ↓
verifier 再判
```

这叫:

```text
LLM 提搜索词,标准概念仍必须来自数据库。
```

回看:

```text
15_ABBRService_多源路由与标准化反思重检索.md
20_LangGraph实验可视化_standardization_graph.md
```

## 需求 12:为什么还要 LangGraph?

生产主链路已经在:

```text
ABBRService.expand_verify_with_retry()
```

LangGraph 不是线上入口。

它只是把单个 mapping 的标准化过程画出来:

```text
route
  ↓
retrieve_snomed / retrieve_rxnorm
  ↓
verify
  ↓
propose_requery
  ↓
re_retrieve
  ↓
re_verify
  ↓
finalize
```

你现在理解它时只要记住一句:

```text
LangGraph 是流程图和 parity 工具,不是生产热路径。
```

回看:

```text
20_LangGraph实验可视化_standardization_graph.md
```

# 第四层:系统到底好不好 —— Benchmark 和 Error Analysis

## 需求 13:怎么知道改完有没有变好?

不能靠手工试一句。

所以有主 benchmark:

```text
74 条 case
当前 71/74 = 0.9594594594594594
```

它主要测:

```text
缩写有没有扩对
多义缩写有没有选对
该弃权时有没有弃权
否定语义有没有保持
fallback 缩写有没有扩对
```

回看:

```text
17_Benchmark评估系统_run_benchmark.md
```

## 需求 14:为什么还要 concept benchmark?

因为主 benchmark 主要判:

```text
abbreviation → expansion
```

但标准化层还要判:

```text
expansion → concept
```

例如:

```text
CP → chest pain
```

主 benchmark 到这里可能已经算对。

但标准化层还要看:

```text
chest pain 是否选到了 Chest pain 这个 SNOMED 概念?
```

所以 concept benchmark 是第二把尺子。

你可以这样记:

```text
主 benchmark       = 扩写考试
concept benchmark = 标准化考试
```

回看:

```text
17_Benchmark评估系统_run_benchmark.md
22_V11项目改进方案与下一步路线.md
```

## 需求 15:错了以后怎么知道错在哪里?

benchmark 只告诉你:

```text
这条 case 错了
```

Error Analysis 进一步告诉你:

```text
是多扩?
是漏扩?
是义项选错?
是语义保持失败?
是低上下文过度扩写?
```

当前错误集中在:

```text
low_context_over_expansion
```

也就是:

```text
ABC / LMN / QRS 这类上下文不够,系统还是扩了。
```

回看:

```text
18_ErrorAnalysis_错误原因分析系统.md
19_MappingSupportVerifier_V10实验与V11删除说明.md
```

19 的意义不是当前有个活模块,而是解释:

```text
以前尝试过用 MappingSupportVerifier 专治低上下文。
但它太保守,误伤正常缩写,所以 V11 删除了它。
```


# 一张最小白板图

如果你现在只能记住一张图,记这张:

```text
POST /expand/simple
  ↓
gate 找缩写
  ↓
候选召回
  ├─ 词典 primary
  └─ LLM fallback
  ↓
coverage 选唯一 expansion
  ↓
确定性 token 替换
  ↓
每个 mapping 独立标准化
  ↓
_route_source(domain)
  ├─ Drug → RxNorm collection
  └─ other → SNOMED collection
  ↓
MedicalRetriever + StdService
  ↓
Milvus 返回候选概念
  ↓
ABBVerifier 选忠实概念
  ├─ 选中 → CODED
  └─ 没有 → WITHHELD
  ↓
必要时 propose_requery 重检索
  ↓
返回 expanded_text + mappings + standardized_entities
```

## 一句话背诵版

> medical-nlp V11 是一个受控的医学缩写扩写和标准化流水线:先用 gate 找缩写,词典或 fallback 召回候选,coverage 根据上下文选唯一 expansion,再用确定性 token 替换生成扩写文本;之后每个 mapping 按 domain 路由到 SNOMED 或 RxNorm 做向量检索,由 verifier 在候选标准概念里选择忠实概念或弃码,必要时提出新检索词重检索。主 benchmark 测扩写,concept benchmark 测标准化,error analysis 定位失败类型。

## 你现在最该形成的脑内模型

不要把它想成 22 个文件。

把它想成 5 个连续问题:

```text
1. 哪些词是缩写?
2. 每个缩写应该扩成什么?
3. 怎么安全替换原文?
4. 扩完以后对应哪个标准医学概念?
5. 系统错了以后怎么评估和定位?
```

项目所有模块都只是这 5 个问题的零件。

如果读某篇文档时不知道它在干嘛,就问:

```text
它在解决上面 5 个问题里的哪一个?
它的输入是什么?
它的输出会被谁消费?
```

能回答这三个,这篇就读懂了。
