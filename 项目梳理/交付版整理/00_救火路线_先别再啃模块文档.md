## 你现在只需要抓住的主线

项目主线不是 22 个模块，而是 6 个动作。

```text
输入病历句子
  ↓
找出可能的缩写
  ↓
为每个缩写召回候选扩写
  ↓
判断候选是否足够可靠，决定扩不扩
  ↓
把可靠缩写确定性替换成完整医学术语
  ↓
把扩写后的术语映射到 SNOMED / RxNorm 标准概念
  ↓
用 LLM 判官校验映射是否忠实，不忠实时反思重检索
  ↓
返回扩写文本、映射结果、失败原因和评估信息
```

这才是你面试时要讲的项目，不是“我有 22 个模块”。

---

## 统一样例

后面所有技术细节都用同一个样例讲，不再每章换例子。

```text
The patient took ASA for CP and denies SOB.
```

你只需要盯住 3 个缩写：

| 缩写 | 可能含义 | 难点 |
|---|---|---|
| ASA | aspirin / American Society of Anesthesiologists / 其他 | 它可能是药品，所以应该走 RxNorm |
| CP | chest pain / cerebral palsy / creatine phosphate | 必须结合上下文判断 |
| SOB | shortness of breath | 较稳定，但仍要走候选和校验 |

这个例子足够覆盖项目核心能力：

- 缩写识别
- 候选召回
- 上下文选择
- 药品与非药品分源
- 向量检索
- LLM 校验
- 失败保守处理

## 接下来学习顺序

不要再从 01 读到 22。按下面顺序学。

### 第 1 关：先理解主状态机

只看：

- `backend/services/abbr_service.py`
- 函数：`expand_verify_with_retry`

你要先知道它在控制什么：

```text
candidate_infos
  → records
  → expanded_text
  → std_cache
  → std_concept
  → mapping_states
```

这一关不追求懂每个子模块，只要懂状态怎么变化：

| 状态 | 含义 |
|---|---|
| `NOT_EXPANDED` | 候选覆盖度不够，不扩写 |
| `PENDING` | 已经选出扩写，等待标准化 |
| `CODED` | 已扩写，并成功映射到标准概念 |
| `WITHHELD` | 已扩写，但标准概念不敢选 |
| `ABSTAIN` | 多轮后仍不可靠，放弃 |

面试说法：

> 我把缩写处理做成了一个显式状态机。每个缩写从候选召回开始就有一条独立 record，后续扩写、标准化、校验、失败原因都写回同一个 record，避免不同模块之间靠隐式字段传递导致问题难排查。

---

### 第 2 关：再理解候选从哪里来

只看：

- `backend/data/abbr_candidates.py`
- `backend/services/abbr_candidate_retriever.py`
- `backend/services/abbr_candidate_fallback_retriever.py`
- `backend/services/abbr_candidate_coverage_evaluator.py`

你要掌握的不是每行代码，而是三层保护：

```text
主候选词典
  ↓ 如果没有
LLM fallback 生成候选
  ↓
coverage evaluator 判断候选覆盖度是否足够
```

面试说法：

> 我没有让 LLM 直接改原文，而是先把缩写扩写变成候选选择问题。优先用结构化候选词典，词典没有再让 LLM 兜底生成候选，最后还有覆盖度闸门，覆盖度不足就不扩写。

---

### 第 3 关：理解为什么是确定性替换

只看：

- `backend/services/abbr_service.py`
- 函数：`_build_expanded_text_deterministic`

你要知道：

- 只替换已经被选中的缩写。
- 用 token 边界避免误替换。
- 从后往前替换避免 offset 错位。
- 不让 LLM 直接重写整句。

面试说法：

> 扩写文本不是让 LLM 生成的，而是用确定性替换生成的。LLM 只参与候选判断和校验，不直接改写整句，这样能减少幻觉和不可控改写。

---

### 第 4 关：理解标准化为什么分 SNOMED 和 RxNorm

只看：

- `backend/services/std_service.py`
- `backend/services/medical_retriever.py`
- `backend/services/abbr_service.py` 的 `_route_source`

现在项目里是两个 Milvus collection：

| 来源 | collection | 用途 |
|---|---|---|
| SNOMED | `concepts_only_name` | 疾病、症状、检查、解剖结构等 |
| RxNorm | `rxnorm_concepts` | 药品 |

不是一个集合两个字段。

路由逻辑很简单：

```python
return "rxnorm" if domain == "Drug" else "snomed"
```

面试说法：

> V11 做了多源标准化。NER 或候选信息判断扩写属于药品时走 RxNorm，否则走 SNOMED。这样避免把 aspirin 这类药品强行拿到 SNOMED 疾病概念里找。

---

### 第 5 关：理解 LLM 判官到底判什么

只看：

- `backend/services/abbr_verifier.py`
- `backend/services/abbr_service.py` 中调用 `verify_mappings` 的地方

它不是重新扩写缩写，而是判断：

```text
这个 expansion
在这句话里
映射到这个标准概念
是否忠实？
```

如果忠实，就选择 `chosen_index`。

如果不忠实，就进入：

```text
WITHHELD
  或
反思生成 requery
  → 重检索
  → 再校验
```

面试说法：

> 我把 LLM 放在 verifier 位置，让它做医学语义忠实性判断。系统先用向量检索给出候选，再让 LLM 在有限候选中选择，而不是让它自由生成 concept id。

---

### 第 6 关：最后才看评估和错误分析

只看：

- `backend/evaluation/run_benchmark.py`
- `backend/evaluation/analyze_errors.py`
- `backend/services/error_collector.py`

你要讲清楚：

- benchmark 评估系统整体表现。
- error collector 记录未解决样例。
- error analysis 把错误归因，指导下一批改动。

面试说法：

> 我后期不是只凭感觉改 prompt，而是把失败样例收集下来，按候选召回失败、覆盖度失败、标准化失败、校验失败等类型做归因，再决定下一轮优化。

## 你现在不要做的事

### 1. 不要继续顺序读 00-22

那套文档现在只用来查：

- 这个类在哪个文件？
- 某个字段是什么意思？
- 某个实验为什么做？

它不是你的学习入口。

### 2. 不要试图一次性理解所有模块

你找工作不需要把每个实验分支都背下来。你需要先讲清楚：

```text
我解决了什么问题
为什么这么拆
核心数据结构怎么流
哪里用确定性逻辑
哪里用 LLM
如何防幻觉
如何评估
```

### 3. 不要先清代码再理解主线

清理盲肠前，必须先确定：

- 主链路是什么
- 哪些是实验
- 哪些是评估
- 哪些只是缓存/日志/旧文档

否则很容易删掉以后才发现测试或文档引用了。
