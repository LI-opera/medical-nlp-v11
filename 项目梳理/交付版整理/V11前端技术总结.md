# V11 前端技术总结

本文总结 Medical NLP V11 当前前端实现。后端主链路见《V11后端技术总结.md》。

## 1. 技术定位

前端是由 FastAPI 托管的原生 HTML/CSS/JavaScript 工作台，不使用 React/Vue 构建链。

~~~text
FastAPI /app
  -> frontend/index.html
  -> frontend_logger.js
  -> app.js ES Module
  -> router / shell / pages / components / state
  -> fetch 调用后端 API
~~~

主要用户场景：

- Analyze：输入临床文本，查看缩写扩写和标准化结果。
- Benchmark Overview：上传 cases，运行 benchmark，查看统计和进度。
- Error Analysis：查看错误交集和 LLM 人话解释。
- Fallback Promotions：审核并写入 fallback 候选。

当前前端服务的是“医学缩写扩写与标准化”主链路，不是完整医学实体 NER 标准化界面。

## 3. 浏览器加载与缓存

index.html 的加载顺序：

1. styles.css。
2. frontend/utils/frontend_logger.js。
3. frontend/app.js。
4. app.js 读取 window.frontendLogger 并装配模块。
5. 健康检查和首次 render。

FastAPI 的 /app 会根据文件修改时间替换：

~~~text
__APP_VERSION__
__LOGGER_VERSION__
__STYLE_VERSION__
~~~

因此 CSS、logger 和 app.js 修改后会获得新的查询参数，降低浏览器继续使用旧文件的概率。benchmark、shell、modal 等容易影响交互的 ES Module 也使用版本参数。

## 4. app.js、router 与 shell

app.js 当前负责：

- 导入 API、state、页面、组件和 router。
- 配置 API client。
- 创建各页面 action 与 renderer。
- 注册 benchmark 完成后的页面刷新策略。
- 绑定全局 error 和 unhandledrejection。
- 执行健康检查和首次渲染。

它是 composition root，不再承载所有页面业务。

router.js 当前路由：

~~~text
analyze
benchmarkOverview
benchmarkErrors
benchmarkPromotions
~~~

shell.js 负责左侧导航、顶部标题、页面容器和事件绑定。页面业务仍由 pages 模块负责。

当前渲染方式是：

~~~text
state 变化
  -> render()
  -> shell.render()
  -> 调用当前 route 的 renderer
  -> 重新生成页面 HTML
  -> bindRouteEvents()
~~~

因此异步轮询不能随意调用全局 render，这是 Benchmark 与 Analyze 解耦的关键。

## 5. 状态管理

state/store.js 是前端状态的单一来源。

主要状态：

~~~text
route / text
  当前页面和 Analyze 输入。

analyzeResult / diagnosis
  单句分析和 LLM 诊断结果。

benchmark / benchmarkUploadJob
  Benchmark 当前结果和后台任务。

errors / triage
  Error Analysis 结构化报告和 LLM triage。

promotions
  Fallback Promotions 候选报告。

singlePromotionItem / singlePromotionApplying
  Analyze 单条 primary 写入状态。

promotionApplyProgress
  批量写入进度。
~~~

刷新浏览器后，内存状态重置。Benchmark 页面会重新从后端 runtime 结果读取，Analyze 结果不会持久化到浏览器。

## 6. API client 与 request_id

api/client.js 统一处理：

- API 地址拼接。
- JSON 请求。
- Content-Type。
- X-Frontend-Request-Id。
- 请求耗时。
- HTTP 错误。
- 后端 request_id。
- api.request_start、api.request_ok、api.request_error。

API 地址默认使用当前页面来源：

~~~text
http://127.0.0.1:8001/app
  -> 请求 127.0.0.1:8001

http://127.0.0.1:8000/app
  -> 请求 127.0.0.1:8000
~~~

正式页面不显示 API 设置栏，但底层 apiBase 逻辑保留。

串联关系：

~~~text
frontend_request_id
  -> X-Frontend-Request-Id
  -> 后端 trace_context
  -> 后端 request_id
  -> 前端 api.request_ok 记录 backend_request_id
~~~

## 7. Analyze 页面

文件：

~~~text
frontend/pages/analyze.js
~~~

模块分为：

~~~text
createAnalyzeActions
  调用 /expand/simple 和 /analysis/diagnose。

createAnalyzeRenderer
  渲染输入、扩写、诊断和 Raw JSON。
~~~

分析流程：

~~~text
点击分析
  -> 清空旧结果
  -> POST /expand/simple
  -> 保存 analyzeResult
  -> 记录 backend_request_id
  -> render
~~~

页面展示：

- 输入文本。
- 扩写文本。
- 当前单句诊断。
- 每个缩写的 source、coverage、status。
- 标准概念、concept_id/code、domain、score。
- Raw JSON 折叠区域。

没有检测到目标缩写时，文本保持原样，并显示“暂无结果”和“未检测到需要扩写的目标缩写。”，这是正常业务状态。

LLM 诊断：

~~~text
已有 analyzeResult
  -> POST /analysis/diagnose
  -> summary
  -> record_notes
  -> next_steps
~~~

单条 primary 写入仅对满足以下条件的候选显示：

~~~text
source == fallback
status == CODED
存在标准概念
~~~

点击后通过 /candidate-promotions/apply-single 确认、进度和结果均由 state 控制。

## 8. Benchmark Overview

文件：

~~~text
frontend/pages/benchmark_overview.js
~~~

页面读取：

~~~text
GET /benchmark/results
POST /benchmark/cases/jobs
GET /benchmark/cases/jobs/{job_id}
~~~

上传 JSON 最小结构：

~~~json
{
  "cases": [
    {
      "id": "demo_001",
      "category": "single_meaning",
      "text": "The patient has SOB.",
      "expected_mappings": [
        {
          "abbreviation": "SOB",
          "expansion": "shortness of breath"
        }
      ]
    }
  ]
}
~~~

任务流程：

~~~text
读取文件
  -> 创建 job
  -> 轮询 job_id
  -> 展示 stage / current / total / progress
  -> 完成后读取本轮结果
~~~

上传期间只更新 Overview 的上传状态容器和按钮，不调用全局 render。因此 Analyze 页面中的输入、结果、LLM 诊断和 primary 弹窗不会被 Benchmark 轮询重建。

任务保护：

~~~text
每次上传生成 active upload token。
轮询同时校验 token 和 job_id。
过期任务不能覆盖当前任务。
~~~

任务完成后：

~~~text
清空 errors / triage / promotions 前端缓存。
当前在 Error Analysis 时刷新错误分析。
当前在 Fallback Promotions 时刷新候选。
当前在 Analyze 时保持 Analyze 不动。
~~~

## 9. Error Analysis

文件：

~~~text
frontend/pages/error_analysis.js
frontend/components/failure_pie.js
frontend/components/triage_report.js
~~~

加载：

~~~text
GET /error-analysis/report
GET /error-analysis/triage
~~~

页面展示：

- 失败总数。
- benchmark mismatch。
- expansion blocked。
- standardization failure。
- benchmark 与标准化失败交集。
- 错误类型饼图。
- LLM triage 人话卡片。

点击错误类型会更新 state.selectedErrorSlice，然后只展示对应 case 的：

~~~text
case id
情况
可能原因
下一步建议
~~~

前端不把大段原始 JSON 作为主要阅读内容，结构化报告仍保留在后端 runtime 文件和 API 中。

## 10. Fallback Promotions

文件：

~~~text
frontend/pages/fallback_promotions.js
~~~

读取：

~~~text
GET /candidate-promotions
~~~

展示：

~~~text
缩写、扩写、domain、support、case ids、new/already exists
~~~

批量写入：

~~~text
确认弹窗
  -> POST /candidate-promotions/apply
  -> 进度条
  -> 成功动画
  -> 刷新候选状态
~~~

Analyze 单条写入复用：

~~~text
POST /candidate-promotions/apply-single
~~~

前端负责交互状态，真正的去重由后端完成。后端按 abbreviation + expansion 去重，因此同一缩写可以保留多个不同扩写，不会因为缩写相同而覆盖旧候选。

## 12. 前端日志

frontend/utils/frontend_logger.js 通过 IIFE 初始化：

~~~text
window.frontendLogger
window.frontendLogUtils
~~~

记录：

~~~text
页面加载、健康检查、路由切换
Analyze 点击、结果、诊断
API 请求开始、成功、失败
Benchmark 上传、job 创建、轮询、完成
Error Analysis 加载
Fallback Promotions 写入
浏览器运行时异常和未处理 Promise
~~~

默认策略：

~~~text
INFO / WARNING 进入 buffer。
ERROR 进入 buffer，并 console.error。
~~~

查看 buffer：

~~~javascript
window.frontendLogger.print()
~~~

打开 INFO 自动输出：

~~~javascript
localStorage.setItem("medicalNlpFrontendLogConsole", "1")
~~~

关闭：

~~~javascript
localStorage.removeItem("medicalNlpFrontendLogConsole")
~~~

临时显示文本 preview：

~~~javascript
localStorage.setItem("medicalNlpFrontendLogPreview", "1")
~~~

前端日志通过 POST /frontend-log 上报到 backend/logs/frontend.jsonl。默认不把完整临床文本写入日志，而是记录 text_len 和 text_hash。

## 13. 页面数据生命周期

~~~text
Analyze
  刷新浏览器后清空当前单句结果。

Benchmark
  读取 backend/evaluation/runtime 的当前结果。
  新任务完成后替换本轮结果。
  新应用会话启动时旧 runtime 结果归档到 archive。

Error Analysis / Fallback Promotions
  读取本轮 benchmark 后处理结果。
  新 benchmark 完成后清空旧缓存并重新读取。
~~~

archive 是历史评估数据，不是当前页面默认数据源。

## 15. 当前边界

1. 前端使用原生 ES Module，没有独立构建和打包流程。
2. render 会重新生成当前页面 HTML，异步任务必须使用局部更新。
3. 页面状态保存在内存，刷新后 Analyze 结果不会保留。
4. Benchmark、Error Analysis 和 Fallback Promotions 的数据来自后端 runtime。
5. 前端不能替代后端的标准化、去重和权限控制。
6. 当前没有独立日志查看页面和远程日志平台。
7. Analyze 只服务缩写主链路，不代表完整医学实体标准化。

## 16. 面试版总结

> 这个前端采用原生 JavaScript ES Module 组织。app.js 是应用装配入口，router 管理 Analyze 和 Benchmark 三个子页面，state/store 保存单一前端状态，api/client 统一处理请求、前后端 request_id 和错误日志，页面业务放在 pages，弹窗、进度、图表和 triage 报告放在 components。Benchmark 上传采用后台 job 和局部状态更新，轮询期间只刷新 Overview 的进度区域，避免影响 Analyze 页面中的 primary 弹窗。前端日志先进入 buffer，再按需输出或上报后端，并通过 frontend_request_id、backend_request_id 和 job_id 串联排查链路。
