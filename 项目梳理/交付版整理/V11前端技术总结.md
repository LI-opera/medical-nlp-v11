# V11 前端技术总结：医学缩写标准化工作台、Benchmark 仪表盘与错误复盘界面

本文只总结当前 medical-nlp V11 的前端部分。后端主链路、Milvus、Benchmark 脚本和 Docker 建库流程见后端技术总结文档。

---

## 1. 当前前端是什么

当前前端不是 React / Vue / Vite 项目，而是一个轻量原生单页应用：

```text
frontend/index.html
frontend/app.js
frontend/styles.css
frontend/utils/frontend_logger.js
```

它由 FastAPI 后端直接托管：

```text
GET /app
GET /frontend/app.js
GET /frontend/styles.css
GET /frontend/utils/frontend_logger.js
```

也就是说：

```text
前端不是单独启动一个 Node 服务。
用户访问 http://127.0.0.1:8000/app 时，后端返回 index.html。
index.html 再加载 app.js / styles.css / frontend_logger.js。
```

这也是 Dockerfile 必须复制 `frontend/` 的原因：

```dockerfile
COPY frontend ./frontend
```

否则容器里只有 backend，`/app` 页面就找不到前端资源。

---

## 2. 前端整体定位

当前前端的定位是：

```text
Medical NLP V11 Workbench
```

它不是普通展示页，也不是纯后台管理页，而是一个围绕后端缩写标准化能力搭建的工作台：

```text
1. Analyze
   单句缩写扩写与标准化。

2. Benchmark Overview
   Benchmark 总览、上传 cases、运行 benchmark、查看准确率和失败案例。

3. Benchmark Error Analysis
   查看 error_analysis_report.json 和 error_triage_report.md。

4. Benchmark Fallback Promotions
   查看 fallback 成功候选，并写入 primary 缩写词典。
```

导航结构：

```text
Analyze

Benchmark
  Overview
  Error Analysis
  Fallback Promotions
```

这个导航结构和后端能力是一一对应的：

```text
Analyze                  -> /expand/simple + /analysis/diagnose
Benchmark Overview       -> /benchmark/summary + /benchmark/results + /benchmark/cases/jobs
Error Analysis           -> /error-analysis/report + /error-analysis/triage
Fallback Promotions      -> /candidate-promotions + /candidate-promotions/apply
```

## 3. 前端文件分工

### 3.1 frontend/index.html

职责：

```text
1. 提供页面根节点 <div id="app"></div>。
2. 加载 styles.css。
3. 先加载 frontend_logger.js。
4. 再加载 app.js。
```

当前结构：

```html
<link rel="stylesheet" href="/frontend/styles.css?v=__STYLE_VERSION__" />
<script src="/frontend/utils/frontend_logger.js?v=__LOGGER_VERSION__"></script>
<script src="/frontend/app.js?v=__APP_VERSION__"></script>
```

这里的版本号不是手写固定值，而是由后端 `/app` 动态替换：

```text
__STYLE_VERSION__
__LOGGER_VERSION__
__APP_VERSION__
```

来源是文件修改时间。

意义：

```text
修改 app.js / styles.css / frontend_logger.js 后，浏览器会自动拉取新版本。
避免浏览器缓存旧 JS 导致“我明明改了代码，页面却没变化”。
```

这是前端主文件。

职责：

```text
1. 保存全局 state。

3. 封装 API 请求。
4. 渲染 Analyze / Benchmark / Error Analysis / Promotions 页面。
5. 绑定按钮事件。
6. 处理上传 benchmark cases 的异步 job 轮询。
7. 处理 fallback candidate 写入 primary。
8. 触发前端日志。
```

它不是组件化框架，而是用：

```text

renderBenchmark()
renderErrors()

bindRouteEvents()
```

组成一个原生 SPA。

## 4. 全局状态 state

前端没有 Redux / Pinia 这类状态库，而是直接在 `app.js` 中维护一个全局对象：

```js
const state = {
  route: "analyze",
  text: samples[0],
  apiBase: localStorage.getItem("medicalNlpApiBase") || "http://127.0.0.1:8000",
  health: null,
  analyzing: false,
  analyzeResult: null,
  diagnosis: null,
  benchmark: null,
  benchmarkUploading: false,
  benchmarkUploadJob: null,
  errors: null,
  triage: null,
  selectedErrorSlice: "all",
  promotions: null,
  applyingPromotions: false,
  singlePromotionResults: {}
}
```

可以把它理解为：

```text
整个页面当前处于什么状态，全靠 state 描述。
render() 根据 state 重新生成 DOM。
bindRouteEvents() 再给新 DOM 绑定事件。
```

当前模式的优点：

```text
简单直接。
没有构建工具依赖。
适合项目展示和交付。
Docker 中只需要静态文件即可。
```

缺点：

```text
app.js 已经较大。
后续如果页面继续复杂，应考虑拆分模块或迁移到 React/Vue。
```

## 5. 路由机制

当前前端不是浏览器 URL 路由，而是内部 state 路由：

```js
state.route
```

路由配置：

```js
const routes = {
  analyze: {},
  benchmarkOverview: {},
  benchmarkErrors: {},
  benchmarkPromotions: {},
}
```

切换路由：

```js
setRoute(route)
```

切换后做三件事：

```text
1. 更新 state.route。
2. 触发 render()。
3. 根据路由懒加载数据。
```

懒加载逻辑：

```js
function loadRouteData(route) {
  if (route === "benchmarkOverview" && !state.benchmark) loadBenchmark();
  if (route === "benchmarkErrors" && !state.errors) loadErrors();
  if (route === "benchmarkPromotions" && !state.promotions) loadPromotions();
}
```

这意味着：

```text
第一次进入某个页面时才请求对应后端数据。
点击刷新按钮可以重新拉取。
```

## 6. API 请求封装

前端统一通过：

```js
fetchJson(path, options)
```

调用后端。

它负责：

```text
1. 拼接 apiBase。
2. 设置 Content-Type。
3. 自动生成 frontend_request_id。
4. 设置 X-Frontend-Request-Id 请求头。
5. 记录 api.request_start。
6. 解析 JSON。
7. 记录 api.request_ok / api.request_error。
8. 把后端返回的 request_id 记为 backend_request_id。
```

关键意义：

```text
前端一次点击
  -> frontend_request_id
  -> 后端 request_id
  -> app.jsonl / pipeline.jsonl / dependency.jsonl
```

这样前后端日志能串起来。

## 9. 当前单句诊断卡片

函数：

```js
renderSingleDiagnosis()
```

数据来源：

```text
result.mapping_states
result.standardized_entities
```

每个 record 会显示：

```text
abbreviation
Expansion
status
source
coverage
failure_type
failure_subtype
standard concept
concept_id / code
concept domain
score
failure reason
suggestion
```

其中：

```text
Expansion 是缩写扩写词。
standard concept 是标准医学概念。
```

例如：

```text
SOB
Expansion: shortness of breath
standard concept: Dyspnea
```

这表示：

```text
shortness of breath 是系统扩写出来的临床表达。
Dyspnea 是标准概念库中匹配到的标准术语。
```

两者不要求字符串完全相同，只要医学语义能忠实对应，就可以 CODED。

## 14. Benchmark 分类表现

函数：

```js
renderCategoryStackedBars(categoryStats)
```

输入：

```text
category_stats
```

展示方式：

```text
每个 category 一条堆叠条。

```

这个设计替代了旧版饼图。


```text
total_cases
correct
accuracy
category_stats
results
```

## 22. 前后端 request_id 串联

每次 `fetchJson()` 请求都会生成：

```text

```

```text
X-Frontend-Request-Id
```

后端 `/expand/simple` 返回：

```text
request_id
```

前端将其记录为：

```text
backend_request_id
```

这样一条 Analyze 请求可以从浏览器追到后端：

```text
frontendLogger.buffer
  -> frontend.jsonl
  -> app.jsonl
  -> pipeline.jsonl
  -> dependency.jsonl
```

这套设计对排查很重要：

```text
用户看到页面分析失败
  -> 查前端 api.request_error
  -> 找 frontend_request_id
  -> 对后端 request_id
  -> 查 pipeline / dependency 是否 LLM 或 Milvus 失败
```

## 23. 当前前端的技术取舍

### 23.1 为什么没用 React / Vue

当前项目重点是后端 NLP 链路和交付展示，不是复杂前端工程。

原生 SPA 的优点：

```text
1. 不需要 npm install。
2. 不需要构建产物。
3. Docker 镜像直接复制 frontend 即可。
4. FastAPI 可以直接托管。
5. 对项目展示足够轻量。
```

缺点：

```text
1. app.js 越来越大。
2. render 函数较长。
3. 状态变化靠手动维护。
4. 复杂交互继续增加后，维护成本会上升。
```

后续如果要做产品化，可以考虑：

```text
React + Vite
Vue + Vite
TypeScript
组件拆分
图表库
更规范的状态管理
```

但当前 GitHub 展示版不一定需要立刻迁移。

---

### 23.2 为什么 Error Analysis 饼图不用图表库

当前图表只有一个饼图，且交互简单：

```text
点击扇区筛选错误解释。
```

手写 SVG 的好处：

```text
减少依赖。
不需要构建。
和当前原生 SPA 保持一致。
```

缺点：

```text
如果后续要做复杂图表，应该引入 ECharts / Chart.js。
```

### 23.3 为什么保留 API Base 输入框

它对普通医疗人员不一定重要，但对本项目交付很有用：

```text
1. 本地后端可能跑在 8000 或 8001。
2. Docker 环境默认 8000。
3. 远程部署时 API 地址会变化。
4. 不需要改前端代码即可切换后端。
```

后续如果面向真实医疗人员，可以隐藏到设置页。

## 25. 面试时可以这样讲前端

可以这样表达：

```text
这个项目的前端是一个由 FastAPI 托管的原生单页工作台，没有单独引入 React 或 Vite。这样做是为了降低部署复杂度，Docker 里只要复制 frontend 静态文件，后端 /app 就能直接提供页面。

前端主要分成 Analyze 和 Benchmark 两个大模块。Analyze 面向单句缩写标准化，调用 /expand/simple 展示 expanded_text、mapping_states 和标准概念信息，并且可以对 fallback 且 CODED 的单条结果写入 primary 候选库。Benchmark 模块包含 Overview、Error Analysis 和 Fallback Promotions，支持上传 benchmark cases 后异步运行 job，展示分类正确率、失败案例、错误类型饼图、LLM triage 人话解释，以及 fallback 成功候选沉淀。

工程上我封装了 fetchJson 统一处理 API Base、frontend_request_id、错误日志和响应日志；前端日志通过 frontend_logger.js 维护 buffer，并可通过 /frontend-log 落盘到 backend/logs/frontend.jsonl。这样前端点击、API 请求和后端 request_id 可以串联起来排查问题。
```

## 26. GitHub 交付前前端检查清单
