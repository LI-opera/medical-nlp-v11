# 13_盲肠清单：主链路外残留与清理建议

> 这一章不是学习文档，而是一次交付前的“盲肠审计”。
>
> 目标：先找出项目里哪些东西是主链路、哪些是辅助能力、哪些是旧链路残留、哪些是运行产物，后续再按清单逐步清理。

---

## 0. 本次审计结论先说

不要现在就大删。

这次审计后，我建议把项目内容分成 5 类：

| 类别 | 处理策略 | 代表 |
|---|---|---|
| 主链路核心 | 保留 | `api/main.py`, `abbr_service.py`, `std_service.py`, `medical_retriever.py`, verifier, candidate/coverage/fallback |
| 仍有价值但需重构 | 先保留，拆出能力后再删壳 | `medical_standardizer.py` |
| 明确盲肠 | 可优先删除或归档 | `backend/services/test.py`, `error_taxonomy_old.py`, 旧运行结果 JSON |
| 交付不该上传 | 加 `.gitignore` 或从 Git 移除 | 大 CSV、日志、benchmark 运行结果、`.env` |
| 学习/施工资料 | 移到 `archive/` 或 `docs/internal/` | `项目梳理/批次*_Codex指令.md`, V9/V11旧模块文档 |

最重要的一点：

> `MedicalStandardizer` 这条“整句级 NER + 检索”旧编排，现在不是 `/expand/simple` 的主标准化链路；但它里面用到的 `NERService` 仍然有用，因为 fallback 之后需要 NER 帮候选 expansion 推断 domain。  
> 所以后续不是直接删 NER，而是把 NER 从 `MedicalStandardizer` 这个旧壳里拆出来。

## 1. 本次怎么检查的

我按 4 条线查：

### 1. API 主入口

入口是：

```text
backend/api/main.py
```

启用接口是：

```text
POST /expand/simple
```

真实调用：

```text
expand_abbreviation_simple()
  -> get_service()
  -> ABBRService()
  -> expand_verify_with_retry()
```

### 2. 主服务调用链

核心链路是：

```text
ABBRService
  -> ABBRCandidateRetriever
  -> ABBRCandidateFallbackRetriever
  -> ABBRCandidateCoverageEvaluator
  -> NERService.is_medical()
  -> MedicalRetriever
  -> StdService
  -> ABBVerifier
  -> error_collector
```

### 3. 静态 import 图

通过 Python AST 扫了 `backend/**/*.py` 的本地 import。

结论：

- `MedicalStandardizer` 被 `ABBRService` import 和初始化；
- 但 `MedicalStandardizer.standardize(text)` 没有被主 API 调用；
- `NERService` 被 `MedicalStandardizer` 使用，并通过 `ABBRService.self.standardizer.ner_service` 间接用于 fallback domain 推断；
- `backend/services/test.py` 没有被任何主链路引用；
- 根目录 `test_*.py` 多数是开发期 smoke check，不是生产链路依赖。

### 4. 文件类型检查

额外看了：

- `backend/services/`
- `backend/evaluation/`
- `backend/tools/`
- `backend/data/`
- 根目录 Docker 和 `.gitignore`
- `项目梳理/` 施工文档

## 2. 当前真正的主链路

GitHub 交付时要围绕这条链路保留：

```text
backend/api/main.py
  -> ABBRService.expand_verify_with_retry()
      -> _get_abbreviation_candidates()
          -> ABBRCandidateRetriever
          -> ABBRCandidateFallbackRetriever
          -> ABBRCandidateCoverageEvaluator
          -> NERService.is_medical()
      -> _build_expanded_text_deterministic()
      -> MedicalRetriever.retrieve()
          -> StdService.search_similar_terms()
      -> ABBVerifier.verify_mappings()
      -> _reflect_refine_standardization()
      -> error_collector
```

一句话：

> V11 主链路已经是 token / abbreviation record 级处理，不再是旧的“整句 NER 后把整句实体全部标准化”。

所以清理时要保护：

- abbreviation candidate；
- fallback；
- coverage；
- deterministic replacement；
- per-record status；
- source route；
- retriever；
- std_service；
- verifier；
- reflection；
- benchmark/error analysis。

## 4. P1 清单：不能直接删，要先重构再清理

### P1-1：`MedicalStandardizer` 是旧整句级编排

路径：

```text
backend/services/medical_standardizer.py
```

它做的事：

```text
text
  -> NERService.extract_entities(text)
  -> 每个 entity 调 MedicalRetriever.retrieve()
  -> 返回整句实体候选
```

也就是旧的：

```text
整句级 NER + 标准化候选检索
```

但 V11 主链路现在不是这样。

V11 主链路是：

```text
每个 abbreviation record
  -> expansion
  -> 按 domain/source 查 SNOMED 或 RxNorm
  -> verifier 选择标准概念
```

真实证据：

```text
ABBRService.__init__()
  self.standardizer = MedicalStandardizer()
  self.ner_service = self.standardizer.ner_service
```

但在 `expand_verify_with_retry()` 主流程里，没有调用：

```python
self.standardizer.standardize(...)
```

所以它当前的真实地位是：

```text
旧整句标准化壳 + NERService 容器
```

### 为什么不能直接删

因为 fallback 后仍然需要：

```python
self.ner_service.is_medical(candidate.get("expansion"))
```

这段用于给 fallback LLM 生成的候选扩写词推断 domain：

```text
aspirin -> Drug -> RxNorm
chest pain -> Condition -> SNOMED
```

如果直接删 `MedicalStandardizer`，`ABBRService` 现在拿不到 `self.ner_service`。

### 正确清理方式

第一步：在 `ABBRService` 里直接引入 NER：

```python
from services.ner_service import NERService
```

然后：

```python
self.ner_service = NERService()
```

第二步：删除或归档：

```python
self.standardizer = MedicalStandardizer()
```

第三步：确认没有其它地方还需要整句标准化能力。

如果不需要：

```text
medical_standardizer.py -> archive/old_pipeline/
test_medical_standardizer.py -> archive/old_pipeline/
```

如果还想保留“未来整句标准化能力”：

```text
medical_standardizer.py 改名为 sentence_standardizer_legacy.py
README 标注：legacy / experimental
```

推荐：

```text
短期先拆 NER 注入。
中期将 MedicalStandardizer 归档。
不要让它继续假装是主标准化模块。
```

## 12. MedicalStandardizer 的最终建议

你提到的判断是对的：

> 之前的 `MedicalStandardizer` 是句子级操作；现在主流程已经是 token / abbreviation record 级处理，所以旧的句子级标准化编排可以清理。

但要加半句：

> 其中的 NER 能力不能删，因为 fallback 后仍然需要 NER 给候选扩写词推断 domain。

推荐最终结构：

```text
backend/services/ner_service.py
backend/services/domain_resolver.py
backend/services/abbr_service.py
```

`domain_resolver.py` 可以做：

```text
expansion
  -> NERService.is_medical(expansion)
  -> NER_LABEL_TO_DOMAIN
  -> domain
```

这样 `ABBRService` 只依赖：

```text
NER/domain resolver
```

不再依赖旧的：

```text
MedicalStandardizer
```

然后：

```text
medical_standardizer.py -> archive/old_pipeline/
```

这就是最干净的清理方式。

## 13. 面试时怎么解释这次清理

可以这样说：

> 项目早期有一条整句级标准化链路：先 NER 抽实体，再对整句实体查 SNOMED。后来 V11 主链路改成 abbreviation record 级处理，每个缩写从候选、coverage、确定性替换，到 SNOMED/RxNorm 检索和 verifier 独立流转。  
> 所以我在交付前做了盲肠清理：保留真正被主链路使用的 NER 能力，但把旧的 MedicalStandardizer 整句编排降级为 legacy；同时清理调试脚本、运行产物、大数据 CSV、旧错误分类和施工文档，让 GitHub 版本只呈现主链路、评估链路和离线建库链路。

这句话很有用。

它不是说：

```text
我删了一堆文件。
```

而是说：

```text
我把项目从历史迭代状态收敛成可交付状态。
```
