# V11 后端技术总结

本文只总结 Medical NLP V11 的后端实现，作为项目理解、面试讲解和 GitHub 交付的技术说明。前端页面与交互设计另有独立文档。

---

## 1. 项目定位

V11 后端的核心能力是：

~~~text
医学缩写扩写
    -> 上下文筛选
    -> 标准概念检索
    -> 候选验证
    -> 结构化输出
~~~

项目聚焦于“医学缩写扩写与标准化”，不等同于覆盖整句所有医学实体的通用 NER 标准化系统。

典型输入：

~~~text
The patient has SOB and CP.
~~~

典型处理结果：

~~~text
SOB -> shortness of breath -> Dyspnea
CP  -> chest pain          -> Chest pain
~~~

如果输入已经是完整医学术语，当前主链路不会把它当作缩写任务重新处理。未来完整实体标准化应设计独立的 /full-standardize 路由，避免混入当前链路。

---

## 2. 后端总体架构

~~~text
FastAPI API 层
    -> ABBRService 缩写主链路
        -> primary 候选召回
        -> fallback LLM 候选召回
        -> coverage 上下文筛选
        -> 确定性文本替换
        -> StdService 标准概念检索
        -> verifier 候选验证
        -> reflection 受控重检索
    -> 结构化响应

评估闭环：
benchmark
    -> error_analysis_report
    -> error_triage
    -> fallback_candidate_promotions
    -> 人工确认后写回 primary

基础设施：
Milvus + etcd + MinIO
LLM + embedding
结构化 JSONL 日志
Docker Compose
~~~

后端目录：

~~~text
backend/
├─ api/                 # FastAPI 接口与 Pydantic 模型
├─ data/                # primary 缩写候选词典
├─ services/            # 生产业务服务
├─ utils/               # LLM、embedding、日志和追踪基础设施
├─ evaluation/          # benchmark、错误分析和候选沉淀
├─ tools/               # 数据构建、检查和 Milvus 建库脚本
├─ graph/               # LangGraph 参考实现和流程图实验
├─ tests/               # 后端测试
└─ logs/                # 运行时日志
~~~

### 3.1 主分析接口

~~~http
POST /expand/simple
Content-Type: application/json
X-Frontend-Request-Id: optional-client-request-id
~~~

请求：

~~~json
{
  "text": "The patient has SOB and CP."
}
~~~

主要响应字段：

~~~text
request_id
success
expansion_success
standardization_success
success_breakdown
expanded_text
mappings
standardized_entities
mapping_states
~~~

字段职责：

~~~text
expanded_text
    确定性替换后的文本。

mappings
    缩写到扩写的简明映射。

standardized_entities
    最终绑定到标准概念的实体。

mapping_states
    每个缩写 record 的完整过程状态、证据和失败原因。

request_id
    后端本次请求的追踪编号。
~~~

### 3.2 三层成功状态

~~~text
success
    当前业务流程是否达到最终成功条件。
    对缩写主链路而言，目标 record 全部 CODED 时才为 true。

expansion_success
    目标缩写是否都拿到了 expansion。

standardization_success
    目标缩写是否都成功绑定到标准概念并成为 CODED。

success_breakdown
    target_count、expanded_count、coded_count、
    withheld_count、not_expanded_count 等数量拆解。
~~~

这样可以区分：

~~~text
没有扩写候选
    -> expansion_success=false

扩写成功但没有忠实标准概念
    -> expansion_success=true
    -> standardization_success=false
    -> record.status=WITHHELD

扩写和标准化都完成
    -> success=true
~~~

## 4. ABBRService 主状态机

核心文件：

~~~text
backend/services/abbr_service.py
~~~

核心入口：

~~~python
ABBRService.expand_verify_with_retry(text, max_retries=2)
~~~

一条输入句子是一个 case，句子中的每个目标缩写是一个 record。record 是本项目最重要的业务状态载体。

典型 record：

~~~json
{
  "abbreviation": "SOB",
  "source": "primary",
  "expansion": "shortness of breath",
  "domain": "Condition",
  "status": "CODED",
  "failure": null
}
~~~

状态含义：

~~~text
PENDING
    已经有扩写，等待标准化。

CODED
    扩写可信，并成功绑定标准概念。

WITHHELD
    扩写可信，但没有足够忠实的标准概念，安全拒绝编码。

NOT_EXPANDED
    没有得到可用扩写。

ABSTAIN
    当前流程最终放弃该 record。
~~~

重要原则：

~~~text
case 是统计和业务展示单位。
record 是句子内部缩写的诊断单位。
一句话有多个 record 时，只要一个 record 失败，整个 case 的标准化结果就可能失败。
~~~

## 5. 目标缩写识别

V11 当前不是对整句做完整 NER，而是先做 token 级目标缩写识别。

ABBRService 会综合以下信息判断 token 是否进入缩写流程：

~~~text
1. token 是否在 primary 词典中。
2. 未知 token 是否具有合理的全大写缩写形态。
3. token 长度是否合理。
4. token 是否更像普通词、数字或无关文本。
~~~

目标是减少普通词误判、无意义的 fallback 调用和低上下文过度扩写。

如果输入没有目标缩写，例如：

~~~text
The patient has shortness of breath and chest pain.
~~~

系统可能原样返回文本，并提示：

~~~text
未检测到需要扩写的目标缩写。
~~~

这表示没有进入缩写链路，不代表 API 失败。

## 6. 候选召回：primary 与 fallback

### 6.1 Primary 候选词典

文件：

~~~text
backend/data/abbr_candidates.py
~~~

一个缩写可以保留多个扩写。primary 只提供候选，不直接决定最终答案，最终选择交给 coverage。

示例：

~~~python
ABBR_CANDIDATES = {
    "SOB": [
        {"expansion": "shortness of breath", "domain": "Condition"},
    ],
    "PA": [
        {"expansion": "physician assistant", "domain": "Occupation"},
        {"expansion": "posteroanterior", "domain": "Observation"},
    ],
}
~~~

### 6.2 Fallback 候选召回

文件：

~~~text
backend/services/abbr_candidate_fallback_retriever.py
~~~

fallback 负责在 primary 没有足够候选时调用 LLM，并记录候选、原因和错误证据。

fallback 不负责：

~~~text
直接改写整句
跳过 coverage
直接写入 primary
直接决定标准概念
~~~

它是补充召回来源，不是绕过系统约束的第二条主链路。

## 7. Coverage 上下文筛选

文件：

~~~text
backend/services/abbr_candidate_coverage_evaluator.py
~~~

coverage 回答：

~~~text
候选扩写在当前句子中是否合理？
多个候选中哪个最适合当前语境？
是否应该保守地不扩写？
~~~

coverage 不负责标准概念检索，只决定 expansion 是否可以进入下一阶段。

典型输出：

~~~json
{
  "coverage_ok": true,
  "confidence": 0.95,
  "issues": [],
  "best_expansion": "shortness of breath"
}
~~~

当前 NOT_EXPANDED 会保留更细的 failure 信息，例如：

~~~text
NO_CANDIDATES
CANDIDATES_REJECTED_BY_COVERAGE
AMBIGUOUS_LOW_CONTEXT
FALLBACK_ERROR
~~~

同时尽量记录 primary/fallback 候选数量、fallback_reason、coverage_issues 和 suggestion，供错误分析使用。

## 8. 确定性扩写

扩写文本由 ABBRService 根据原始文本和当前 records 重新渲染：

~~~text
原始 text 是事实源。
records 是状态源。
expanded_text 是渲染结果。
~~~

每一轮不在上一轮 expanded_text 上继续替换，避免 retry 或 reflection 造成状态残留。

替换规则：

~~~text
有 expansion 且不是 ABSTAIN 的 record 才参与替换。
NOT_EXPANDED 不替换。
ABSTAIN 不替换。
WITHHELD 可以替换，因为扩写本身可信，只是标准概念不敢绑定。
~~~

## 9. 标准概念检索

### 9.1 StdService

文件：

~~~text
backend/services/std_service.py
backend/services/medical_retriever.py
~~~

主要方法：

~~~python
StdService.search_similar_terms(query, limit=5, source="snomed")
~~~

当前使用两个独立的 Milvus collection：

~~~text
SNOMED CT:
    concepts_only_name

RxNorm:
    rxnorm_concepts
~~~

这不是一个集合里的两个字段，而是两个逻辑集合。source 用来选择 collection，domain 用来辅助判断候选的医学领域和检索策略。

路由规则：

~~~text
domain == Drug
    -> RxNorm

其它 domain
    -> SNOMED CT
~~~

环境变量：

~~~env
MILVUS_URI=http://127.0.0.1:19530
MILVUS_COLLECTION_NAME=concepts_only_name
MILVUS_RXNORM_COLLECTION=rxnorm_concepts
~~~

Docker Compose 会覆盖为：

~~~env
MILVUS_URI=http://milvus:19530
~~~

### 9.2 检索结果

标准化候选通常包含：

~~~text
concept_id
concept_name
domain_id
concept_code
score
rerank_score
~~~

向量相似不等于医学语义等价，所以候选不能直接作为最终结果。

## 10. Verifier 与反思重检索

验证文件：

~~~text
backend/services/abbr_verifier.py
~~~

verifier 判断检索候选是否真正表达当前 expansion，并决定 CODED 或 WITHHELD。

例如：

~~~text
Airway, Breathing, Circulation
~~~

检索可能返回：

~~~text
Breathing
Ventilation
Route of breathing
~~~

这些概念可能词面相近，但不能代表组合实体本身，因此安全结果可以是 WITHHELD。

反思机制只用于改进检索查询：

~~~text
当前标准化候选质量不足
    -> verifier 提出新的同义检索词
    -> 再次检索 Milvus
    -> verifier 重新选择
    -> 只有质量严格提升时才采纳
~~~

它不会让 LLM 直接发明标准概念，也不会绕过 verifier。

## 11. Milvus 建库

脚本：

~~~text
backend/tools/milvus/rebuild_milvus.py
backend/tools/milvus/rebuild_rxnorm_milvus.py
~~~

输入：

~~~text
backend/data/snomed_clinical.csv
backend/data/rxnorm_clinical.csv
~~~

建库流程：

~~~text
读取 CSV
    -> 加载 embedding 模型
    -> 连接 Milvus
    -> 重建 collection schema
    -> 批量生成向量
    -> 批量插入
    -> flush / load
    -> 执行检索自测
~~~

执行：

~~~powershell
docker compose exec api python tools/milvus/rebuild_milvus.py
docker compose exec api python tools/milvus/rebuild_rxnorm_milvus.py
~~~

rebuild 会重建 collection，不是增量追加。CSV、BGE-M3 cache 和 Milvus 数据体积较大，通常不提交 GitHub。

统一入口：

~~~text

~~~

~~~text
examples/benchmarks/abbr_benchmark_cases.json
~~~

~~~

并行运行：

~~~powershell
$env:BENCH_WORKERS="2"

~~~

代码调用也可以直接传入：

~~~python

~~~

网页上传使用异步 job：

~~~text
POST /benchmark/cases/jobs
    -> 创建 job_id
    -> 后台执行 run_benchmark
    -> 前端轮询 /benchmark/cases/jobs/{job_id}
    -> 完成后执行后处理
~~~

同步接口仍保留：

~~~text
POST /benchmark/cases/run
~~~

Benchmark 的统计单位是 case；record 只用于解释 case 的成功或失败。

Benchmark 会关注：

~~~text
expected_mappings
predicted_mappings
mapping_correct
expected_text_contains
text_check
mapping_states
~~~

必须区分：

~~~text
benchmark correct
    测试集 gold 与预测是否一致。

business success
    业务主链路是否完成。

expansion_success
    缩写是否都扩写。

standardization_success
    缩写是否都成功 CODED。
~~~

## 14. 错误分析链路

结构化分析：

~~~text
backend/evaluation/error_analysis_report.py
~~~

输入：

~~~text
backend/evaluation/runtime/benchmark_results.json
~~~

输出：

~~~text
backend/evaluation/runtime/error_analysis_report.json
~~~

LLM 人话归因：

~~~text
backend/evaluation/error_triage.py
~~~

输出：

~~~text
backend/logs/triage/error_triage_report.md
~~~

失败类型：

~~~text
benchmark_mismatch
    gold 与预测不一致。

expansion_blocked
    目标缩写没有得到可接受扩写。

standardization_failure
    有扩写，但没有全部成功 CODED。
~~~

一个 case 可以同时拥有多个标签，例如 benchmark + standardization。这样可以保留错误交集，不把所有失败粗略归为单一原因。

## 15. Fallback 候选沉淀

脚本：

~~~text
backend/evaluation/collect_fallback_candidate_promotions.py
backend/evaluation/apply_fallback_candidate_promotions.py
~~~

候选进入推荐列表通常需要：

~~~text
来源是 fallback
record 状态是 CODED
存在标准概念
满足当前沉淀规则
~~~

产物：

~~~text
backend/evaluation/runtime/fallback_candidate_promotions.json
backend/evaluation/runtime/fallback_candidate_promotions.md
~~~

人工确认后写入：

~~~text
backend/data/abbr_candidates.py
~~~

写入规则：

~~~text
按 abbreviation + expansion 去重。
同一个 abbreviation 可以保存多个 expansion。
新增候选追加到列表，不覆盖已有候选。
每批写入保留一个时间批次注释。
~~~

闭环：

~~~text
fallback 成功
    -> 标准化 CODED
    -> 生成候选沉淀报告
    -> 人工审核
    -> 写入 primary
    -> 后续请求优先使用 primary
~~~

## 17. 日志系统

实现：

~~~text
backend/utils/structured_logger.py
backend/utils/trace_context.py
~~~

日志文件：

~~~text
backend/logs/app.jsonl
backend/logs/dependency.jsonl
backend/logs/pipeline.jsonl
backend/logs/benchmark.jsonl
backend/logs/audit.jsonl
backend/logs/frontend.jsonl
~~~

职责：

~~~text
app.jsonl
    API 启动、请求级事件和接口异常。

dependency.jsonl
    LLM、embedding、Milvus、collection load 和向量检索。

pipeline.jsonl
    ABBRService 主链路阶段、状态和耗时。

benchmark.jsonl
    benchmark job、case 进度和后处理。

audit.jsonl
    写入 primary 等有副作用的操作。

frontend.jsonl
    浏览器通过 POST /frontend-log 上报的前端事件。
~~~

日志与 record 的区别：

~~~text
record 是业务响应的一部分，回答“这个缩写最终怎样”。
log 是运行轨迹，回答“请求何时进入哪个阶段、依赖是否可用、哪里耗时或异常”。
~~~

追踪字段：

~~~text
request_id
frontend_request_id
job_id
case_id
~~~

日志默认不记录完整临床文本，而记录长度和 hash；是否记录短文本预览由 LOG_TEXT_PREVIEW 控制。

## 20. 当前边界与风险

1. 主链路处理医学缩写，不承诺覆盖整句所有医学实体。
2. fallback、coverage、verifier 和 triage 依赖可用的 LLM API。
3. 标准化效果依赖 Milvus collection、embedding 模型、数据质量和检索阈值。
4. WITHHELD 是安全拒绝，不应解释成标准化成功。
5. benchmark accuracy 是 case 级 gold 指标，不等价于 record 级成功率。
6. SNOMED CT、RxNorm、模型和向量数据需要单独确认许可证与分发权限。
7. 当前 LangGraph 代码是参考实现和流程展示，不作为 FastAPI 生产主入口。

## 22. 面试版总结

> 这个项目的后端是一个面向临床文本的医学缩写标准化系统。请求进入 FastAPI 后，由 ABBRService 按 record 编排缩写候选召回、fallback、coverage 和确定性替换；扩写结果再由 StdService 根据 domain 路由到 SNOMED CT 或 RxNorm 对应的 Milvus collection 做向量检索，最后由 verifier 判断候选是否忠实，必要时进行受控反思重检索。系统没有把所有过程压缩成一个 success，而是返回 expansion_success、standardization_success、success_breakdown 和 mapping_states，便于前端展示和错误分析。Benchmark 结果会继续进入 error analysis、LLM triage 和 fallback candidate promotions，经过人工审核后沉淀回 primary 词典，形成可迭代闭环。
