## 1.success判定很宽松不规范

success 是怎么计算的 主状态机里：

```
python
success = len(_expanded(records)) > 0 and all(
    r["status"] in ("CODED", "WITHHELD") for r in _expanded(records)
)
```

拆开看： ### 条件 1：至少有一个扩写

```
python
len(_expanded(records)) > 0
```

如果一句话里没有任何可靠扩写：

```
text
success = False
```

\### 条件 2：所有已经扩写的 record 都到达终态

```
python
r["status"] in ("CODED", "WITHHELD")
```

也就是说：

```
text
只要一个缩写已经被扩写，它最后要么 CODED，要么 WITHHELD。
不能还停在 PENDING。
```

为什么 `WITHHELD` 也算 success？ 因为这里的 success 更偏向：

```
text
扩写流程是否完成，而不是每个扩写是否都有标准编码。
```

> ````
> **更准确的语义应该拆开**
> 我建议不要把一个 `success` 扛所有含义，而是拆成 3 层：
> 
> ```
> request_success: API/流程是否正常跑完
> expansion_success: 目标缩写是否都成功扩写
> standardization_success: 已扩写实体是否都成功 CODED
> ```
> 
> 如果你坚持“项目目标是全部实体标准化成功”，那严格版应该类似：
> 
> ```
> success = (
>     len(target_records) > 0
>     and all(r["status"] == "CODED" for r in target_records)
> )
> ```
> 
> 但这里有个关键边界：你得先定义 `target_records` 是什么。
> 
> 如果目标是“句子里所有医学实体”，那现在 ABBRService 只围绕缩写 records 工作，并不等于完整 NER 标准化全覆盖。更现实的 V11 定义应该是：
> 
> > 对系统识别/候选召回到的医学缩写进行扩写，并尽量标准化。
> 
> 所以我建议面试里这样修正：
> 
> > 当前代码里的 `success` 是流程完成语义，不等于所有医学实体都标准化成功。更严谨的 API 应该增加 `all_coded` 或 `standardization_success` 字段，只有所有目标 record 都是 CODED 时才为 true。WITHHELD 应该算“安全拒绝”，不应该算“标准化成功”。
> ````

## 2.对于过度扩写的情况

**如果 `NOT_EXPANDED` 已经发生在 primary + fallback 之后，再做一次“同类型 fallback”通常收益不大。**

尤其是这个例子：

```
Patient has XYZ.
```

这里的问题不是“系统少试了一次”，而是：

```
上下文信息太少 + 候选源没有证据
```

这种情况下再次让 LLM fallback，很容易从“兜底”变成“编一个看起来像医学缩写的答案”。

**所以我建议不是 retry fallback，而是把 NOT_EXPANDED 分型。**

可以分成三类：

```
1. no_candidates
   primary 没有，fallback 也没有。
   代表系统没有任何候选证据。

2. candidates_rejected
   有候选，但 coverage 判断上下文不支持。
   代表候选存在，但当前语境不够安全。

3. ambiguous_candidates
   有多个候选都可能成立，但上下文不足以消歧。
   代表不是没答案，而是不敢选。
```

对于 `Patient has XYZ.` 这种，它属于：

```
no_candidates + low_context
```

最合理的动作不是再次 fallback，而是：

```
ABSTAIN / NOT_EXPANDED
```

并记录：

```
{
  "failure_type": "NO_CANDIDATES_LOW_CONTEXT",
  "suggestion": "Need more clinical context or abbreviation dictionary update"
}
```

**真正有意义的精进方向不是“再问一次 LLM”，而是换信息源。**

比如：

```
primary dictionary
  -> local abbreviation table

fallback LLM
  -> 生成候选，但不可信

external abbreviation source
  -> UMLS / ADAM / clinical abbreviation dataset / CASI / 自建高频缩写表

corpus evidence
  -> 看项目语料里 XYZ 是否和某些 expansion 共现

user/debug feedback
  -> 把人工确认过的 XYZ 加回候选库
```

也就是说，fallback 后还失败，下一步应该是：

> candidate source enhancement，而不是 prompt retry。

**我会这样改进当前系统：**

1. `NOT_EXPANDED` 不只写一个笼统状态，而是写清楚原因：

```
NO_CANDIDATES
CANDIDATES_REJECTED_BY_COVERAGE
AMBIGUOUS_LOW_CONTEXT
TOKEN_GATE_REJECTED
```

1. 对 `NO_CANDIDATES`，不要自动再 LLM 扩写，进入错误收集：

```
unresolved_cases.jsonl
```

后续批量分析哪些缩写值得补词典。

1. 对 `CANDIDATES_REJECTED_BY_COVERAGE`，可以允许一次“候选重排/上下文解释”，但不是生成新答案。
2. 对 `AMBIGUOUS_LOW_CONTEXT`，宁可返回 ABSTAIN，并提示需要更多上下文。
3. 只有在缩写是高频、近似单义、且 LLM 候选能被外部知识库或标准库支持时，才允许从 fallback 进入扩写。

所以你的例子最终应该是：

```
Patient has XYZ.
```

系统输出：

```
不扩写，不标准化，记录为 NO_CANDIDATES_LOW_CONTEXT。
```

这不是系统弱，而是系统稳。

面试里可以这样讲：

> fallback 失败后我不会简单重复调用 LLM，因为这通常只会增加幻觉风险。我的策略是把 NOT_EXPANDED 分型：如果是 no candidates，就进入 unresolved case 收集，用于后续词典和候选源扩充；如果是有候选但 coverage 拒绝，才考虑一次受控的候选重检索或上下文重判。这样能提高系统覆盖率，同时避免把项目调成过度扩写。

3.在第一轮withheld之后如果生成新的同义扩写词进行新的一轮检索之后标准词阶数提升被采纳那么我们并不去更换新的同义扩写词去句子中因为可能会更改句意，只是在records中新增一个参数去记录最后真正使用的检索词query是谁
