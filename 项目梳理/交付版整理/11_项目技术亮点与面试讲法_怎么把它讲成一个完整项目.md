# 11_项目技术亮点与面试讲法：怎么把它讲成一个完整项目

> 这一章不再新增底层链路细节。
>
> 它的作用是：把 02-10 章里那些零散的技术点，压缩成你在面试里能讲出口的一套完整项目故事。

---

## 0. 你现在最需要解决的不是“再懂一个模块”

你前面已经看过：

- 请求怎么进来；
- 缩写候选怎么找；
- coverage 为什么要拦；
- records 状态机怎么流转；
- expanded_text 为什么不能让 LLM 随便写；
- SNOMED / RxNorm 怎么分源检索；
- verifier 为什么要挡住向量 Top1；
- reflection 怎么把一部分 WITHHELD 救回来；
- API 最终怎么返回；
- benchmark 和 error analysis 怎么定位错误。

但是面试时不能按“模块文档目录”从 01 背到 10。

面试官也不会问：

> “请你逐章背诵一下 ABBRService、StdService、MedicalRetriever、Verifier、Benchmark 的实现。”

他真正想知道的是：

> “你这个项目解决了什么问题？  
> 为什么这样设计？  
> 你用了 LLM / RAG / 向量库，但它们到底解决了哪个具体问题？  
> 如果系统错了，你怎么知道错在哪里？”

所以这章要帮你把项目讲成一个完整故事。

---

## 1. 最不要这样讲

不要上来就说：

> “我做了一个 medical-nlp 项目，用了 FastAPI、Milvus、BGE-M3、LLM、LangGraph、Docker……”

这句话听起来工具很多，但没有项目判断。

面试官听完只会继续追问：

- 你为什么要用这些？
- LLM 具体做了什么？
- 向量库查的是什么？
- 出错时怎么定位？
- 你怎么证明它不是在乱改医学文本？

所以你要把工具藏到问题后面。

---

## 2. 一句话项目定位

你可以这样说：

> 我做的是一个面向英文临床文本的医学缩写扩写与标准概念标准化系统。它不是让 LLM 直接重写病历，而是把缩写处理拆成候选召回、coverage 闸门、确定性替换、SNOMED / RxNorm 向量检索、LLM 忠实性校验和错误分析闭环，尽量让医学文本处理可控、可解释、可评估。

这句话里面有几个关键词：

- **英文临床文本**：说明场景，不是泛泛 NLP。
- **缩写扩写**：例如 `ASA -> aspirin`，`CP -> chest pain`，`SOB -> shortness of breath`。
- **标准概念标准化**：不是只把句子变长，还要尽量映射到 SNOMED / RxNorm 这样的医学知识库。
- **不是 LLM 直接重写**：这是你的项目和普通“调用大模型改写文本”的根本区别。
- **可控、可解释、可评估**：这是面试里最有价值的工程判断。

---

## 3. 30 秒讲法

如果面试官只给你很短时间，你说这一版：

> 这个项目处理的是英文临床文本里的医学缩写，例如 `The patient took ASA for CP and denies SOB.`  
> 系统会先用候选词典和 fallback 检索找出每个缩写的可能扩写，再用 coverage 闸门判断有没有足够证据。真正替换句子时不用 LLM 自由生成，而是用确定性替换得到 `aspirin / chest pain / shortness of breath`。之后按实体类型路由到 RxNorm 或 SNOMED 向量库检索标准概念，再用 verifier 判断候选是否真的忠实。最后 API 返回扩写文本、每个缩写的映射、标准概念，以及哪些项没有被安全编码。

这一版的核心是：

- 有例子；
- 有链路；
- 有“不是 LLM 直接改写”的安全意识；
- 有最终结果。

## 4. 1 分钟讲法

如果面试官让你展开一点，你说这一版：

> 我这个项目的核心目标是把临床文本里的缩写变成可读、可检索、可追踪的医学结构化结果。  
> 以 `The patient took ASA for CP and denies SOB.` 为例，系统会把 `ASA` 识别为药物，扩写成 `aspirin`，后面走 RxNorm；把 `CP` 和 `SOB` 识别为症状或临床概念，走 SNOMED。  
> 设计上我没有让 LLM 直接生成整句，因为临床文本里否定、范围和缩写含义都很敏感。系统先做候选召回，再用 coverage 判断证据是否足够，只有通过的缩写才进入确定性替换。  
> 替换后，每个扩写项会进入多源标准化：Drug 类查 RxNorm，其它医学概念查 SNOMED。向量检索只是候选召回，不能直接相信 Top1，所以后面还有 LLM verifier 判断候选是否忠实。如果不忠实，就标记为 WITHHELD，必要时触发 reflection 生成更好的检索词重试。  
> 最后 benchmark 和 error analysis 会把错误定位到候选召回、coverage、标准化检索、verifier 或 gold 标注问题，而不是只看一个总准确率。

这一版已经能让面试官感受到：

- 你知道项目从输入到输出怎么流；
- 你知道每个模块为什么存在；
- 你不是堆工具；
- 你知道医学 NLP 的风险点。

## 5. 3 分钟讲法：完整项目故事

如果面试官让你详细介绍项目，你可以按 8 步讲。

### 第一步：问题定义

临床文本里有大量缩写。

例如：

```text
The patient took ASA for CP and denies SOB.
```

人能读懂：

- `ASA` 大概率是 aspirin；
- `CP` 是 chest pain；
- `SOB` 是 shortness of breath。

但系统不能只把它当普通文本改写，因为医学场景有几个风险：

- 同一个缩写可能有多个含义；
- 否定词不能被破坏，例如 `denies SOB`；
- 扩写对了，不代表标准编码一定对；
- 向量相似度高，不代表医学概念忠实；
- 错误必须能定位，否则后续很难调优。

所以项目目标不是“把句子改得通顺”，而是：

> 在尽量不破坏原文语义的前提下，把缩写扩写成可读文本，并尽量映射到可信医学标准概念。

### 第二步：入口 API

主入口是 FastAPI 的简单扩写接口。

用户传入：

```json
{
  "text": "The patient took ASA for CP and denies SOB."
}
```

API 不直接做复杂逻辑，而是把请求交给 ABBRService 主编排。

这层的价值是：

- 对外提供稳定接口；
- 隔离 HTTP 请求和内部处理链路；
- 最终把内部复杂状态压缩成前端能用的响应字段。

### 第三步：ABBRService 主状态机

ABBRService 是缩写扩写链路的主编排器。

它不是一次性返回一个字符串，而是为每个缩写维护一个 record。

每个 record 大概回答这些问题：

- 这个缩写原文是什么？
- 找到了哪些候选扩写？
- coverage 是否通过？
- 最终 expansion 是什么？
- 标准化有没有成功？
- 如果失败，失败在哪一步？

所以它不是“句子级黑盒”，而是“缩写级状态机”。

状态大致可以理解为：

```text
NOT_EXPANDED -> PENDING -> CODED
                         -> WITHHELD
                         -> ABSTAIN
```

含义是：

- `NOT_EXPANDED`：还没有安全扩写。
- `PENDING`：扩写已确定，等待标准化。
- `CODED`：扩写和标准编码都成功。
- `WITHHELD`：扩写可以给，但标准编码不够忠实，所以扣住 code。
- `ABSTAIN`：扩写证据都不够，干脆放弃。

这里是项目的第一个亮点：

> 扩写成功和标准编码成功是解耦的。

这就避免了一个常见问题：

> 只要标准库没查到，就把整个缩写处理判失败。

实际上，扩写文本可能是对的，只是标准概念暂时不能安全给。

### 第四步：候选召回和 coverage 闸门

缩写不能让 LLM 拍脑袋扩写。

所以系统先找候选：

- 主候选来自缩写候选词典；
- 如果主候选不足，再用 fallback 检索补候选；
- coverage 会判断这些候选是否覆盖了文本中的缩写。

coverage 的本质不是为了“提高复杂度”，而是一个安全闸门：

> 只有当系统对某个缩写有足够候选证据时，才允许它进入后续扩写。

如果 coverage 不通过，这个缩写不会被硬改。

面试里可以这样讲：

> 我把 LLM 放在受限位置，而不是让它自由解释所有缩写。候选召回和 coverage 先决定哪些缩写有资格被处理，这样能减少无依据扩写。

### 第五步：确定性替换生成 expanded_text

这一点非常重要。

`expanded_text` 不是 LLM 直接写出来的。

它是系统在确认每个缩写的 expansion 后，对原文做确定性替换得到的。

比如：

```text
The patient took ASA for CP and denies SOB.
```

变成：

```text
The patient took aspirin for chest pain and denies shortness of breath.
```

为什么不用 LLM 生成整句？

因为临床文本里一个小改动都可能改变含义。

例如：

- 把 `denies SOB` 改成 `has shortness of breath` 就是严重错误；
- 把 uncertain 改成 certain 也是错误；
- 把时间、程度、否定范围改掉也会影响医学意义。

确定性替换的好处是：

- 保留原文结构；
- 不重写无关内容；
- 每个替换都能追溯到具体缩写；
- 更容易评估和 debug。

面试里可以说：

> LLM 参与判断，但最终改写文本不是 LLM 自由生成，而是基于 records 的确定性替换。这是为了保护临床文本的局部语义。

### 第六步：多源标准化

扩写完成后，系统还要把 expansion 映射到标准医学概念。

这里有两类知识库：

- 药品类走 RxNorm；
- 非药品医学概念走 SNOMED。

例如：

- `aspirin` 是药物，路由到 RxNorm；
- `chest pain` 是临床症状，路由到 SNOMED；
- `shortness of breath` 也是临床概念，路由到 SNOMED。

向量库层面可以理解为两个集合：

- `rxnorm_concepts`：药品概念集合；
- `concepts_only_name`：SNOMED 概念集合。

它们不是同一个集合里的两个字段，而是两个不同的检索目标。

这样设计的原因是：

- 药品和临床概念的标准体系不同；
- 字段结构、概念 ID、语义范围不同；
- 混在一个集合里容易把药物和症状互相召回；
- 分源后可以按实体类型选择更合适的知识库。

面试里可以这样讲：

> 我的标准化不是一个通用向量库查到底，而是根据实体类型做 source routing。Drug 类查 RxNorm，其它临床概念查 SNOMED，避免不同医学本体混召回。

### 第七步：verifier 阻止错误编码

向量检索只能说明“文本相似”，不能保证“医学含义忠实”。

比如 `chest pain` 的向量 Top1 可能召回到一个相关但不等价的概念，例如某种评分、量表或上下文项。

如果直接相信 Top1，就可能把错误 code 返回给用户。

所以项目里有 verifier。

verifier 的任务不是生成新医学结论，而是做受限判断：

> 在这些候选标准概念中，哪一个和当前 expansion 最忠实？如果都不忠实，就返回 null。

这就是为什么会出现 `WITHHELD`。

`WITHHELD` 不是系统崩了，而是系统做了一个保守判断：

> 我可以告诉你这个缩写扩写是什么，但我不能安全给出标准 code。

这在医学场景里是合理的。

因为宁可不编码，也不能乱编码。

### 第八步：reflection 和错误分析闭环

如果 verifier 认为候选不忠实，系统可以进入 reflection。

reflection 的作用不是让 LLM 改答案，而是让它生成更好的检索词或重检索方向。

也就是说：

```text
扩写项 -> 初次检索 -> verifier 拒绝 -> reflection 生成新 query -> 重检索 -> 再验证
```

如果仍然不通过，就保持 WITHHELD。

这体现的是一个工程取舍：

> 可以努力救回，但不能无限重试，也不能为了成功率强行接受。

最后 benchmark 和 error analysis 会统计：

- 哪些缩写扩写错了；
- 哪些扩写对了但标准化没成功；
- 是候选召回问题；
- 是 coverage 问题；
- 是检索召回问题；
- 是 verifier 过严或过松；
- 还是 gold 标注本身存在问题。

这样项目就不是一次性 demo，而是有调试闭环。

## 7. 项目 6 个技术亮点

### 亮点 1：不是让 LLM 直接改写病历

普通做法可能是：

```text
Prompt: 请把这句话里的医学缩写全部扩写。
```

然后直接相信 LLM 输出。

你的项目不是这样。

你的设计是：

```text
候选召回 -> coverage -> records -> 确定性替换 -> 标准化 -> verifier
```

LLM 被限制在几个位置：

- 候选不足时辅助补候选；
- verifier 判断候选是否忠实；
- reflection 生成更好的检索词；
- error analysis 辅助分析错误类型。

它不是最终文本的自由作者。

面试说法：

> 我没有把 LLM 当成黑盒改写器，而是把它放进一个受约束的医学文本处理流水线里，让它做判断和辅助检索，而不是直接生成最终临床结论。

### 亮点 2：每个缩写都有独立生命周期

很多系统会把整句处理成一个整体：

```text
input sentence -> output sentence
```

但你的项目是：

```text
input sentence -> 多个 abbreviation records -> 每个 record 独立状态
```

这让系统可以表达更细的结果：

- `ASA` 成功扩写并编码；
- `CP` 成功扩写但标准化被扣住；
- `SOB` 可能扩写成功，也可能因为候选不足而放弃；
- 整个 API 仍然可以返回可用结果。

面试说法：

> 我把错误隔离在缩写级别，而不是让一个缩写失败拖垮整句。这样最终结果既能局部成功，也能清楚说明哪些地方没被安全处理。

### 亮点 3：确定性替换保护原文语义

医学文本最怕“看起来通顺但语义变了”。

所以你的系统不让 LLM 直接写 `expanded_text`。

它只替换已确认缩写。

这能保护：

- 否定词；
- 时间信息；
- 原句结构；
- 未处理实体；
- 非医学内容。

面试说法：

> expanded_text 是由 records 驱动的 deterministic replacement 生成的，不是 LLM paraphrase。这样可以最大限度保留原文结构，降低临床语义漂移。

### 亮点 4：SNOMED / RxNorm 分源标准化

项目里不是只有一个向量库。

更准确地说：

- SNOMED 是一个标准概念集合；
- RxNorm 是另一个药品概念集合；
- 系统根据实体类型选择对应 source。

好处是：

- 药品不会混到症状库里查；
- 症状不会被药品概念干扰；
- 后续扩展 ICD、LOINC 等也更自然；
- 每个 source 可以有自己的 schema 和检索策略。

面试说法：

> 我把标准化层设计成多源路由，而不是一个混合知识库。Drug 查 RxNorm，临床概念查 SNOMED，这样更符合医学本体边界。

### 亮点 5：verifier 把“相似”变成“可接受”

向量检索 Top1 不是最终答案。

这是非常重要的工程意识。

因为：

- embedding 相似可能只是词面相近；
- 医学概念可能上下位不一致；
- 某些候选是属性、评分、量表，不是疾病或症状本身；
- 错 code 比不给 code 更危险。

所以 verifier 负责把候选分成：

- 可以接受；
- 不可以接受。

面试说法：

> 向量检索在我的项目里只是 recall，不是 final decision。最终是否返回 code，要经过 verifier 做医学语义忠实性判断。

### 亮点 6：benchmark + error analysis 形成调试闭环

项目不是只跑一次 demo。

它有评估闭环。

benchmark 负责把系统输出和 gold 对比。

error analysis 负责解释为什么错。

错误可能来自：

- 缩写候选没有召回；
- coverage 阈值过严或过松；
- fallback 给错候选；
- 标准库检索没召回正确概念；
- verifier 拒绝了正确候选；
- verifier 接受了错误候选；
- gold 标注本身不一致。

面试说法：

> 我没有只看最终准确率，而是把错误拆到链路节点上。这样调系统时能知道应该改词典、改 coverage、改检索、改 verifier，还是修 gold。

## 8. 面试官问：“这和普通 RAG 有什么区别？”

你可以这样答：

> 普通 RAG 更多是问答场景：用户问一个问题，系统召回文档，再生成答案。  
> 我的项目里的 RAG 更像医学概念标准化里的候选 grounding。系统不是召回长文档让 LLM 编答案，而是把扩写后的医学 mention 映射到 SNOMED 或 RxNorm 的标准概念候选，再用 verifier 判断这些候选是否忠实。  
> 所以这里的重点不是生成自然语言答案，而是让缩写扩写和医学标准编码可追踪、可拒绝、可评估。

这个回答能区分：

- QA RAG；
- 标准化 RAG；
- 候选召回；
- 最终决策。

## 9. 面试官问：“为什么不用 LLM 直接扩写？”

你可以这样答：

> 因为医学文本对语义漂移非常敏感。LLM 直接扩写整句时，可能会改写否定、补充不存在的信息，或者把不确定表达变成确定表达。  
> 所以我把扩写拆成候选召回、coverage 判断和确定性替换。LLM 可以参与候选判断或 verifier，但不直接自由生成最终病历文本。

如果面试官继续追问：

> “那 LLM 还有什么价值？”

你可以补：

> 它的价值在于处理规则难覆盖的语义判断，比如候选是否忠实、失败后该换什么检索词、错误案例可能属于哪类。但这些都是受限任务，不是让它随便生成医学结论。

---

## 10. 面试官问：“为什么不用向量 Top1？”

你可以这样答：

> 向量 Top1 只能说明语义相似，不代表医学等价。  
> 比如 `chest pain` 可能召回到包含 chest pain 的评分项、量表项或上下文概念，它们和症状本身并不等价。  
> 所以我把向量检索当作召回层，后面用 verifier 判断候选是否真的忠实。如果没有忠实候选，就返回 WITHHELD，而不是强行给 code。

这里一定要强调：

> recall 和 decision 是两层。

这是很多 RAG 项目容易忽略的点。

## 11. 面试官问：“WITHHELD 是不是失败？”

不要说“是失败”。

要说：

> WITHHELD 表示扩写可以确认，但标准编码不能安全确认。  
> 在医学场景里，这是一种保守输出策略。它比返回错误 code 更安全，也方便后续 error analysis 定位是检索问题、verifier 问题还是知识库覆盖问题。

然后用例子解释：

```text
CP -> chest pain
```

如果扩写正确，但 SNOMED 候选都不忠实，那么：

- `expanded_text` 里仍然可以出现 `chest pain`；
- `mappings` 里仍然可以记录 `CP -> chest pain`；
- `standardized_entities` 里不返回错误 code；
- record 状态是 `WITHHELD`。

这就是“扩写”和“编码”解耦。

## 12. 面试官问：“你怎么评估这个系统？”

可以分三层答。

### 第一层：扩写是否正确

看系统有没有把缩写扩成 gold expansion。

例如：

- `ASA -> aspirin`
- `CP -> chest pain`
- `SOB -> shortness of breath`

### 第二层：标准化是否正确

看 expansion 是否映射到了正确的标准概念。

例如：

- 药品是否查到了正确 RxNorm；
- 症状是否查到了正确 SNOMED；
- verifier 是否接受了忠实概念。

### 第三层：错误发生在哪

不仅看对错，还要看失败节点：

- 没有候选；
- coverage 不通过；
- 替换没发生；
- 检索没召回；
- verifier 拒绝；
- reflection 仍失败；
- gold 本身可能有争议。

面试说法：

> 我的评估不是只输出一个 accuracy，而是把错误映射回 pipeline 节点。这样后续改进可以有方向，而不是盲目调 prompt。

## 13. 面试官问：“你在项目中最核心的设计取舍是什么？”

你可以答三个。

### 取舍 1：准确率 vs 安全拒绝

系统不是为了让每个缩写都有 code。

它允许：

- CODED；
- WITHHELD；
- ABSTAIN。

也就是说，系统有拒绝机制。

面试说法：

> 我宁愿让系统在不确定时不给 code，也不让它为了覆盖率返回错误医学编码。

### 取舍 2：LLM 能力 vs 工程约束

LLM 很强，但不能无限制使用。

你的项目把 LLM 放在：

- 候选辅助；
- verifier；
- reflection；
- error analysis。

面试说法：

> 我不是不用 LLM，而是把 LLM 限制在可验证的局部任务里。

### 取舍 3：模块复杂度 vs 可调试性

项目看起来链路比较长。

但每一层都对应一个明确问题：

- 候选召回解决“可能是什么”；
- coverage 解决“证据够不够”；
- 替换解决“怎么安全改文本”；
- 标准化解决“对应哪个标准概念”；
- verifier 解决“候选是否忠实”；
- benchmark 解决“错在哪”。

面试说法：

> 这个链路不是为了堆复杂度，而是为了把医学文本处理中的风险点拆开，让每一步都有输入、输出和可解释失败状态。

## 15. 项目当前局限，要主动讲清楚

成熟的项目介绍不应该只说优点。

你可以主动讲几个局限。

### 局限 1：候选词典规模有限

目前缩写候选主要依赖项目内候选词典和 fallback。

这适合 demo 和实验，但真实临床场景需要更大规模、更权威的数据源。

面试说法：

> 目前候选词典更偏实验规模，后续需要接入更完整的 UMLS、医院本地缩写表或人工审核数据。

### 局限 2：标准库覆盖依赖本地数据

SNOMED 和 RxNorm 的检索效果依赖本地导入的数据质量。

如果某些概念没有入库，系统就算设计正确也召回不到。

面试说法：

> 标准化效果不只取决于模型，还取决于知识库覆盖和数据清洗质量。

### 局限 3：`success` 不等于所有实体都 CODED

API 里的 `success` 更偏请求级成功，不代表每个缩写都完成标准编码。

真正判断每个缩写结果，要看：

- `mappings`
- `standardized_entities`
- 内部 `mapping_states`
- record 状态

面试说法：

> 请求成功和每个实体标准化成功是两层语义，不能混在一起看。

### 局限 4：简单 API 隐藏了一些内部状态

对前端来说，简单 API 更好用。

但对调试来说，它可能隐藏：

- WHY_NOT_EXPANDED；
- verifier 拒绝原因；
- reflection 是否重试；
- 每个 record 的详细状态。

后续可以提供 debug endpoint 或详细模式。

### 局限 5：同一缩写多义场景还需要增强

如果同一句话中同一个缩写在不同上下文里有不同含义，当前系统未必能完美处理。

后续需要：

- 更细粒度 mention offset；
- 上下文窗口；
- 缩写级别 disambiguation；
- 更强 gold 数据。

### 局限 6：还有遗留代码和展示清理工作

项目经历了很多轮 AI 辅助开发，里面可能存在：

- 未接入主链路的实验文件；
- 旧字段；
- 旧 prompt；
- 测试脚本残留；
- cache / log / 临时输出；
- 文档和代码命名不一致。

这就是后续 GitHub 交付前要清理的“盲肠”。

面试说法：

> 这个项目的一个现实问题是迭代速度快，遗留代码比较多。我现在的交付工作就是把主链路、实验链路和废弃代码分清楚，保证 GitHub 展示的是可运行、可解释的核心版本。
