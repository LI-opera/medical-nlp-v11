# 重构版（V10 改进）运行流程图与设计决策

> 这是基于"逐模块深挖"讨论后,把现状 V9 重构成更合理的一版。**不是现有代码,是改进设想**——面试时作为"我能批判并重构这个项目"的弹药。
> 核心五处改动:① NER 从"整句标准化"挪到"fallback 候选校验+给 label"；② coverage 选出**唯一最佳** expansion；③ 扩写 LLM 换成**确定性替换**；④ 检索用 NER 的 label 软约束 domain；⑤ verify 瘦身(句子级可省)、reflect 按 mapping 粒度增量。

## 二、五处改动逐条对照 V9（为什么改）

| # | V9 现状 | 重构版 | 为什么 |
|---|---|---|---|
| ① NER 位置 | 在 `standardize(整句)`,冗余、不喂 verify | 挪到 **fallback 候选校验**(is_medical) + **给 label** | 让 NER 承重、免整句合并问题、跨模型缓解同源盲区 |
| ② coverage 输出 | 返回 plausible **列表**,扩写 LLM 再选 | coverage 直接选出**唯一最佳** expansion | 消歧本就是 coverage 该拍板的;选唯一才能确定性替换 |
| ③ 扩写 | **LLM 造句**(引入丢否定/幻觉风险) | **确定性 token 边界替换** | 既然已选定,替换更简单/便宜/安全;从源头消除否定保持问题 |
| ④ 检索 domain | `domain_filter` 一直传 None(死参数) | 用 NER 的 label **软加分**约束 domain | 激活死参数 + 提检索精度;软加分避免映射错时弄丢答案 |
| ⑤ verify/reflect | 句子级+映射级双查;整句重写;all 偏严 | 句子级**可省**;reflect 按 **mapping 粒度**换候选;**增量**重算 | 替换不会改坏句子→句子级没必要;只修出错项→省成本、不误伤对的 |

## 三、这版用到的设计心法（面试金句）

1. **确定性优先,贯彻到底**:能用规则/替换确定地做的(扩写),就别用 LLM——LLM 生成能力越强越易引入副作用(改意/幻觉),然后还得加校验去补救。V9 在扩写这步没贯彻,这版补上。
2. **让每个模块的输出被下游当必要输入**:NER 的 label → 驱动 domain 检索;不再"算了存档"。模块"有用"的标准是产出被消费。
3. **召回路要输出一致字段**:词典候选(元数据 label)和 fallback 候选(NER label)都带 label,避免"出身决定检索待遇"。
4. **输入侧筛选 ≠ 输出侧终判**:NER 杀候选(输入筛选)和早停判空(输出终判)是两层,都保留。
5. **增量计算 / 缓存失效**:reflect 只改一个缩写 → 只对变化的 expansion 重跑 NER+检索,没变的复用。
6. **跨模型校验缓解同源盲区**:NER(本地模型)≠ 生成/coverage(deepseek),引入不同源的第二意见。

## 三·补 · 为什么 mapping 粒度重试能成立（失败隔离原理）

> 这是第⑤条改动的底层原理。一句话:**把"状态"分阶段锁 + 把工作单位从"句子"降到"单个 mapping" + 用确定性替换做物理隔离,三者合起来,才能做到"一个缩写失败只重修它自己、锁定其它全部"。**

### 1. 状态要分阶段锁,不是笼统一个"成功/失败"

每个缩写不该只有一个 `is_valid`,而该带一条**自己的小状态链**:

```text
缩写 CP ：选扩写✅ → 已替换✅ → SNOMED检索✅ → 校验✅  → status=LOCKED_OK  冻结，不再动
缩写 MS ：选扩写✅ → 已替换✅ → SNOMED检索✅ → 校验❌  → status=VERIFY_FAIL 只重修这一个
缩写 XYZ：选扩写✅ → 已替换✅ → SNOMED检索❌ → —      → status=RETRIEVE_FAIL 只重修这一个
```

注意 MS 和 XYZ 是**两种不同的失败**:MS 是"扩写选错了"(要换候选),XYZ 是"扩写可能没错、但库里没这个概念"(未必换候选)。分阶段锁,才能分别诊断、分别处理——比 V9 一个笼统 `is_valid=false` 精细得多。

### 2. V9 的病根:工作单位是"整句"

```text
reflect：LLM 拿着【整个 expanded_text 字符串】重写一遍
   → 哪怕只有 MS 错，重新生成整句时可能把对的 CP→chest pain 也改了措辞、甚至改坏否定
overall_valid = all(...)：一个错 → 整轮 fail → 整句再来
```

所以"一群词因为一个词重写而都要重写"**不是偶然,是架构选择的必然**:让 LLM 重写整句,就没法保证它只动该动的那个。

### 3. 关键前提:确定性替换是隔离的"使能条件"★

这是最该看见的一点——第③条(确定性替换)和第⑤条(mapping 粒度)是**绑死的**:

```text
若仍用 LLM 重写整句  → 做不到隔离：LLM 一开口就是整句，无法命令它"只改 MS、别碰 CP"
改成确定性替换       → 天然隔离：改 MS 那个 token 位置，物理上不可能影响 CP 那个位置
```

**你想要的"避免连累",前提是先把扩写从"LLM 造句"换成"确定性替换"。** 没有这一步,mapping 粒度隔离是空中楼阁。两者是同一个决策的两面。

### 4. 跑例对比

输入 `"Patient has CP and MS"`,CP 对、MS 错:

```text
【V9 整句重来】
 轮1: "...chest pain...multiple sclerosis" → MS错 → all()→整轮fail
 轮2: LLM 重写整句 → 有概率连 chest pain 也动；即使没动也白白重检索了 CP
      成本：2次整句检索 + 2次verify + 1次整句LLM重写

【重构 mapping 粒度】
 轮1: CP→chest pain ✅LOCKED；MS→multiple sclerosis ❌FAIL
 轮2: 只 reflect MS→mitral stenosis；只重检索+重校验 MS；CP 原封不动复用缓存
      成本：1次单项检索 + 1次单项verify     ← CP 完全没被惊动
      组装：原句上 CP位置、MS位置各自替换 → 完成
```

### 5. 实现要点 / 局限

- **替换要按位置偏移小心**:多个替换同时应用时,前面的替换会让后面字符位置偏移——用原始 token 索引、**从后往前替**,避免 offset 错位。
- **极少数 expansion 之间有语法交互**:纯隔离会漏,保留一个轻量整句保意终检兜底即可。
- **失败模式细分了,但判定仍是 LLM**:"扩写错"vs"SNOMED 缺概念"的区分还靠 verify,同源盲区仍在,靠跨模型/规则补。
- **净收益要 benchmark 量化**:粒度化省的是成本 + "误伤对的"概率,但对最终 accuracy 的净提升得用第17篇 benchmark 量出来。

> 面试金句:**"V9 的整句重写让一个词的失败连累整句;我把工作单位降到单个 mapping、给每个 mapping 分阶段锁状态,失败只隔离重修它自己。而这套隔离能成立的前提,是先把扩写换成确定性替换——因为只有局部编辑才物理上互不干扰,LLM 整句重写做不到。这俩改动是绑在一起的。"**

## 四、重构版自己的诚实局限（别只吹优点）

- **NER 判孤立短语不稳**:`is_medical` 对孤立 expansion 可能误判 → 不硬中断,降为低置信信号送 error analysis 复核。
- **label→domain 映射不完美**:所以用**软加分**而非硬过滤,映射错顶多排序略偏、不弄丢答案。
- **coverage 选唯一 = 把消歧压力全给 coverage**:它仍是 LLM、仍会 over-abstention;靠 verify(映射级)+reflect 兜。
- **确定性替换处理不了"需语法改写"的罕见情况**:绝大多数医学缩写扩成名词短语能干净替入,极少数别扭的可回退给 LLM 改写。
- **改动大、要重测 benchmark**:任何重构都得用第 17 篇的 benchmark 量化"net 是否变好",不能拍脑袋上——这正是 V10 实验(第 19 篇)的教训。

## 五、面试怎么用这张图（架构级回答）

> "我把这个项目重构了一版。最大的改动是:扩写本质是'把缩写换成选定的全称',V9 用 LLM 造句反而引入了丢否定和幻觉的风险——而这些正是它后面 verify 要费劲补救的。所以我让 coverage 直接选出唯一最佳扩写,然后用确定性 token 边界替换,从源头消除了否定保持问题,verify 也能瘦身成只查'选得对不对'。NER 我也重新安置——V9 里它在整句标准化、算了不用;我把它挪到 fallback 后做候选校验、并产出类型 label 去约束 SNOMED 检索的 domain,顺手激活了那个一直传 None 的死参数,而且 NER 是本地模型、和生成用的 LLM 不同源,等于引入跨模型的第二意见。reflect 也从整句重写改成按 mapping 粒度只修出错项、增量重算。这版的指导思想就是'确定性优先、让每个模块的输出真正被下游消费'。当然这是设想,真要上线必须用 benchmark 量化净收益——这正是项目里 V10 实验失败教我的:加模块要看对整体指标的净影响。"
