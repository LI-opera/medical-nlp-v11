# Medical NLP 项目地图 + 运行流程图 · V11

> 用途：面试 RAG / Agent 工程师前的项目自查(V11 版,反映已合入的批次 1/2/3-rev/4/5/6/7/8/9/10 + SNOMED 全量库升级 + verify 窗口放宽 + **标准化反思闭环(10)+ 数据流统一(11A)+ 错误分析系统(11B-14)+ 多源路由 + LangGraph 可视化 + 生产有界迭代反思(L3 Stage1-6e,RxNorm)**)。读完应能在白板上画出整条链路,并讲清每个设计决策的 why + trade-off。
>
> **状态名口径(11A 起统一)**:每个缩写从召回到出口是同一条 record,生命周期 `NOT_EXPANDED → PENDING → CODED / WITHHELD / ABSTAIN`,并带 `failure` 字段。下文第 3、6 节凡见旧名 `LOCKED_OK` 按 `CODED+WITHHELD` 读、`LOCKED_ABSTAIN` 按 `ABSTAIN` 读;反思 + 错误分析系统见**第 8 节**。
>
> 一句话定位：**把"医学缩写扩写 + SNOMED 术语标准化"做成一条"确定性替换 + mapping 粒度失败隔离 + RAG 检索 + LLM 标准化卡点"的可评估后端服务。**
>
> 与 V9 版的关系:V9 用 LLM 造句扩写 + 整句重写重试 + LangGraph 工作流;V11 重构成 **coverage 选唯一 → 确定性 token 替换 → 每个 mapping 走自己的状态机、失败只隔离重修它自己**,并在批次 7 **彻底删掉 V9 遗产**(旧方法 / LangGraph 目录 / 反思服务 / MappingSupportVerifier)。下面凡标 **【V11 改】** 的就是相对 V9 的变化。

## 0. 30 秒电梯陈述(先背这段)

这是一个医学文本标准化后端。用户发一段临床文本(含 SOB、CP 这类缩写),系统先做**候选召回 + coverage 裁决**给每个缩写选出**唯一最佳展开**;然后用**确定性 token 边界替换**把缩写换成全称(不再用 LLM 造句,从源头消除丢否定/幻觉)。工作单位降到**单个 mapping**:每个 mapping 走自己的状态链(检索 SNOMED → **verify 选忠实概念** → 锁定),**一个词失败只隔离重修它自己**(从候选池换下一个,候选用尽就弃权),对的直接冻结复用。检索用 **bge-m3 + Milvus + 规则重排 + domain 软加分**。**verify 不再二次校验扩写(那是 coverage 的活,冗余),而是当 SNOMED 标准化卡点**:在检索回来的概念里挑忠实的标准编码,挑不到就弃码(宁可不给码也不给错码)。出口除扩写文本,还返回每个成功项的 **SNOMED 标准编码(standardized_entities)**。底层 SNOMED 库已从 5000 稀疏样本升级到**全量临床标准概念**(OHDSI Athena),标准化平台真正点亮。**【L3 多源路由】**再加第二个知识源 **RxNorm(药品库)** + 按 NER domain 的**确定性路由**(疾病→SNOMED、药品→RxNorm),把"单库标准化"升级成**多源 agentic RAG**(系统自己决定该去哪个库检索);实测成分级药品标准化 SNOMED 0/5 弃码 → RxNorm 5/5 命中。整套用 **FastAPI** 暴露,有 **74 例(含 CASI 真实数据)benchmark + SNOMED concept_id 判等**的离线评估。

面试四个关键词:**确定性优先(能用替换就不用 LLM 生成)、失败隔离(mapping 粒度状态机)、verify=标准化卡点(放对位置才不冗余)、可评估 + 诊断驱动(benchmark + 真实数据 + 用诊断定位瓶颈在数据还是代码)、多源路由(agentic RAG:按 domain 选 SNOMED/RxNorm)**。

## 3. 模块地图(按调用层级 · 标注 V11 改动)

### ⭐ ABBRService —— 主编排器
`services/abbr_service.py`

- **做什么**:`__init__` 实例化复用子服务(LLM、标准化器、检索器、verifier、候选三件套、**【V11】`self.ner_service` 复用 standardizer 的 NER**);对外只暴露主链路 `expand_verify_with_retry()`。
- **【V11批次7】文件已瘦身到只剩 5 个方法**:`__init__` / `_build_expanded_text_deterministic` / `expand_verify_with_retry` / `_get_abbreviation_candidates` / `_should_consider_abbreviation`。V9 的 `simple_llm_expansion` / `expand_abbreviations` / `expand_and_standardize` / `expand_standardize_and_verify` / `_rebuild_expanded_text` / `_filter_mappings_by_context_support` **全部删除**。
- **核心方法**:
  - `expand_verify_with_retry()` —— ① 召回(`_get_abbreviation_candidates`)给每个缩写选唯一 `best_expansion` → ② `_build_expanded_text_deterministic` 确定性替换 → ③ **per-mapping 状态机循环**(每个 mapping `PENDING→检索 SNOMED→verify 选忠实概念→LOCKED_OK/LOCKED_ABSTAIN`,失败隔离 + 增量重检索) → ④ 出口汇总 LOCKED_OK + 各自 SNOMED 编码。**【V11批次7】循环里 V9 遗留的整句 `standardize` 盲肠已砍**(输出无人消费、零判分)。
  - `_build_expanded_text_deterministic()` —— **【V11 新】** 用 `\bABBR\b` token 边界正则、从后往前替换,不误伤子串、否定天然保住。
  - `_get_abbreviation_candidates()` —— 召回入口:① coverage 返回 `best_expansion`(唯一);② **fallback 弃权门**(batch3-rev):非词典缩写 coverage 不自信(coverage_ok=false 或 confidence<0.8)→ best=None,不背书;③ **fallback 后跑 NER 产 domain**(batch4,只产 domain 不过滤);④ 填 `chosen_domain`。
  - `_should_consider_abbreviation()` —— gate(已知缩写/全大写≤8字符放行)。
- **面试重点**:确定性替换是"mapping 粒度隔离"的物理前提(只有局部替换才能改 MS 不动 CP);状态机三终态(LOCKED_OK 冻结复用 / LOCKED_ABSTAIN 安全失败 / 未进 mappings);增量重检索(只对 expansion 变过的 mapping 重检索)。

### ABBRCandidateCoverageEvaluator —— 候选覆盖度 + 选唯一
`services/abbr_candidate_coverage_evaluator.py`

- **【V11 改】**:除 `plausible_candidates`,新增 **`best_expansion`**——直接从候选里选出**唯一最契合上下文**的那个(coverage_ok=false 则 null)。**消歧(扩写)的拍板权收到 coverage 这一处**——这也是后面 verify 不再碰扩写的原因(已有人管了)。

### NERService —— 医学实体识别
`services/ner_service.py`

- **【V11 改】**:重新加 `is_medical(text) → (bool, label, score)`。**本版只取 label 推断 domain,不做候选过滤、不改 fallback 数量**。NER 是本地模型、与 DeepSeek 不同源 → 跨模型第二意见。
- 注:V10/batch3 曾用 NER 做 is_medical 过滤 + fallback top-k,因制造"像医学的假候选"net 下降被回退;V11 只保留它无害的"产 domain"职责。

### ⭐ MedicalRetriever —— RAG Retriever 层
`services/medical_retriever.py`

- **【V11 改】**:`retrieve` / `_rerank_results` 新增 **`domain_boost`**——domain_id 命中目标域 +0.2 软加分(激活那个一直传 None 的死参数 `domain_filter`)。**软加分非硬过滤**:映射错顶多排序略偏,不弄丢答案。
- **【V11批次6】过滤口径修正**:score 过滤卡 `max(raw, rerank)` 而非只卡 raw——重排把好概念抬上来后,不该再被原始相似度阈值否决。
- 重排规则:=query +0.5 / 开头 +0.3 / 包含 +0.15 / 过长扣分;**【V11】+ domain 命中 +0.2**。

### ⭐ ABBVerifier —— SNOMED 标准化卡点(LLM-as-judge)【V11批次8 彻底改造】
`services/abbr_verifier.py`

- **【V11批次8 核心重构】verify 的职责从"二次校验扩写"换成"标准化卡点"**。
  - **为什么改**:批次 7 打点量化发现旧 verify 的 FAIL/SWAP/ABSTAIN = **0/0/0**——它是个**橡皮图章**,74 例里从不否决。根因:扩写已由 coverage 拍板,verify 再用**同源 LLM** 复查同一件事 = 冗余且循环自证。
  - **改成什么**:coverage 之后唯一的**新数据**是 SNOMED 检索结果,所以 verify 唯一不冗余的位置是**判标准化质量**。新 `verify_mappings` 给每个候选标 0 起的 `index`,让 LLM 返回每个 mapping 的 **`chosen_index`(选哪个 SNOMED 概念忠实) + `standardization_faithful`**;都不忠实就 `chosen_index=None` → 弃码。状态机据此取 `std_concept`,**verify 不再改扩写**。
  - **抓的就是"扩对但标准化错"**:别把"chest pain"标成"Chest pain rating(量表)"、别把"hypertension"标成"Hypertension monitoring(监测服务)"。
- **【V11 verify 窗口】top-3 → top-10**:状态机原来只喂检索 top-3 给 verify;因规则重排会被字面相似度骗(`Hypertension service` 字面比 `Hypertensive disorder` 更像 query、被顶到前面),忠实概念可能排在第 7~9 位、落在窗外。改成把检索回来的**全部 top-10** 喂给 verify。**设计点**:检索撒大网、verify 收网挑忠实——**LLM 判官比字面重排更懂"忠实",把选择权交给它,而不是去调重排权重**。

### 候选召回 / 标准化 / StdService
- `ABBRCandidateRetriever`:词典查找,**【V11】返回 `{abbreviation, expansion, domain}`**(词典加了 domain 元数据)。
- `ABBRCandidateFallbackRetriever`:本地空→LLM 生成候选(只产候选不改句),数量不变(未加 top-k)。
- `MedicalStandardizer` / `StdService`:NER + bge-m3 + Milvus 检索。
- **【V11批次7】`ABBRReflectionService` 已删**:状态机里的"反思"已被确定性的"从候选池取下一个未试候选"取代(不再用 LLM 整句重写),所以反思服务连同 LangGraph 一起退出代码库。

### FastAPI App —— 服务入口
`api/main.py` + `api/schemas.py`

- **【V11 改】**:`POST /expand/simple` 响应除 `success/expanded_text/mappings`,新增 **`standardized_entities`**——每个 LOCKED_OK mapping 由 verify 选中的 SNOMED `chosen_concept`(concept_id/concept_code/concept_name/domain_id);verify 弃码的项不进。兑现"标准化平台"。

### 离线工具 tools/ 【V11 新】
`tools/build_concept_csv.py` + `tools/rebuild_milvus.py`

- `build_concept_csv.py`:把 OHDSI Athena 下的 CONCEPT.csv(130万+,Tab 分隔,OMOP 格式)按 `vocabulary_id=SNOMED & standard_concept=S & invalid_reason 空 & 临床域` 筛成 `snomed_clinical.csv`。**`MAX_ROWS=None`(全量)**——之前设 12 万时 `head()` 按行序随机切,把"高血压/冠心病"这些**通用母概念**切没了。
- `rebuild_milvus.py`:批量 `embed_documents` 重建 Milvus(同 collection `concepts_only_name`、同 schema),比逐条的 `create_milvus_db.py` 快几十倍,代码侧一行不改。

### Benchmark + Error Analysis —— 评估闭环
`evaluation/`

- cases 50→**74**:加 24 例 **CASI 真实数据**(`abbr_benchmark_cases_casi.py`,18 例多义 fallback 消歧 + 6 例 fallback-该扩 探测器)。
- `concept_match.py`:**SNOMED concept_id 语义判等 + accept 接受集**(不死卡字符串,"primary care provider≡physician"算对)。
- `run_benchmark_parallel.py`:**线程池 + 每线程独立 service** 并行。注:全量库后每线程各载一份 bge-m3 + Milvus 吃内存大,4 worker 可能 OOM → 跑串行 `run_benchmark.py` 或 `BENCH_WORKERS=2/1`。
- **重要口径**:benchmark 判的是 **(缩写, 扩写)**,**不判最终 concept**——所以它**测不到 verify 标准化层的好坏**;concept 质量只能靠诊断脚本 + 抽查接口看。

## 4. 两条检索链路(RAG 面试核心 · V11)

### 4.1 缩写候选召回 + 确定性扩写链 【V11 重写】

```text
文本 → 切 token → gate(_should_consider_abbreviation)
     → 路1 词典召回(自带 domain 元数据)  primary
     → 查不到 → 路2 LLM fallback 生成候选(数量不变)
                  → NER 产 domain(只挂元数据,不过滤)
     → CoverageEvaluator(LLM) 选【唯一最佳】best_expansion + plausible
     → 【弃权门】fallback 缩写 coverage 不自信(<0.8) → best=None(不背书)
     → chosen = 仅含选出的: [{abbr, expansion, label, domain}]
     → _build_expanded_text_deterministic: \bABBR\b token 边界替换(否定天然保住)
```

### 4.2 SNOMED 检索链(向量 RAG + 规则重排 + domain 软加分 + verify 选忠实)【V11】

```text
每个 mapping 的被选中 expansion
     → MedicalRetriever.retrieve(query=expansion, domain_boost=mapping.domain, top_k=10, source=_route_source(domain) ★L3)
     → StdService → bge-m3 embed → Milvus TopK(全量临床 SNOMED 库)
     → _rerank_results(规则重排 + domain 命中 +0.2) → max(raw,rerank) 过滤
     → 【V11】全部 top-10 候选喂 verify(不再只 top-3)
     → verify: chosen_index=忠实概念 / None=都不忠实就弃码
     → LOCKED_OK 取 chosen_concept(SNOMED code) 或 LOCKED_ABSTAIN
```

## 7. V11 改进弧线(面试最值钱的故事)

V11 不是一次写对的,是**带 benchmark 卡关、失败就回退、用诊断定位瓶颈**走出来的——这条弧线比"功能堆好了"更能体现工程判断:

1. **批次 1**:coverage 选唯一 + 确定性 token 替换。消除 LLM 造句的丢否定/幻觉。net 0.92→0.94,消歧用例顺手改善。
2. **批次 2**:per-mapping 状态机 + 失败隔离 + 增量重检索。**分数持平,价值在架构**——它使能后面的 domain 约束和按项重修。
3. **批次 3(第一版)失败 → 回退**:本想 fallback top-k + NER 过滤治过度扩写,结果 top-k 让 LLM 造出更多"像医学的假候选",net 0.94→0.90,**立即回退**。
4. **识破过拟合 + 补强评测**:意识到"为弱 benchmark 砍 fallback"是过拟合(现有用例就靠 fallback);引入 **CASI 真实数据**(明大 35 万病历缩写库)把评测做扎实,并用 **SNOMED concept_id + 接受集**判语义等价。
5. **批次 3 重构版**:改成"低置信 fallback 才弃权"(安全失败,不砍 fallback)。net +1,low_context 2/5→3/5,过度弃权护栏(casi/fallback)没掉。
6. **批次 4**:NER 只产 domain(不重蹈 batch3) + domain_boost 软加分,**激活死参数**;benchmark 持平(domain_boost 改的是"配哪个概念",不改"选哪个扩写")。
7. **批次 5**:出口加 standardized_entities(SNOMED 编码),兑现标准化平台。
8. **批次 6**:修检索过滤口径(卡 `max(raw,rerank)` 而非只 raw,重排抬上来的好概念不被原始阈值否决)。
9. **批次 7 架构收敛 + 量化 verify**:彻底删 V9 死代码(6 个旧方法、LangGraph 目录、反思服务、MappingSupportVerifier、9 个 stale 测试、主循环整句 standardize 盲肠)。**临时打点量化 verify 贡献 = FAIL/SWAP/ABSTAIN 0/0/0**——证明旧 verify 是橡皮图章。**这一步把"感觉 verify 没用"变成"测出来 verify 没用",是最关键的工程时刻。**
10. **批次 8 verify 重定位**:既然复查扩写冗余(coverage 已拍板、同源 LLM 循环自证),就把 verify 挪到唯一有新数据的地方——**SNOMED 标准化卡点**:在检索候选里选忠实概念、选不到就弃码。verify 从 0 → 86 次弃码(真干活)。但 standardized_entities 大量空。
11. **诊断驱动定位瓶颈(数据 vs 代码)**:写 `diagnose_withholding.py` 对 8 个常见缩写三路诊断(库没有 / 检索没捞上 / verify 过严)。结论:**8 个全是库限制**——5000 样本只有叶子/特化概念,缺通用母概念。**证明瓶颈在数据不在代码**——这是别瞎调代码、先定位的样板。
12. **全量 SNOMED 库升级**:从 OHDSI Athena 下全量 CONCEPT.csv,筛成临床标准库重建 Milvus,**代码一行不改**,标准化平台点亮(CP→'Chest pain'/0.99)。验证了诊断。
13. **verify 窗口坑 + 放宽**:全量库后 HTN 仍弃码——但这次不是库,是忠实概念('Hypertensive disorder')被检索到了却排第 9、落在 verify 的 top-3 窗外(噪声字面更像 query 被重排顶前)。**窗口 top-3→top-10**:检索撒网、verify 收网。**诚实取舍**:放宽让 CAD 翻成弃码(它看到更多具体兄弟后,判唯一的上位父概念太宽、库里无精确节点),但这是可辩护的保守(无精确匹配就 abstain)。8 探针 7/8 给码,benchmark 持平 0.9595。
14. **批次 9 verify 判定标准收敛 + 建 concept 尺子**:给 verify prompt 加忠实度优先级(精确同义 > 与扩写一致的具体亚型 > 无精确时的忠实父概念 > 弃码;多个忠实选最具体、不加扩写没说的限定词)。治 CAD 过度弃码(FAIL→PASS)。为量这一层先建 **concept benchmark**(标准化层尺子,11 例,用概念名判等,PASS=忠实/canonical=最规范两指标)——因为主 benchmark 测不到 concept 层。结果:concept PASS 10/11→11/11、主 benchmark 持平。
15. **批次 10 标准化反思闭环**:verify 选到"非精确同名"概念时,LLM 提同义检索词→换词重检索→重选(**带新证据,非同源复判**)。SOB 'Difficulty breathing'→'Dyspnea' 升 canonical。**项目唯一带 LLM 在环的反思,且有 concept bench 量真实收益**——区别于 V9 被实测为橡皮图章的整句反思。Codex 自加保守门(重选概念名须==某个改写词才采纳,防漂移)。
16. **批次 11A 状态机数据流统一(纯行为中性重构)**:把"召回阶段 candidate_infos / 状态机阶段 states 两种形状"统一成一条 record(status 生命周期 + failure)。整段替换一个方法、逐字段对齐旧出口;**主 benchmark 逐例一致 71/74**。为错误遥测打地基。
17. **批次 11B→14 错误分析系统**:全收→确定性打标→读时筛→LLM 提假设→数据自检→LLM 写摘要(详见**第 8 节**)。从"报告全是噪声"做到"噪声确定性隐身、真信号留存、抗误用内置"。

**一句话纪律**:加任何东西都看对**整体指标的净影响**,不是看它解决的单点;净收益不为正就回退;**分不清是数据还是代码的问题,就写诊断先定位,别瞎调**。这是项目里 V10(MappingSupportVerifier 过度弃权回退)、batch3 第一版(过拟合)、verify 量化(测出橡皮图章)反复教的。

> 诚实补一句:benchmark 只判扩写、74 例偏小、且测不到 concept 层;CAD 因全量库仍缺精确节点而保守弃码;LLM temp=0 仍有 ±1~2 例噪声地板——所以判净收益要看"具体哪例翻了 + 每类",不被单例涨跌带偏。

## 8. V11 后续:标准化反思 + 数据流统一 + 错误分析系统(批次 9-14)

> 第 3-7 节是"扩写 + 标准化"主链路。本节是在主链路之上长出来的三件事:**把标准化做到最规范(反思)、给标准化层一把尺子(concept benchmark)、把系统跑出来的错误攒成可分析的资产(错误分析系统)**。

### 8.1 标准化反思精炼(批次 10)
`abbr_service._reflect_refine_standardization` + `abbr_verifier.propose_requeries`

- **触发**:verify 选中的概念**不是扩写的精确同名**(或弃码)时——说明可能有更规范的概念没被原 query 检索到。
- **机制(真闭环,带新证据)**:反思 LLM 只提**同义/规范检索词**(不产概念)→ 用新词重检索 → 并入候选池 → verify 在更全的池里重选。
- **实测**:SOB "shortness of breath" 原 query 只够到 'Difficulty breathing';反思提 "dyspnea" 重检索 → 'Dyspnea' 进池 → verify 选它 = canonical。
- **为什么不是橡皮图章**:它靠"换词重检索带回**不同**的概念"改变证据,而不是同一个 LLM 对同一份证据复判(那个我们在批次 7 实测是 0 贡献)。**项目里唯一带 LLM 在环的反思,且有 concept bench 量得出真收益。**
- **保守门**:重选概念名须 == 某个反思改写词才采纳,防止候选池变大后漂移到机理性更窄概念(CAD 没被推到 'Coronary arteriosclerosis')。

### 8.2 concept benchmark —— 标准化层的尺子(批次 9 建)
`evaluation/concept_benchmark_cases.py` + `run_concept_benchmark.py`

- **为什么要**:主 benchmark 判 `(缩写, 扩写)`,**测不到 verify 自批次 8 起的全部标准化工作**(选忠实概念 / 弃码)。没尺子就只能嘴上说 verify 有用。
- **怎么测**:给定**正确扩写**(绕过 coverage,只测标准化)→ 检索 top-10 → verify 选概念 → 判选中概念名是否忠实。两个指标:**PASS(忠实命中)** 与 **canonical(选到最规范)**;两者差 = 反思能改善的余量。
- **口径诚实**:用**概念名**判等不用 concept_id(id 取决于具体 Athena 下载);11 例 confirmed 硬 gold(探针/诊断/首跑核过)。
- **分层验收**:此后任何标准化改动,**主 bench 持平=没外溢到扩写、concept bench 涨=标准化真改好**。当前 PASS 19/19、canonical 17/19(L3 后扩到 19 例;非 canonical = CAD + HBP,均为忠实但非最规范的库覆盖灰区)。

### 8.3 错误分析系统(批次 11A → 14)
`services/error_collector.py` · `evaluation/analyze_errors.py` · `evaluation/error_triage.py`

把系统**自己跑出来的错误**攒成可分析、可驱动改进、且**抗误用**的资产。五步:

```text
推理/benchmark 跑
  → 【11A】每个缩写一条 record,搞不定的把原因写进 record.failure(留痕的地基)
  → 【11B】error_collector 落盘到 unresolved_cases.jsonl(一个库收两类错误):
        · 运行时 known-unknowns(系统知道自己放弃:没扩/弃码/弃权/coverage失败)
        · 评测 unknown-unknowns(GOLD_MISMATCH:系统以为对、对答案才知错)
  → 【13】每条打 expected 标签:全收不丢、确定性打标、读时筛
        · expected=True  符合 gold(如 gold 不要求扩的正确弃权)→ 默认隐藏
        · expected=False 真错(GOLD_MISMATCH)         → 保留
        · expected=None  无标准答案(真实流量/概念层)  → 永不筛、全留
        （打标用 gold 确定性、不用 LLM 猜;有答案才标、没答案不筛 → "误判拍死"两头不发生)
  → 【12+14】error_triage 三段式:
        ① LLM 提假设(判断)+ 吐 checkable_claims
        ② 数据自检(无 LLM):lib_concept 查 Milvus、dict_abbr 查词典;被打脸的标 ⚠ 且不产草稿
        ③ LLM 写执行摘要(写作,非判断):拿已被数据判过的结论,给非专业读者写
           ✅值得跟进 / ⚠已排除假信号 / ◻需人工 三栏
```

**核心设计原则(面试高价值)**:

- **不在写入端丢数据**:错误源全收;过滤是可逆的读时决策(`--all` 看全量)。怕的是硬删误判把真错也埋了。
- **打标用答案、不用 LLM 猜**:能确定性算的(gold 是否要求扩它)就别问 LLM——又一次"能算就别问 LLM"。
- **LLM 提、数据裁、LLM 写**:triage 把"判真假"交给数据自检、把"写人话"交给 LLM;两次 LLM 一次判断一次写作,第二次喂的是确定性结论,不是循环复判。
- **抗误用三重**:① 数据自检给假设打脸;② 摘要把被打脸的明确归"已排除";③ 被打脸的假设不产候选草稿。**怀疑内置进工具,不依赖读者够懂**——实测把"往 SNOMED 加 heart rate"这种骗了 LLM 三轮的假信号拦住了。
- **known-unknowns + unknown-unknowns 互补**:运行时遥测收"系统知道的坑",gold benchmark 收"系统不知道的坑(自信答错)",两套同库同表。

**诚实边界**:triage 价值随**真实流量**增长,当前 22 条(且一半是 benchmark 合成负例)上结论偏薄;`gold_labeling/verify_rubric` 类假设无 checkable_claim,只能标"需人工",自检管不到。

### 8.4 这几批的"什么不该优化"(同样是工程判断力)
- **低上下文过度扩写(ABC/QRS/LMN,coverage_003/005/006)**:**故意不修**。低上下文无消歧信号,唯一修法(调严弃权门)会过拟合这 3 例 + 引发实测更糟的过度弃权(V10/batch3 教训)。定性为"可辩护局限:低上下文倾向于扩,过度扩写下游可纠、过度弃权丢信息"。
- **verify 对 PA/PCP/ABC 弃码**:**不是 bug**,它们本就不是可编码的"病",verify 弃码正确;triage 把它误标"库缺概念"是 LLM 噪声,靠数据自检 + 人裁决挡掉。

## 9. 多 LLM / 异质判官 + "自生自查"全审(批次 15-16 + 一次诚实复盘)

> 这节是面试最值钱的一段——**不是"我加了多模型/multi-agent",而是"我精确定位了哪里才是真循环、哪里换模型没用、为什么"**。

### 9.1 LLM 配置 + 工厂(批次 15)
`utils/llm_config.py` + `utils/llm_factory.py`(仿 embedding 那套)

- `LLMProvider` 枚举(deepseek/qwen)+ `LLMConfig` 数据类 + `DEEPSEEK_CONFIG`/`QWEN_CONFIG`;`create_llm(config)` 按配置建模型。
- 千问走 **DashScope 的 OpenAI 兼容端点**(`ChatOpenAI` 指过去),接口与 `ChatDeepSeek` 一致(都 `.invoke().content`)——所以换模型**逻辑零改**,只换 config。

### 9.2 verify 异质化(批次 16)+ 诚实复盘
- 做法:`ABBVerifier(__init__)` 走 `create_llm(QWEN_CONFIG)`——coverage 仍 DeepSeek、verify 用千问。实测 **concept PASS 11/11、canonical 10/11、主 bench 0.9595,完全持平**,无异常弃码。
- **★诚实复盘(关键)**:为什么持平?因为**post-batch8 的 verify 判的是 bge-m3 检索回来的概念(外部数据),不是 DeepSeek 自己生成的东西**——它**不是自生自查**。所以"DeepSeek vs 千问"根本没循环可破,换谁都一样。
- 所以 verify 的真正安全垫是"**只能在真实检索结果里挑、扎根数据**",跟用哪个模型关系不大。**保留千问的口径应是"扎根检索的质量门 + 跨模型鲁棒性",不是"破自生自查"**(那个说法用在 verify 上是错的)。附带小收益:CAD 边界两个独立模型都选父概念、都拒 stenosis/rupture/device → **跨模型独立一致,增强了"CAD 这么判对"的置信**。

### 9.3 "自生自查"全审(定义钉死后逐对审)
**自生自查 = 同一个模型,既【生成 X】又回头【检查这同一个 X】**;不是"凡 DeepSeek 节点都算"。逐个生成-判断对:

| 生成-判断对 | 是不是自生自查 | 现状 / 最优解 |
| --- | --- | --- |
| 旧 verify 复查扩写 | **是**(同模型查自家扩写) | 已删(batch7/8) |
| fallback 造候选 → coverage 判 | **是**(DeepSeek 造、DeepSeek 判) | **弃权门**缓解(详见下);异质不划算 |
| verify 选概念 | **否**(判 bge-m3 检索,扎根数据) | 扎根检索即可;模型次要(故持平) |
| 词典候选 → coverage 判 | 否(候选人工策展) | 无需动 |
| triage 假设 → 数据自检 | 否(查 Milvus/词典) | 已是最强异质(数据) |

### 9.4 "fallback+coverage 要不要换异质判官?"——分析:不换,弃权门更优
真循环只有 fallback+coverage 这处还没上异质。分析过该不该换 Qwen 判 fallback:

- **弃权门(现状)**:fallback 候选 coverage 不自信(<0.8)→ 弃权。免费、确定性、保守;**挡住了"幻觉类"**(NOP→Nocturnal Oxygen Protocol 那种低置信的)。
- **残留失败(QRS/ABC/LMN)不是幻觉,是"真词低上下文误扩"**:"QRS complex" 是 QRS 的**正确**扩写,错在低上下文不该扩。换千问**照样觉得它合理**——两模型在这里**盲点一致(缺的是上下文不是知识)**,异质修不了。
- 换 Qwen 判 fallback:**预期收益≈0**(治不了真残留)、**和弃权门重叠**(治得了的它已治)、**还有回归风险**(可能在现 18/18 全过的 CASI fallback 上过度弃权)。→ **不换,弃权门就是这里的最优解。**

### 9.5 升级版认知(几句面试金句)
- **异质判官治"相关的幻觉",治不了"相关的信息缺失"**(两个模型都没上下文时,换模型无用)。
- **项目真正的安全垫是"数据扎根 + 确定性弃权门",不是模型异质**——异质只在"同模型查自家生成的可信幻觉"那处才有不可替代的价值。
- **multi-agent ≠ 治自生自查**:同模型扮多角色根本没破循环;治它的是【异质检查者】(最好是确定性数据,其次是不同模型)。
- **运行时 vs 离线**:同一套系统两个用法——离线带 gold 考试(benchmark 脚本,线上不跑),线上无 gold、靠"扎根库+护栏+弃权";想给线上加独立检查只能上异质第二意见(对答案运行时没 gold 用不了)。

## 10. L3 多源路由 + Agentic 自纠环 —— agentic RAG 升级(批次 L3 Stage 1-6e)

> 这节是 V11 之后最大的能力升级:从"所有词只查一个库(SNOMED)"升级成"**按词的类型,把它路由到最合适的知识库**"——疾病→SNOMED、药品→RxNorm。这是 agentic RAG 的一种形态(routing)。但要诚实:**这里的路由是确定性的(NER 标签驱动),整条链路真正由模型自主的是 verify 的判忠实+弃权、以及自纠改写环**(见 10.7 / 10.8)。

### 10.1 为什么要做(动机 + 一句话)
- 一句话:**没有一个知识库什么都擅长。** SNOMED 为"疾病/症状/操作"设计,查 chest pain、diabetes 很准;但对**成分级纯药名**很别扭——库里 aspirin 相关只有"中毒/过敏/含某成分的制品/检测",没有干净的"阿司匹林(成分)"概念。
- 解法:再加一个**专门的药品库 RxNorm**,并加一个**路由器**按词的 domain 决定查哪个库。像图书馆按主题分区,而不是把所有书堆一个架子上找。

### 10.2 怎么搭(五个 Stage,每步都 benchmark 卡关)
| Stage | 做什么 | 性质 |
| --- | --- | --- |
| 1 | 建第二个 Milvus 库 `rxnorm_concepts`(同 Athena CONCEPT.csv,筛 RxNorm&标准&有效&Drug → 157190,Ingredient 14610);**SNOMED 库不动** | 建数据,行为中性 |
| 2 | 检索器加 `source` 开关(`search_similar_terms(..., source)`、`retrieve(..., source="snomed")`),**默认 snomed 向后兼容** | 加能力,行为中性 |
| 3 | 路由器:`_route_source(domain)` = `rxnorm if domain=="Drug" else "snomed"`,接在状态机主检索 + 反思重检索两处 | 确定性、零 LLM、不花钱 |
| 4 | 真喂药品:词典补 5 个成分级缩写(ASA→aspirin 等,标 `domain=Drug`)+ concept bench 补 5 条药品 gold(带 `domain` 字段) | 点亮路由 |
| 5 | 量化:A/B 脚本 `run_source_ab.py`,5 药 ×{snomed,rxnorm} 并排测 | 测收益 |

- **路由驱动 = NER domain(延续"激活死参数")**:NER 早就给每个词产了 domain 标签(批次 4 只拿它做软加分 domain_boost),L3 把它**升级成"选哪个库"的硬开关**。误路由的 LLM 兜底=**有意不做**(理由见 10.8)。
- **函数先行、框架后置**:先把"路由→检索→verify→反思"做成普通函数、benchmark 测通;LangGraph 可视化留到最后可选(Stage-6)。框架要"挣到钱"才上,不为贴标签套框架。
- **向后兼容**:老 11 条 concept 用例没有 `domain` 字段 → `_route_source(None)="snomed"` → 行为完全不变;主 benchmark 全程持平 71/74=0.9595(主 bench 里没有药品缩写 → 中性)。

### 10.3 量化结果(L3 收官数字)
```text
drug                source  chosen                code   domain   top1
aspirin             snomed  (弃码)                 None   None     0.8064
aspirin             rxnorm  aspirin               1191   Drug     1.0
methotrexate        snomed  (弃码)                 None   None     0.8073
methotrexate        rxnorm  methotrexate          6851   Drug     1.0
acetaminophen       snomed  (弃码)                 None   None     0.7196
acetaminophen       rxnorm  acetaminophen         161    Drug     1.0
hydrochlorothiazide snomed  (弃码)                 None   None     0.7965
hydrochlorothiazide rxnorm  hydrochlorothiazide   5487   Drug     1.0
nitroglycerin       snomed  (弃码)                 None   None     0.8081
nitroglycerin       rxnorm  nitroglycerin         4917   Drug     1.0
```
- **结论:药品标准化 SNOMED 0/5 → RxNorm 5/5**(concept bench 从 11/11 一路扩到 **19/19 PASS、17/19 canonical**;Stage-6d 又补了 3 条俗称硬 case)。
- **关键读法(面试要讲这个,不是只报 5/5)**:SNOMED 那列**不是没检索到**(top1 相似分 0.72~0.81 都过了 0.6 阈值),是 **verify 主动弃码**——因为 SNOMED 近邻都是"中毒/过敏/制品/检测"这类**不忠实**概念,弃码门正确拒绝把 aspirin 标成"aspirin 中毒"。换 RxNorm 后才有正确成分概念可锁(top1=1.0,均为真 RxNorm CUI)。
- **所以增益是"真覆盖",不是"换排序"、更不是"放松护栏"**:护栏(忠实度弃码门)一直对,L3 只是给了它对的库去命中。这条证据链比"我加了 RxNorm 分数涨了"有说服力得多。
- **端到端冒烟**:`The patient took ASA for chest pain.` → `aspirin`,ASA 的 domain=Drug、标准化命中 RxNorm 概念 aspirin / code 1191。检测→扩写→domain→路由→RxNorm 标准化整链打通。

### 10.4 诚实边界(主动讲 = 可信度)
"药品"有好几种形态,难度天差地别,我们只证明了最容易的一档:
- **成分级纯药名(我们测的)**:aspirin、acetaminophen——RxNorm 主场,5/5。
- **复方药**(如 oxycodone + acetaminophen):要拆/合成分,未覆盖。
- **商品名**(Tylenol ≡ acetaminophen):需多一步"商品名→成分"映射(RxNorm TTY / RxNav),未覆盖。
- **剂型/剂量**(aspirin 100 MG 口服片):比成分更细,朴素 query 不一定对得上,未覆盖。
- **疾病类词**:RxNorm 里根本没有疾病——误路由过去会一个都给不出(所以疾病必须走 SNOMED)。
- 一句话:**"对成分级药名,domain 路由到 RxNorm 确定性有用(0/5→5/5);复方/商品名/剂型/疾病不在此结论内。" 这是一个有边界、可量化的能力,不是普适胜利。**

### 10.5 几句面试金句
- **agentic RAG 的本质是"系统掌控检索决策"(去哪查、查不到怎么办),不是套某个框架**;L3 的"按 domain 路由到不同库"就是检索决策的一种(routing)。
- **多源的价值要测出来、并说清边界**:同一批药 SNOMED 0/5、RxNorm 5/5,且 SNOMED 的失败是诚实弃码(候选在 0.7~0.8 但不忠实)——证明是"补了正确知识源",不是"放松了判定"。
- **路由用确定性信号(NER domain)、零额外 LLM**:能确定性做的就别加模型——延续全项目"能算就别问 LLM"的口径。

### 10.6 LangGraph 可视化(Stage-6 / 6c / 6e)—— 框架待在可视化层
- 把"**单个缩写的标准化之旅**"用 LangGraph 画成可读状态图:`route` 岔路(Drug→RxNorm / 其它→SNOMED)+ `verify` 判忠实/弃权 + **有界自纠环**(`propose_requery→re_retrieve→re_verify` 回边)。图见 `项目梳理/L3_pipeline.mmd` / `L3_面试讲稿.md`。
- **不进生产热路径**:生产是轻量确定性状态机 `expand_verify_with_retry`;LangGraph 是并行的可视化 + 备用入口,用 **parity 测试**(`graph.run` vs 生产逐字段一致,ALL PASS)保证"框架版没改变行为"。
- **诚实回答"LangGraph 给生产带来了什么":没进热路径,是有意选择。** 它真正值钱的 checkpoint / 流式 / human-in-the-loop / 节点重试 / 动态扇出,这条短链路都用不上;能说清它在这个规模带不来新能力,所以只让它做画图 + 回归测试。**框架要挣到钱才进热路径。**(反例:V9 把逻辑塞进 LangGraph,后被证明是死重量整目录删——见第 7 节批次7。)
- 图的单位 = 单 mapping(agentic RAG 标准画法);"一句话多个缩写"由外层逐个跑图。代价:verify 在图里逐词调用、生产批量调用 → parity 是"**结果级一致**"非"逐字节等价"(诚实记一笔)。

### 10.7 反思升级成有界自纠环(Stage-6d,进生产)
- 批次10 的反思是"单趟";Stage-6d 把它升级成**有界迭代环并放进生产** `abbr_service`:每轮换同义词重检索重判,**质量秩(`_std_rank`:2 精确同名 / 1 忠实非同名 / 0 弃码)严格变高才继续**,最多 2 轮。
- **保证终止、不空转**:秩 ≤2 只能涨两次、首轮横移只采纳一次即停、无新改写词 / 无新候选 / 结果不变即停、`max_iter`(env `REFLECT_MAX_ITER`,默认 2)兜底。
- **ablation 诚实**:`REFLECT_MAX_ITER=1` vs `2` 在现有数据上结果**完全相同**——第 2 轮零额外收益(每个 case 一轮内收敛)。**保留为安全余量、如实报告 ROI,不吹**;代价为零(现数据都第 1 轮早停,无额外 LLM 调用)。
- 面试点:**自纠环的价值不在"我建了环",而在"建了 + ablation 诚实量化它在我数据上没额外收益、却保留为安全余量"——测量与诚实边界本身是加分。**

### 10.8 有意不做的(及为什么)—— 工程判断 > 堆 agentic
- **不用 LLM 路由**:2 源时路由只是"是不是药"的二分类,NER 标签又快又免费又可靠;LLM 做同一二分类严格更差(延迟 / 成本 / 幻觉)。**源多到 3-4 个、路由成真决策才升级**;真上也只让模型在"已有的几个源"里**选**(分类),绝不让它自由生成一个还要去对应、可能对不上的 domain key——**约束动作空间到合法选项**。
- **不让模型"选动作"(reroute / broaden 等)**:action 空间小,固定恢复策略("弃码→换源→放宽→真弃")就够且可预测;让模型当调度器只是把确定性换成"慢、可能错、难调"的不确定性。**agency 有成本,只加在决策真难处。**
- **统一口径**:能确定性算的就别问 LLM;框架 / agency 挣到钱才上。**整条链路真正"由模型自主"的只有两块——verify 判忠实+弃权、有界自纠改写环;路由是确定性,上述两项是有意不做。** 知道"何时该上、何时是表演",比到处套 agentic 更专业。
- 完整口述见 `项目梳理/L3_面试讲稿.md`。
