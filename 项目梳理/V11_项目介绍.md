# Medical NLP · V11 项目介绍(干净版)

> 用途:重新熟悉项目 + 面试自查。读完应能在白板上画出整条链路,讲清每个模块**做什么、数据进来长什么样、在这一步被怎么处理、出去变成什么**。
>
> 一句话定位:**把"临床缩写扩写 + 医学术语标准化"做成一条「候选召回→确定性替换→每个词走自己的状态机(检索→LLM 判忠实→弃码/锁定→反思自纠)」的、多源(SNOMED+RxNorm)、可评估的后端服务。**

---

## 0. 30 秒电梯陈述

用户发一段临床文本(含 SOB、CP、ASA 这类缩写)。系统先对每个缩写做**候选召回 + coverage 裁决**,选出**唯一最佳展开**;然后用**确定性 token 边界替换**把缩写换成全称(不用 LLM 造句,从源头杜绝丢否定/幻觉)。之后工作单位降到**单个 mapping**:每个词走自己的状态链——按它的 **NER domain 路由**到 SNOMED 或 RxNorm 库做向量检索(bge-m3 + Milvus + 规则重排)→ **LLM verify 在候选里挑忠实概念、挑不到就弃码**(宁可不给码也不给错码)→ 不够规范时进**有界自纠环**(换同义词重检索重判,最多 2 轮)。一个词失败只隔离重修它自己,对的直接锁定。出口给扩写文本 + 每个成功项的标准编码(standardized_entities)。整套 FastAPI 暴露,有主 benchmark(判扩写)+ concept benchmark(判标准化)双层评测,LangGraph 把这条链路画成可视化图(不进生产热路径)。

三个关键词:**RAG(多源向量检索 + 规则重排)、Agent(verify 判忠实/弃权 + 有界自纠环)、Evaluation(双层 benchmark + 错误分析系统)**。

## 3. 模块地图(按调用层级,从入口到底层)

每个模块给:**做什么 / 输入→输出 / 数据在这一步怎么变 / 被谁调 / 面试重点**。⭐ 为面试必问。

### ⭐ ABBRService —— 主编排器
`services/abbr_service.py`

- **做什么**:`__init__` 一次性实例化复用所有子服务(DeepSeek LLM、MedicalStandardizer 及其 NER、MedicalRetriever、ABBVerifier、候选三件套);对外只暴露主链路 `expand_verify_with_retry()`。
- **输入→输出**:`text, max_retries=2` → `{original_text, final_expanded_text, success, attempts, final_result}`。
- **核心方法**:
  - `_get_abbreviation_candidates(text)` —— 召回 + coverage 选唯一(见 4.1)。
  - `_build_expanded_text_deterministic(text, chosen)` —— `\bABBR\b` token 边界、从后往前替换,不误伤子串、否定天然保住。
  - `_route_source(domain)` —— `"rxnorm" if domain=="Drug" else "snomed"`(L3 确定性路由)。
  - `_std_rank(s)` —— 标准化质量秩:`2` 精确同名 / `1` 忠实非同名 / `0` 弃码(给反思的迭代判据)。
  - `_reflect_refine_standardization(s, ...)` —— 有界自纠环(秩升才继续,最多 2 轮)。
  - `expand_verify_with_retry(text)` —— 主链路状态机(见第 4、6 节)。
- **注**:`self.abbr_dict`(__init__ 里那个 6 条小字典)是**历史死代码**,主链路不用它;真正的词典是 `data/abbr_candidates.py`。
- **面试重点**:子服务放 __init__ 复用(避免重复加载模型);确定性替换是"mapping 粒度隔离"的物理前提;状态机三终态(CODED 锁定 / WITHHELD 弃码 / ABSTAIN 弃扩)。

### ABBRCandidateRetriever —— 主候选召回(词典)
`services/abbr_candidate_retriever.py`

- **做什么**:从 `data/abbr_candidates.py` 的 `ABBR_CANDIDATES` 按缩写查候选。纯字典、无 LLM、零延迟。
- **输入→输出**:`abbreviation` → `[{abbreviation, expansion, domain}]`。
- **数据怎么变**:`"ASA"` → `[{"abbreviation":"ASA","expansion":"aspirin","domain":"Drug"}]`;`domain` 是**人工策展**的,直接决定后面路由去哪个库。
- **面试重点**:第一层召回用词典而非 LLM——确定性、省 token、可控,且 domain 可信。

### ABBRCandidateFallbackRetriever —— 兜底候选(LLM)
`services/abbr_candidate_fallback_retriever.py`

- **做什么**:词典查不到时,LLM 结合上下文**只生成候选列表,不改写原句**。
- **输入→输出**:`abbreviation, context_text` → `{abbreviation, candidates:[{expansion,...}], reason}`。
- **数据怎么变**:词典没有的缩写 → LLM 产候选;之后主编排器对每个候选跑 `NERService.is_medical(expansion)` 拿 label → 经 `NER_LABEL_TO_DOMAIN` 补上 `domain`(词典候选自带 domain,fallback 候选靠 NER 补)。
- **面试重点**:职责严格限制为"只产候选不改句",便于后面 coverage 统一裁决;且对 fallback 后面会收紧弃权门(见下)。

### ABBRCandidateCoverageEvaluator —— 候选覆盖度 + 选唯一(LLM)
`services/abbr_candidate_coverage_evaluator.py`

- **做什么**:LLM 判断候选里哪些被上下文支持,并**直接选出唯一最佳展开**。
- **输入→输出**:`original_text, abbreviation, candidates` → `{coverage_ok, confidence, plausible_candidates, best_expansion, reason, issues}`。
- **数据怎么变**:候选列表 + 上下文 → 收敛成**一个** `best_expansion`(逐字从候选里拷贝,不许 LLM 改写);`coverage_ok=false` 时 `best_expansion=null`。**消歧(选哪个扩写)的拍板权收在这一处**——这也是后面 verify 不再碰扩写的原因。
- **面试重点**:它是"召回与替换之间的闸门",把"缩写扩写问题"变成"候选选择问题"。

### NERService / MedicalStandardizer —— 实体识别(只产 domain)
`services/ner_service.py` + `services/medical_standardizer.py`

- **做什么**:NER 抽医学实体并**合并相邻实体**(chest + pain → chest pain);`is_medical(text)→(bool,label,score)` 给孤立短语推 label。
- **输入→输出**:`text` → `[{text,label,score,start,end}]`;`is_medical` → `(True,"MEDICATION",0.9)` 之类。
- **数据怎么变**:label 经 `NER_LABEL_TO_DOMAIN`(MEDICATION→Drug、DISEASE_DISORDER→Condition…)变成 SNOMED domain,**只用来产 domain,不做候选过滤**。
- **注**:V9 的"整句 standardize"盲肠已在批次 7 删除;`MedicalStandardizer` 现在主要就是给主链路提供这颗 NER。
- **面试重点**:NER 是本地模型、与 DeepSeek 不同源 → 跨模型第二意见;V10 曾用它做候选过滤因制造"像医学的假候选"被回退,只保留无害的"产 domain"。

### ⭐ MedicalRetriever —— RAG 检索 + 规则重排 + 路由
`services/medical_retriever.py`

- **做什么**:调 StdService 做向量检索 → 规则重排 → 过滤 → 包成 document。
- **输入→输出**:`query, top_k=10, domain_filter, domain_boost, score_threshold=0.6, source="snomed"` → `[{page_content, metadata:{concept_id, concept_name, domain_id, concept_code, score, rerank_score}}]`。
- **数据怎么变**:query 文本 →(StdService 向量召回 top_k 条)→ `_rerank_results` 给每条加 bonus(**=query +0.5 / 开头 +0.3 / 包含 +0.15 / domain 命中 +0.2 / 过长扣分**)→ 按 `rerank_score` 排序 → 用 `max(raw, rerank) ≥ 0.6` 过滤(修过:不只卡 raw,避免重排顶上来的好概念被原始分误删)。
- **面试重点**:`source` 把检索路由到 snomed/rxnorm 两个库之一(L3);向量召回后规则重排——语义相似 ≠ 术语精确,短而精确的标准术语应优先。

### ⭐ StdService —— Embedding + Milvus 多源底座
`services/std_service.py`

- **做什么**:建 bge-m3、连 Milvus、按需 `load_collection`;`search_similar_terms` 把 query 向量化后在**指定源的 collection** 上搜。
- **输入→输出**:`query, limit, source` → `[{input, concept_id, concept_name, domain_id, concept_code, FSN, score}]`。
- **数据怎么变**:`collections = {"snomed": concepts_only_name, "rxnorm": rxnorm_concepts}`;query →`embed_query`→ 向量 → `client.search(collection=该源)` → 取回概念 + `score=round(distance,4)`。**默认 snomed,首次用某源才 load**(懒加载 collection)。
- **面试重点**:双库同 schema 同 bge-m3,只靠 `source` 切;这是 L3 多源的物理底座。

### ⭐ ABBVerifier —— 标准化卡点(LLM-as-judge)+ 反思改写词
`services/abbr_verifier.py`

- **做什么**:① `verify_mappings` 在检索候选里**选忠实概念或弃码**;② `propose_requeries` 给反思产同义检索词。
- **输入→输出**:
  - `verify_mappings(original_text, expanded_text, mapping_standardizations:[{abbreviation, expansion, candidates}])` → `{mapping_validations:[{abbreviation, expansion, chosen_index, standardization_faithful, reason}], overall_valid}`。
  - `propose_requeries(expansion, current_concept, seen_concepts)` → `["dyspnea", ...]`(最多 2 个,想不出返回空)。
- **数据怎么变**:给每个候选标 0 起的 index → LLM 返回 `chosen_index`(选哪个忠实)+ `standardization_faithful`;都不忠实就 `chosen_index=null` → 弃码。**verify 不碰扩写**(那是 coverage 的活)。
- **面试重点**:批次 7 实测旧 verify(复查扩写)= 橡皮图章(FAIL/SWAP/ABSTAIN 0/0/0),所以挪到唯一有新数据的地方——判 SNOMED/RxNorm 标准化质量;抓"扩对但标准化错"(chest pain→"Chest pain rating"那种)。

### LLM / Embedding 工厂
`utils/llm_config.py` + `utils/llm_factory.py` + `utils/embedding_config.py` + `utils/embedding_factory.py`

- **做什么**:用 config(枚举 + 数据类)+ factory 把"用哪个模型"和"业务逻辑"解耦。换 DeepSeek/Qwen 只改一行 config(Qwen 走 DashScope OpenAI 兼容端点,接口与 ChatDeepSeek 一致)。
- **面试重点**:仿 embedding 工厂做的 LLM 工厂;曾用它做"verify 换异质模型"实验,实测与 DeepSeek 持平(因 verify 判的是检索结果、非自家生成,本就不是自生自查)。

### 错误分析系统
`services/error_collector.py` + `evaluation/analyze_errors.py` + `evaluation/error_triage.py`

- **做什么**:把系统**自己跑出来的错误**攒成可分析、抗误用的资产。收集两类(运行时 known-unknowns:弃码/没扩/弃权;评测 unknown-unknowns:GOLD_MISMATCH)→ 确定性打标(expected True/False/None)→ 读时筛 → LLM 提假设 + **数据自检(查 Milvus/词典打脸)** + LLM 写中文执行摘要。
- **面试重点**:LLM 提假设、**数据/benchmark 裁决**、LLM 写报告——三段分工;抗误用三重(自检打脸 + 摘要排除 + 假信号不产草稿)。

### LangGraph 可视化图(不进热路径)
`graph/standardization_graph.py` + `graph/render_graph.py`

- **做什么**:把"单个缩写的标准化之旅"用 LangGraph 画成状态图:`route →(domain)→ retrieve_rxnorm/retrieve_snomed → verify →(需反思)→ propose_requery→re_retrieve→re_verify⇄ → finalize`。节点只调 ABBRService 现有方法。
- **当前状态**:**并行的可视化 + 备用入口,不接 `/expand/simple`**;用 parity 测试保证图版与生产版输出一致。
- **面试重点**:框架待在可视化层,要挣到钱(流式/断点/审核)才进热路径;详见 `L3_面试讲稿.md`。

### FastAPI App —— 服务入口
`api/main.py` + `api/schemas.py`

- **路由**:`GET /`、`/health`、`/benchmark/summary`、`/error-analysis/summary`;`POST /expand/simple`(完整 debug 版 `/expand` 被注释)。
- **输入→输出**:`{text}` → `SimpleExpandResponse{success, expanded_text, mappings, standardized_entities}`。
- **数据怎么变**:从 `final_result.mapping_standardizations[].chosen_concept` 取每个成功项的 SNOMED/RxNorm 编码,组装成 `standardized_entities`;弃码项不进。
- **面试重点**:懒加载(首个请求才建 ABBRService);出口只端忠实编码(弃码的诚实地不给)。

### 离线工具 tools/
- `build_concept_csv.py` / `build_rxnorm_csv.py`:从 OHDSI Athena 的 CONCEPT.csv(OMOP 格式)按 vocabulary_id 筛 SNOMED / RxNorm 临床标准概念。
- `rebuild_milvus.py` / `rebuild_rxnorm_milvus.py`:批量 `embed_documents` 重建两个 collection(比逐条快几十倍)。

## 4. 三条核心数据链路

### 4.1 缩写候选召回 + 确定性扩写(先 gate 后召回,coverage 选唯一)

```text
文本 → 切 token → gate(_should_consider_abbreviation)
        ├ 已知缩写(在词典 key 里)        → 放行
        ├ 未知但全大写 ≤8 字符            → 放行走 fallback
        └ 小写普通词                      → 跳过(防噪声/过度扩写)
     → 路1 词典召回(自带 domain)                         primary
     → 查不到 → 路2 LLM fallback 生成候选 → NER 补 domain  fallback
     → CoverageEvaluator(LLM):选【唯一】best_expansion + coverage_ok/confidence
     → 【弃权门】fallback 且 coverage 不自信(<0.8) → best=None(不背书)
     → _build_expanded_text_deterministic:\bABBR\b 从后往前替 → expanded_text
```

### 4.2 标准化检索链(按 domain 路由 + 向量 RAG + 规则重排 + verify 选忠实)

```text
每个 mapping 的 best_expansion
     → _route_source(domain):Drug→rxnorm,其它→snomed
     → MedicalRetriever.retrieve(query=expansion, top_k=10, domain_boost=domain, score≥0.6, source=源)
         → StdService.search_similar_terms(query, limit=10, source)
             → bge-m3 embed_query → Milvus 在该源 collection 搜 TopK
         → _rerank_results(=query+0.5 / 开头+0.3 / 包含+0.15 / domain+0.2 / 过长扣分)
         → max(raw,rerank)≥0.6 过滤 → std_cache(最多 10 个候选概念)
     → verify_mappings:chosen_index 选忠实概念 / null=都不忠实→弃码
     → CODED(存 chosen_concept) 或 WITHHELD(弃码)
```

### 4.3 标准化反思自纠环(批次10 + Stage-6d,有界迭代)

```text
verify 选到非精确同名(或弃码) 且 秩<2:
   propose_requeries(LLM 出同义检索词,过滤已试过的)
   → 按 domain 路由重检索,新概念并入候选池(去重、按 score 取前 15)
   → re-verify 在更全池里重选(保守门:重选概念名须 ∈ 本轮改写词)
   → 秩严格变高才再来一轮;否则停(最多 2 轮)
   停止条件:秩=2 / 无新改写词 / 无新候选 / 结果不变 / 到上限
```

## 5. 一条 record 的数据形状演变(这是数据流的核心)

V11 批次 11A 起,**每个缩写从召回到出口是同一条 `record` 字典**,字段随阶段填充、status 标生命周期。跟着 "CP" 走一遍:

```text
① 召回后(_get_abbreviation_candidates → 建 record):
   { abbreviation:"CP", source:"primary",
     candidates:[{expansion:"chest pain",domain:"Condition"}, ...],
     coverage:{coverage_ok:true, confidence:0.9, best_expansion:"chest pain", ...},
     expansion:"chest pain",          # = coverage 选的 best
     label:None, domain:"Condition",  # domain 来自选中候选
     std_cache:None, std_concept:None,
     status:"PENDING", failure:None }
        │  (没扩出来的:expansion=None, status="NOT_EXPANDED")

② 确定性替换:expanded_text = "...chest pain..."(record 不变,只产文本)

③ 路由 + 检索后(填 std_cache):
   domain="Condition" → _route_source → "snomed"
   std_cache = [ {concept_id, concept_name:"Chest pain", domain_id:"Condition",
                  concept_code, score:0.99, rerank_score}, ... 最多10条 ]

④ verify 后:
   chosen_index=0, standardization_faithful=true
   → std_concept = std_cache[0]("Chest pain"), status="CODED", failure=None
   (若都不忠实:std_concept=None, status="WITHHELD", failure={type:"CODE_WITHHELD",...})

⑤ 反思(非精确同名才触发;CP 已精确同名 rank=2 → 跳过)

⑥ 出口:
   final_result.mapping_standardizations += {abbreviation:"CP", expansion:"chest pain",
        candidates:std_cache, chosen_concept:std_concept}
   final_result.mapping_states += {abbreviation:"CP", expansion:"chest pain",
        status:"CODED", failure:None}
   API → standardized_entities += {concept_id/code/name/domain_id of "Chest pain"}
```

三种终态:**CODED**(给码,冻结复用)/ **WITHHELD**(扩对了但库里无忠实概念,弃码)/ **ABSTAIN**(候选用尽没锁住,弃扩)。

## 6. 运行流程图(盒式分段 · 等宽字体看)

```text
═══════════════ 一次性:启动(懒加载)═══════════════
  Uvicorn 加载 api.main:app → service=None(不加载模型)
        │  首个 POST /expand/simple 到达
        ▼
  get_service() → ABBRService.__init__()  一次性复用:
     DeepSeek · MedicalStandardizer(NER) · MedicalRetriever
     · StdService(bge-m3 + Milvus 双库) · ABBVerifier · 候选三件套

═══════════════ 一次请求:expand_verify_with_retry ═══════════════
   POST /expand/simple { text:"Patient has CP and took ASA", max_retries=2 }
        │
        ▼
┌─ ① 召回 + 选唯一 + 确定性替换(循环外,只算一次)──────────────────┐
│  逐 token → gate → 路1 词典(带 domain) / 路2 fallback(LLM+NER 补 domain) │
│  coverage(LLM) 选【唯一】best_expansion;fallback 不自信→弃权门 best=None │
│  → 建 records(每词一条,status=PENDING/NOT_EXPANDED)                      │
│  _build_expanded_text_deterministic:\bABBR\b 从后往前替 → expanded_text  │
└──────────────────────────────────────────────────────────────────────────┘
        │
        ▼
   ◇ 没有任何 expansion? → coverage_failed,success=False 返回
        │否
        ▼
═══ ② 状态机循环(per-mapping,失败隔离)═══
   对每个 PENDING record:
     c. 路由检索:_route_source(domain) → retrieve(top_k=10, source) → std_cache
     d. verify_mappings:chosen_index 选忠实 / null 弃码
     e. CODED(存 chosen_concept) 或 WITHHELD
     f. 反思自纠环:非精确同名→换词重检索重判(秩升才继续,≤2 轮),可把 WITHHELD 救成 CODED
        │
        ▼
   循环末仍 PENDING → ABSTAIN(弃扩)
        │
        ▼
═══ ③ 出口 ═══
   final_result{ expanded_text, mappings, mapping_standardizations[chosen_concept], mapping_states }
   API 组装 standardized_entities(只端忠实编码)
        │
        ▼
   HTTP 200 → { success, expanded_text, mappings, standardized_entities }

跑例:CP 路由 snomed→"Chest pain" 锁定;ASA domain=Drug 路由 rxnorm→"aspirin"/code 1191。
两者各走各的状态机,一个失败不牵连另一个。

─────────── 离线评测(与 HTTP 链路并行,不在请求里跑)───────────
  主 benchmark(74 例含 CASI):run_benchmark → 判【扩写】accuracy=0.9595
  concept benchmark(19 例):run_concept_benchmark → 判【标准化】PASS 19/19 / canonical 17/19
  A/B(run_source_ab):多源量化,药品标准化 SNOMED 0/5 → RxNorm 5/5
  错误分析:error_collector → analyze_errors → error_triage
```

## 7. 面试速记表

| 维度 | 一句话答案 |
| --- | --- |
| 最重要的 5 个类 | `ABBRService` / `MedicalRetriever` / `StdService` / `ABBVerifier` / `ABBRCandidateCoverageEvaluator` |
| 最重要的 5 个函数 | `expand_verify_with_retry` / `_get_abbreviation_candidates` / `_build_expanded_text_deterministic` / `retrieve`(domain_boost+source) / `verify_mappings` |
| RAG 体现在哪 | 候选召回(词典+LLM 兜底+coverage)+ 多源向量检索(bge-m3+Milvus+规则重排+domain 路由) |
| Agent 体现在哪 | verify 判忠实+弃权(真自主)+ 有界自纠改写环(带新证据迭代) |
| 数据流主线 | 一条 record 走到底:召回填 expansion → 检索填 std_cache → verify 填 std_concept+status → 反思可升级 → 出口 |
| 确定性优先 | 能用替换/规则/NER 确定地做的就不用 LLM;LLM 只放在"判忠实"和"出改写词" |
| 工程亮点 | mapping 粒度失败隔离、确定性替换保否定、verify 放对位置、双层 benchmark、多源 0/5→5/5、错误分析系统 |
| 诚实可讲的不足 | 多源只证明成分级药名;CAD/HBP 是 canonical 缺口;自纠环第2轮现数据零增益(留作安全余量);benchmark 偏小 |
